## 0. TL;DR

```
─── 每天 08:00 前 ───────────────────────────────────────
Desktop 啟動或日內首次 tick:
  GET /api/work/my-config  ──▶ 拉一次配置(含模板 ID,§8 後)
  檢查本地有 unsynced 紀錄 → 補傳

─── 工時 08:00-17:00 ───────────────────────────────────
Desktop tick(每 N 分鐘,本地隨機 jitter 削峰)
  ├─ 本地 idle 偵測(鍵鼠空閒)→ 寫本地 SQLite,**不打 server**
  └─ active → POST /api/work/analyze(jpeg + 元數據)
                server 同步調 Qwen-VL → 回 {category, ...}
                **server 不落庫**,只透傳結果
                Desktop 拿到結果 → 寫本地 SQLite

─── 17:00 後或啟動補傳 ─────────────────────────────────
Desktop:
  POST /api/work/sync-daily(批次,200 條/批)
  server 一次性 INSERT 落庫

─── 管理員任何時候 ─────────────────────────────────────
GET /api/admin/work/users          ── 列出有採集的人
GET /api/admin/work/users/{id}/*   ── 查單人配置 / 紀錄 / 統計
PATCH /api/admin/work/configs      ── 批次改配置(userIds 陣列)
```

**為什麼這樣設計**:

- 工時內 server stateless(不落庫)→ 不需要隊列 / Worker pool / Redis
- 落庫集中在 17:00 後批次 → 峰值 < 20 req/s,單實例 + SQL Server 2008 輕鬆 hold
- desktop 本地 SQLite 是 cache,server 是 source of truth → 跨機 / 重灌可 backfill
- 截圖**永遠不落地**(desktop / server 都不存)

## 1. 整體架構

```
┌──────────────────┐    HTTPS(JWT)    ┌─────────────────────────┐
│  Desktop (500x)  │ ────tick────────▶│  TMBOM 後端 (.NET 7)     │
│  - dHash         │ ◀──config────────│  - /api/work/analyze     │
│  - idle 偵測     │                  │  - /api/work/my-config   │
│  - 本地 SQLite   │                  │  - /api/work/sync-daily  │
│  - 個人 UI       │                   │                          │
└──────────────────┘                  │  SQL Server 2008(SAP)   │
                                      │  ├─ Work_Records         │
                                      │  ├─ Work_User_Configs    │
                                      │  └─ Work_Category_*(§8) │
┌──────────────────┐                  │                          │
│   Admin Web      │ ──HTTPS(JWT)────▶│  /api/admin/work/*       │
│   (TMBOM 子站)   │                  │                          │
│  - 用戶列表       │                  │                          │
│  - 批次配置       │                  │                          │
│  - 查紀錄/統計    │                  │                          │
└──────────────────┘                  └─────────────────────────┘
```

**核心政策**:

1. **截圖端到端不落地** — desktop 是記憶體 Buffer 送完即釋,server 處理完即丟,任何環節禁 `fs.write*` 截圖
2. **本地 SQLite 是 cache** — UI 永遠讀本地;離線可用;跨機透過 `/my-records` backfill
3. **批次 sync** — 工時內不落庫,17:00 後一次性傳;safety net 處理關機 / 跨日邊界
4. **時區鎖死北京(UTC+8)** — 工時 8-17 永遠按北京時間判斷,不跟使用者本機時區走

## 2. 採集流程(Desktop)

### 2.1 Tick 邏輯

```
WorkCollectorScheduler.tick()(每 N 分鐘,主進程)
  │
  ├─ 1. 主視窗未就緒 / 螢幕鎖 / 非工時 → skip
  ├─ 2. desktopCapturer 抓全屏 NativeImage
  ├─ 3. computeDHash(9x8 灰階差分,16 hex)
  ├─ 4. collectVisibleWindowTitles()(最多 10 個)
  │
  ├─ 5. idle 判定:
  │     - 系統 idle 秒數 ≥ 間隔秒數 - 5s grace
  │     - OR dHash 距離 ≤ 10 且前台視窗未變
  │     → 寫 idle 紀錄到本地 SQLite,**不打 AI**
  │
  └─ 6. active → renderer 走 createHttpClient(帶 JWT)
        POST /api/work/analyze(multipart):
          screenshot=jpeg
          activeWindow / appName / allWindows / capturedAt
        ← server 同步返回 {category, description, confidence, reason}
        renderer IPC 回 main,寫 work_records
```

### 2.2 為什麼一張截圖 + 全部視窗清單

- 截圖只截 primary 一張(多螢幕成本高,先簡單做)
- `activeWin()` 只回前台單一視窗,多視窗並排 / 後台脈絡會丟
- 加 `allWindows`(只標題沒縮圖)補上「同時開了什麼」
- AI 收到「焦點 + 全部視窗 + 截圖」三層訊號互補

### 2.3 Tick 削峰(jitter)

```ts
// 啟動延遲一個隨機 jitter,避免 500 台 desktop 整點對齊
const jitterMs = Math.floor(Math.random() * intervalMs * 0.5)
setTimeout(() => {
    this.tick()
    this.timer = setInterval(() => void this.tick(), intervalMs)
}, jitterMs)
```

500 台分佈在前 50% 區間 → 單秒峰值降到 ~17 req/s。

### 2.4 Sync 觸發(三條互補)

```
觸發 1:startup(desktop 啟動)
  └─ 檢查本地 unsynced records → 有就 sync(處理「16:55 關機」)

觸發 2:工時剛結束(tick 內判斷)
  └─ 上次 tick 在工時內、這次發現非工時 → 觸發 sync

觸發 3:safety net(每小時跑一次,工時內)
  └─ unsynced > 50 → 提前 sync(防多天不關機累積過多)
```

單靠 17:00 太脆弱(關機 / 跨日 / 異常重啟都會丟),三條保證最終一致性。

## 5. 流量 / 承壓

| 時段                | 動作                               | 估算                          |
|-------------------|----------------------------------|-----------------------------|
| 8:00-8:15(配置同步)   | 500 人各拉一次                        | 平均 0.55 req/s,jitter 後峰值 ~2 |
| 8:00-17:00(工時內)   | 500 人 × 12 tick/h × 50% non-idle | 平均 0.83 req/s(打 Qwen-VL)    |
| 17:00-18:00(批次上傳) | 500 人各 1 次 × 100 條               | 平均 0.14 req/s,峰值 ~10        |
| Admin 操作          | 偶發                               | 可忽略                         |

**最大峰值 < 20 req/s**,單實例 .NET + SQL 2008 完全 hold。**完全不需要**隊列 / Redis / SignalR / Worker pool。

## 6. 時區(全鏈路鎖死北京 UTC+8)

| 層              | 儲存                                              | 顯示 / 判斷                                        |
|----------------|-------------------------------------------------|------------------------------------------------|
| Desktop SQLite | `capturedAt` Unix ms UTC                        | UI 顯示 `formatInTimeZone(ms, 'Asia/Shanghai')`  |
| Scheduler 工時   | —                                               | `isInWorkHours()` 顯式用北京時間 hour                 |
| Server SQL     | `CapturedAt BIGINT` + `CapturedAtUtc DATETIME2` | 切日期時 `DATEADD(hour, 8, CapturedAtUtc) AS DATE` |
| Admin Web      | —                                               | 全部 `formatInTimeZone(..., 'Asia/Shanghai')`    |
| 配置觸發           | `lastConfigSyncDay = '2026-01-15'`(北京)          | 每天比對北京日期                                       |

```ts
// 不引入 date-fns-tz,直接 +8h 偏移(北京無 DST)
function getBeijingHour(): number {
    return (new Date().getUTCHours() + 8) % 24
}

function getBeijingDate(): string {
    const b = new Date(Date.now() + 8 * 3600_000)
    return `${b.getUTCFullYear()}-${String(b.getUTCMonth() + 1).padStart(2, '0')}-${String(b.getUTCDate()).padStart(2, '0')}`
}
```

**寫進 DB 的 `capturedAt` 永遠是 UTC ms**(不存「北京時間 ms」),顯示層才轉。

## 7. 後端文字 sanitize

DB 是 Big5/CP950 環境,簡體字落庫變 `?`。所有寫入字串欄位前走 `TextSanitizeService`:

- `Sanitize(s)` — 簡轉繁 + SQL escape(純文字欄位用)
- `SanitizeHtml(s)` — HTML-aware 切段繁化(只動 `<tag>` 之間文字,保護 URL / attrs)
- `ToTraditional(s)` — 只繁化不 escape(回前端 response 用)

**簡轉繁實作**:`ToolGood.Words.WordsHelper.ToTraditionalChinese(s)`(純托管,跨平台,Linux 可跑)。全程 try/catch,失敗
fallback 原字串,**寧可落 `?` 也不讓 sync 崩**。

調用點:`WorkCollectController.SyncDaily` 落庫前對 `activeApp` / `activeWindowTitle` / `description` / `reason` 統一
sanitize。

> ⚠️ **重要**:`Work_Records` 所有中文欄位**必須是 NVARCHAR**(`ActiveApp` / `ActiveWindowTitle` 之前是 VARCHAR 造成 `登?`
> 乱码,已修正)。

## 8. 下一步:模板化 + 動態 Prompt + idle 隱藏

> 三件事**串起來形成閉環**,不要拆開做。

### 8.1 目標

1. **idle 對使用者隱藏**(後台保留全量,**不給開關**)
2. **管理頁可配置分類模板**,使用者透過採集設定掛上模板(**沒設定即不採集**)
3. **動態 Prompt**:模板語意注入 Qwen-VL,讓 AI 反推「正在做什麼業務」

### 8.2 關鍵架構變更

#### 8.2.1 單軸分類取代寫死 8 類

**現在**:`category ∈ { coding, documenting, browsing, communicating, meeting, designing, idle, other }`(
寫死,動作維度,粗)

**之後**:`category ∈ 模板裡定義的 code 集合 ∪ { "OTHER" }`(模板決定,業務維度)
例:`BOM_MAINT` / `IT_REPAIR` / `ECO_REVIEW` / `OTHER`

**歷史資料保留**:舊值(`coding`/`documenting`/`idle`)留檔不重分類,前端顯示「歷史 / 未分類」灰塊。

#### 8.2.2 idle 從 category 拆出來變狀態欄位

**新增**:`Work_Records.ActivityState NVARCHAR(20)` ∈ `{ 'active', 'idle' }`

| 場景          | desktop 行為          | category          | activityState |
|-------------|---------------------|-------------------|---------------|
| 鍵鼠 5min 無活動 | **本地判定,不發 analyze** | `NULL`            | `idle`        |
| 鍵鼠有活動       | 發 analyze           | 模板 code 或 `OTHER` | `active`      |

**好處**:idle 不調 AI 省成本;category 語義純粹;隱藏 idle 變成「過濾 `activityState='idle'`」。

#### 8.2.3 模板 ↔ 人合進既有 Work_User_Configs

`Work_User_Configs` 加一欄 `CategoryTemplateId BIGINT NULL`。

採集啟用判定:
| Work_User_Configs 狀態 | 結果 |
|---|---|
| 無 row | **不採集** |
| `Enabled=0` | 不採集 |
| `Enabled=1`,`CategoryTemplateId=NULL` | **不採集**(設定不完整) |
| `Enabled=1`,有模板 | 採集 + 注入模板 Prompt |

**啟用但無模板 = 不採集** —— 簡化系統行為,避免 fallback 分支。

```
{
  "category": "BOM_MAINT | ECO_REVIEW | ... | OTHER",
  "description": "string(<= 50字)",
  "reason": "string(<= 80字)",
  "confidence": "number 0-1"
}
```

**Prompt 快取**:key = `(templateId, version)`,管理員 save 模板 → `Version++` → 舊快取自然失效,**無需主動推送 desktop**。

**AI 兜底**:若 AI 回了模板外的 code,server validate 強制改 `OTHER` + log warning。

### 8.5 管理頁

```
管理頁 > 工作採集
    ├── 採集總覽(現有)
    ├── ✨ 分類模板管理(新)
    │       ├── 模板列表(可搜尋 / 啟用切換)
    │       └── 模板編輯頁:基本資訊 + Items(三層展開) + Prompt 預覽
    └── ⭢ 採集設定(既有,擴一欄)
            └── 為每個使用者選 CategoryTemplateId(必填,啟用時)
```

**沒有獨立「成員指派頁」** —— 採集設定頁本來就在管使用者參數,模板 ID 是該頁多加的一個下拉。

**模板編輯頁三層展開**:模板 → items → examples,管理員可拖拽排序。**Prompt 預覽**展示組合後給 AI 的完整 prompt,降低配錯成本。

## 9. 不做 / 邊界

### 9.1 永遠不做

- ❌ **存原始截圖** —— desktop / server 全鏈路禁止
- ❌ **即時監控告警 / 排行榜** —— 合規紅線,定位是「個人回顧 + 團隊結構分析」,不是監控
- ❌ **顯示 idle 開關** —— 任何路徑都不暴露給使用者
- ❌ **手機 / 平板採集** —— 只跟 desktop 對接
- ❌ **跨組織共享 / 部門排行** —— 隱私敏感
- ❌ **管理員強制覆寫使用者本機 `Enabled`** —— 使用者本機關閉採集的權利保留(合規 / 員工關係)

### 9.2 暫不做(等真實訴求)

- ❌ **多模板 / 矩陣崗位** —— 一人一模板,矩陣需求等真實出現再擴
- ❌ **自動聚類生成模板 / Ontology / 知識圖譜** —— 字典要人維護,自動生成的是垃圾;§8 受控詞表已拿到 80% 主題歸類價值
- ❌ **工作成果層** —— 螢幕採集看不到產出,要接 ERP/PLM 是另一條數據線
- ❌ **AI 貢獻度因果推斷** —— 偽科學,只做「AI 工具時長佔比」這種可量化指標
- ❌ **Kafka / Pulsar / 大型 MQ** —— 500 人量級 SQL Server 2008 直接吃下,引入大型 MQ 是 over-engineering
- ❌ **異步隊列 / Worker Pool / SignalR** —— 對齊修訂後峰值 < 20 req/s,完全不需要
- ❌ **預聚合表 Work_Daily_Stats** —— 直接 GROUP BY 帶索引 < 100ms,卡了再加
- ❌ **模板版本歷史回溯** —— `Version` 只用於快取失效,不存歷史快照

### 9.3 已敲定的決策

| 決策                         | 影響                                          |
|----------------------------|---------------------------------------------|
| 使用者本機可關採集(`Enabled=false`) | desktop UI 保留總開關;admin PATCH 不允許覆寫 Enabled  |
| 連續 sync 失敗不告警              | desktop 端 sync 失敗只 log + 等下次再試;server 端不做告警 |
| 全鏈路北京時間                    | 工時 / 日期切割全部 UTC+8,使用者海外 VPN 不影響             |
| `Work_Records` 永久保留        | 不做 cron 清理,不分區;3 年後再評估                      |
| 管理員權限不分細                   | 第一版 admin role 看所有人;不分部門                    |
| 異常報表不做                     | 管理頁只有「使用者列表 + 詳情 + 配置」                      |
