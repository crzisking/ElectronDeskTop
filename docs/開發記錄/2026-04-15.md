## 修正：開發環境自動登錄邏輯（LoginView.vue）

### 問題描述

`onMounted` 中的自動登錄邏輯直接呼叫 `authStore.login()`，**完全跳過了表單**：
- 表單欄位 `form.userName` / `form.password` 始終為空
- Element Plus 的表單驗證（`el-form`）從未執行
- 表單 UI 不顯示任何值，外觀上看起來像空表單就登進去了

### 修改方式

改為「填入 form → 觸發 handleLogin()」，走正常的登錄流程：

```typescript
// 修改前：直接繞過 form 呼叫 store
onMounted(async () => {
  if (import.meta.env.DEV && import.meta.env.VITE_DEV_AUTO_LOGIN === 'true') {
    const username = import.meta.env.VITE_DEV_USERNAME
    const password = import.meta.env.VITE_DEV_PASSWORD
    if (username && password) {
      loading.value = true
      try {
        await authStore.login(username, password)   // ← 繞過 form
        await router.push({ name: 'unified-platform' })
      } catch (err) {
        errorMsg.value = err instanceof Error ? err.message : '自動登錄失敗'
      } finally {
        loading.value = false
      }
    }
  }
})

// 修改後：填入 form，由 handleLogin() 統一走驗證 + 登錄流程
onMounted(() => {
  if (import.meta.env.DEV && import.meta.env.VITE_DEV_AUTO_LOGIN === 'true') {
    const username = import.meta.env.VITE_DEV_USERNAME
    const password = import.meta.env.VITE_DEV_PASSWORD
    if (username && password) {
      form.userName = username    // ← 填入 form
      form.password = password
      handleLogin()               // ← 走正常流程（含表單驗證）
    }
  }
})
```

### 為什麼順序沒問題

- `onMounted` 的 callback 在元件掛載後才執行，此時 `form`（reactive）和 `handleLogin`（function 宣告，hoisted）都已初始化完畢
- `loading`、`errorMsg` 的管理統一由 `handleLogin` 負責，不需要在 `onMounted` 另外處理

### 涉及文件

| 文件 | 修改內容 |
|------|---------|
| `src/views/Login/LoginView.vue` | `onMounted` 改為填入 form 再呼叫 `handleLogin()`，移除 async/await 和冗餘的 try/catch/finally |

## 修正：auth.store.ts 登錄後拋出假錯誤的 Bug

### 問題描述

登錄 API 實際已成功（後端回傳了 token 和 user），但 UI 仍顯示「登錄失敗，請確認帳號密碼」。

### 根本原因：資料層層提取後，再用外層結構的欄位去判斷

後端回傳的原始 JSON：

```json
{
  "code": 200,
  "message": "登录成功。",
  "data": {
    "user": { ... },
    "token": "eyJ..."
  }
}
```

實際資料流：

```
後端 JSON
{code:200, message:'...', data:{user, token}}
        ↓
  Axios 攔截器（auth.interceptor.ts）
  response interceptor: return response.data
  → 回傳整個 {code:200, message:'...', data:{user,token}}
        ↓
  auth.api.ts
  const { data } = await getClient().post(...)
  → data = {user, token}     ← 已提取一層
  return data
        ↓
  auth.store.ts
  const res = await authApi.login(...)
  → res = {user, token}      ← 此時已是最內層的 data

  // BUG：res 是 {user, token}，根本沒有 .code，
  // undefined !== 200 → true → 直接拋錯！
  if (res.code !== 200) {
    throw new Error(res.message || '登錄失敗，請確認帳號密碼')
  }
```

### 修改方式

移除 `res.code !== 200` 的錯誤校驗（攔截器已處理業務碼錯誤，非 200 的 code 在攔截器層就會 `ElMessage.error`），直接解構使用 `{token, user}`：

```typescript
// 修改前
async function login(userName: string, password: string): Promise<void> {
  const res = await authApi.login({ username: userName, password })
  console.log(res)

  if (res.code !== 200) {   // ← BUG：res 已是 data 層，沒有 .code
    throw new Error(res.message || '登錄失敗，請確認帳號密碼')
  }

  const { token, user: userInfo } = res.data  // ← res.data 是 undefined！
  ...
}

// 修改後
async function login(userName: string, password: string): Promise<void> {
  // authApi.login 返回的是已提取的 data 層 { token, user }
  // 攔截器已統一處理業務碼錯誤（ElMessage.error + reject）
  const { token, user: userInfo } = await authApi.login({ username: userName, password })
    as unknown as { token: string; user: UserProfile }

  await window.electronAPI.auth.setToken(token)
  accessToken.value = token
  user.value = userInfo
  isAuthenticated.value = true
}
```

### 為什麼要用 `as unknown as {...}`

`authApi.login()` 的 TypeScript 宣告回傳型別是 `Promise<LoginResponse>`（含外層 `code/message/data`），但執行時真正回傳的是 `data` 層（`{user, token}`）。這是 Axios 型別系統的限制——它無法自動感知攔截器對回傳值的轉換。

加 `as unknown as { token: string; user: UserProfile }` 是讓 TypeScript 對齊執行時的真實型別，屬於有根據的 type cast，不是掩蓋問題。

### 涉及文件

| 文件 | 修改內容 |
|------|---------|
| `src/stores/auth.store.ts` | `login()` 移除 `res.code !== 200` 校驗與 `res.data` 解構；改為直接從 `authApi.login()` 結果解構 `{token, user}` |

## 數據流說明（供往後開發者參考）

理解這個項目的 HTTP 回應格式非常重要，避免再踩同樣的坑：

### 攔截器對回傳值的影響

`src/api/interceptors/auth.interceptor.ts` 的響應攔截器做了以下事情：

```typescript
instance.interceptors.response.use(
  (response: AxiosResponse) => {
    const { code, message } = response?.data ?? {}
    if (response.status === 200) {
      if (code && code !== 200) {
        ElMessage.error(`${code} - ${message}`)  // 業務碼錯誤直接彈提示
      }
      return response.data   // ← 返回整個 JSON body（含 code/message/data）
    }
    ...
  }
)
```

所以 `await axiosInstance.post(...)` 的解析值是 **後端 JSON body**（即 `{code, message, data}`），不是 Axios 預設的 `AxiosResponse` 對象。

### auth.api.ts 又再提取一層

```typescript
const { data } = await getClient().post<LoginResponse>(...)
// 此時 await 的值已是 {code, message, data:{user,token}}（攔截器回傳的）
// 解構 { data } 得到的是 {user, token}
return data  // 回傳 {user, token}
```

### 結論

拿到 `authApi.login()` 的結果後，**不要再去讀 `.code` 或 `.data`**，直接用 `{user, token}` 即可。業務碼的錯誤處理已在攔截器層完成。
