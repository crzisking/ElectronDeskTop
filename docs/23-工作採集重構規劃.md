## 0. TL;DR

兩個目標:

1. **架構調整 — 模板處理移到客戶端**
    - 之前:server 每次 analyze 查 template → 組 prompt → 調 AI(每次 1 條 SQL)
    - 之後:client 啟動拉一次 template → 落本地 SQLite → 每次 tick 本地組 prompt → 送 server 只做 AI 代理
    - 收益:server 端 0 SQL 查詢、prompt 邏輯集中前端、未來變更不用 server 重發

2. **代碼整理 — 拆分 / 歸類 / 去冗餘**
    - 後端:`WorkCollectService` 已經 600+ 行,拆三個專職 service
    - 桌面端:`src/features/work-collect/` 內部分層,composables / chart-options / api 等放整齊
    - 管理端:`views/WorkCollect/` 已經有 components/templates/analytics 子目錄,只做清理不重組

## 1. 架構調整:模板移到客戶端

### 1.1 數據流變化

**之前**(server-centric):

```
desktop tick(jpeg + meta)
    │
    ▼ POST /api/work/analyze
        │
        ├─ 1. server: SELECT * FROM Work_User_Configs(找 user → templateId)
        ├─ 2. server: SELECT * FROM Work_Category_Template + Items + Examples
        ├─ 3. server: 組 prompt(快取在 in-memory ConcurrentDictionary)
        ├─ 4. server: 調 Qwen-VL
        └─ 5. server: 校驗 AI 回的 code 在白名單 → 回 desktop
```

**之後**(client-centric):

```
desktop 啟動 / 每天 8 點:
    POST /api/work/my-config  →  返回 config + **整份模板詳情**(items + examples)
                                  │
                                  ▼
                            desktop 寫本地 SQLite work_template_cache
                            (一張表,單列存 templateId + JSON blob)

desktop tick:
    1. 從本地 SQLite 讀 template
    2. 本地組 prompt(把現在 server 端的 RenderPrompt 邏輯搬過來)
    3. POST /api/work/analyze(jpeg + **systemPrompt** + **allowedCodes**)
            │
            ▼ server 只做:
                ├─ 拼一下 user message(視窗 / app 名)
                ├─ 把 client 給的 prompt + json_schema 包進 Qwen 請求
                ├─ 調 Qwen-VL
                ├─ 校驗 AI 回的 code 在 allowedCodes 內(不在就改 OTHER)
                └─ 回結果
```

### 1.2 接口變化

#### `/api/work/my-config`(擴 response)

```ts
// 之前
{
  userId, enabled, intervalMinutes, workStartHour, workEndHour, version,
  categoryTemplateId, templateName    // 只有 ID + 名稱
}

// 之後:多帶 templateDetail
{
  ...原欄位,
  templateDetail: {
    templateId, version, name, description, promptSnippet,
    items: [
      { code, label, description, color, examples: [...] }
    ]
  } | null    // 未綁則 null
}
```

理由:多帶這份 ~ 5KB JSON,1 個 HTTP,省掉後續每次 analyze 都查 DB 的 3 條 SQL。

#### `/api/work/analyze`(改 input)

```ts
// 之前(server 自己組 prompt)
multipart:
  screenshot, userName, activeWindow, appName, allWindows, capturedAt

// 之後(client 給 prompt)
multipart:
  screenshot, userName, activeWindow, appName, allWindows, capturedAt,
  prompt: string,           // client 組好的整段 system + 業務分類塊
  allowedCodes: string[],   // 模板的合法 code 集合(+ "OTHER")
```

server 端 `AnalyzeAsync` 簡化:不再注入 ITemplateService,不再查 config 取 templateId,不再走 prompt 快取。*
*WorkCategoryTemplateService 的 GetPromptBlockAsync + 快取邏輯整段刪掉。**

#### `/api/workcategorytemplate/*`(不變)

管理頁的 CRUD endpoints 保留,給 admin 增刪改模板用。template 詳情走 my-config 一起下發。

### 1.3 SQLite Schema 新增

```ts
// drizzle 0009_work_template_cache.sql
export const workTemplateCache = sqliteTable('work_template_cache', {
  // 永遠只有一行(使用者只會有一個模板),用固定 PK = 1
  id: integer('id').primaryKey(),
  templateId: integer('template_id').notNull(),
  version: integer('version').notNull(),
  // 整份 templateDetail JSON 序列化,本地不關心結構,直接 deserialize 用
  detailJson: text('detail_json').notNull(),
  updatedAt: integer('updated_at').notNull(),
})
```

只一行的設計:單使用者單模板,不用維護多筆。my-config 寫入時 INSERT OR REPLACE id=1。

### 1.4 工時 / 影響

| 變更                                                             | 工時        |
|----------------------------------------------------------------|-----------|
| Server: my-config response 多帶 templateDetail                   | 0.5d      |
| Server: analyze 改吃 client 的 prompt                             | 0.5d      |
| Server: 刪 WorkCategoryTemplateService.GetPromptBlockAsync + 快取 | 0.3d      |
| Desktop: schema migration + service                            | 0.5d      |
| Desktop: 把 RenderPrompt 邏輯從 server 搬過來(prompt-builder.ts)      | 1d        |
| Desktop: 改 analyze 帶 prompt + allowedCodes                     | 0.5d      |
| **合計**                                                         | **~3.3d** |

## 2. 後端代碼整理

### 2.1 當前痛點

```

ichia.Service/Implementation/WorkCategoryTemplateService.cs   ← OK,專職

```

WorkCollectService 內混:

- `GetMyConfigAsync` / `BatchGetConfigsAsync` / `BatchPatchConfigsAsync`(配置)

### 2.2 拆分方案

```

  Implementation/
    WorkCollect/                          ← 新增子目錄,集中採集相關
      ├─ WorkAnalyzeService.cs            ← AI 代理:純 prompt 透傳 → Qwen-VL → 校驗
      ├─ WorkConfigService.cs             ← my-config / batch get / batch patch
      ├─ WorkRecordService.cs             ← sync-daily / my-records / batch query
      ├─ WorkOverviewService.cs           ← admin overview(聚合)+ user list + 員工搜尋
      ├─ WorkCategoryTemplateService.cs   ← 既有,搬進來保持位置一致
      └─ TextSanitizeService.cs           ← 既有(下面 §2.3 重新考慮位置)

  Contract/
    WorkCollect/                          ← 對應接口子目錄
      ├─ IWorkAnalyzeService.cs
      ├─ IWorkConfigService.cs
      ├─ IWorkRecordService.cs
      ├─ IWorkOverviewService.cs

```

**為什麼這麼拆**:每個 service 對應 controller 的一塊,單檔 < 200 行,職責清晰。

它不是 work-collect 專屬(repair / 其它欄位也需要繁化),保留在 `ichia.Service/Implementation/` 根目錄不挪。

## 5. 執行順序

每個 phase 結束後可獨立 build / 測,避免長期半成品狀態。

6. **A6 (desktop)** prompt-builder.ts(從 server 搬邏輯)
7. **A7 (desktop)** analyze 帶 prompt + allowedCodes

**驗證**:重啟 desktop,看 my-config response 包含 templateDetail,analyze 請求帶 prompt,server 沒 SQL 查 template。

## 6. 不做 / 邊界

- ❌ **不上 Kafka / MQ** — 採集量級不需要
- ❌ **不把 Qwen API key 移到 client** — 安全紅線,server 仍是 AI 代理(只是不查 DB)
- ❌ **不重新設計 i18n** — 只清理 8 類過時 key,不換 i18n 框架
- ❌ **不動已建立的 server 端 schema** — 模板表結構維持,只變「誰來查」
- ❌ **不把 store 拆成多個 store** — 一個 useWorkCollectStore 夠用
- ⚠️ **client 組 prompt 是有信任假設的**:client 可以發任意 prompt 給 server。第一版接受(內網企業環境,使用者可信)
  。若日後對外用,要加 prompt 校驗或回到 server-side 組裝

---

## 7. 風險

1. **A1 / A7 之間有時間差**:server 改完接受新欄位,desktop 還沒改完 → analyze 沒帶 prompt → server 應做向後兼容(若 prompt
   缺,降級為現有邏輯)。Phase A 完整跑完再合主幹。

2. **本地 cache 跟 server template 不一致**:管理員 save 模板後,desktop 拉新 my-config 之前用的還是舊 prompt。預期內,desktop
   啟動 / 每天 8 點 pull 一次能容忍。緊急情況可加「強制刷新」按鈕。

3. **舊 desktop 沒升級時,server 端先動了** → analyze 仍能跑(沒 prompt 時 server fallback 到舊邏輯查 DB)。一段時間後再刪
   server 端 prompt building 邏輯。
