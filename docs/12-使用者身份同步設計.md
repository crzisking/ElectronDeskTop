# 使用者身份同步 設計

> 登入後從後端拉取使用者的釘釘 ID(dingId / unionId)+ 基本資料,落地到本機 SQLite。
> **單帳號模型** —— 一台電腦同時只服務一個 AD 帳號;偵測到帳號變更時,清空
> 前一人的所有本機資料(目前涵蓋 `user_profiles` + `work_records`)。
>
> 本文檔取代了原「代辦事項」設計 —— 代辦事項功能取消開發,但身份同步基礎能力獨立保留,
> 提供給工作採集 / 未來其他 per-user 功能共用。

## 1. 動機 + 不做什麼

### 為什麼需要

公司未來可能會做需要釘釘 ID 的功能(訊息推播、代辦同步、組織架構查詢...)。
這些功能都需要先知道「**當前登入的 AD 帳號對應哪個釘釘員工**」。
把這層查詢結果落地到本機 SQLite,讓後續功能直接讀本地、不必每次重打後端。

### 本期做什麼

- 後端加一支 `GET /api/User/info`,從 JWT 解 userName → 查公司目錄 / 釘釘 user.get → 回傳 dingId + unionId + displayName + email
- 桌面端加 `user_profiles` 表,登入後寫入
- AD 帳號變更時:清空 `user_profiles` + `work_records`(以及未來新增的所有 per-user 表)
- 整合進 LoginView + AD 自動登入流程

### 本期**不做**什麼

- ❌ 代辦事項視窗 / 同步 / CRUD(已決定不開發)
- ❌ 任何使用 dingId / unionId 的業務功能(留給未來 feature 各自取用本表)
- ❌ 多帳號切換 UI(維持單帳號模型)
- ❌ user_profiles 之外的個人資料快取(姓名、頭像、組織等暫不存)

## 2. 整體流程

```
App 啟動 → 主進程初始化 DatabaseManager / UserProfileService
   │
   ▼
登入成功(LoginView 或 AD auto-login)→ 拿到 JWT(內含 userName / 工號)
   │
   1. 渲染端 → userProfileStore.syncAfterLogin()
      ├─ 從 JWT 解出 currentUserId(工號)
      ├─ IPC USER_PROFILE_GET_ACTIVE → 拿到 DB 內既有 active row(可能為 null)
      ├─ 比對 existing.userId === currentUserId ?
      │   ├─ 一致     → 沿用既有資料(可選擇是否重抓一次刷新 displayName/email)
      │   └─ 不一致   → AD 帳號變更!IPC ACCOUNT_CHANGED_CLEAR(現任 / 新任 userId)
      │                  主進程清 user_profiles + work_records + 未來其他 per-user 表
      ├─ GET /api/User/info
      │   ├─ 成功 → IPC USER_PROFILE_UPSERT → 寫 user_profiles
      │   └─ 失敗 → store.profileState = 'syncError',UI 顯示「身份同步失敗,部分功能受限」
      └─ store.profileState = 'ready'(或 syncError)
   ▼
其他 feature 需要 dingId 時:
   │
   2. featureStore → IPC USER_PROFILE_GET_ACTIVE → 拿 dingId / unionId
      └─ 或:讀 userProfileStore 內的 reactive cache(若已 ready)
```

**設計重點**:

- **以 JWT 的 userName(工號)為帳號 key** —— 穩定、後端可信、跟現有 auth 鏈路無縫接
- **AD 帳號變更偵測在「登入後」做** —— 此時 JWT 已 valid,可以正確比對;啟動時不做(沒 JWT)
- **失敗不阻塞登入** —— /api/User/info 掛掉只會讓 dingId 拿不到,登入本身仍然成功
- **沿用既有 profile 是預設** —— `existing.userId === currentUserId` 時不強制重打 /api/User/info,顯著降低後端負擔。提供 `forceRefresh` 參數供「手動重整」用

## 3. SQLite Schema

新增 1 張表 `user_profiles`,放在 `electron/main/db/features/user-profile/schema.ts`。

```ts
user_profiles
  userId          TEXT PRIMARY KEY     -- 工號 / JWT subject;單帳號模型下永遠只會有 1 行
  dingId          TEXT NOT NULL        -- 釘釘員工 ID
  unionId         TEXT NOT NULL        -- 釘釘 unionid
  displayName     TEXT                 -- 顯示名,可空
  email           TEXT                 -- email,可空
  syncedAt        INTEGER NOT NULL     -- 上次 /api/User/info 寫入時間 Unix ms
```

**為什麼用 userId 為 PK 而不是 singleton row id**:
- 單帳號模型下確實只會有 1 行,但 PK 用 userId 更清楚表達「這 row 屬於誰」
- 帳號變更時清空靠 `DELETE FROM user_profiles WHERE userId != currentUserId`(其實也可以直接 truncate,選實際做法時看 §6)
- 跨檔查詢時 join 條件 `WHERE userId = ?` 自然安全

**不放在 user_profiles 的資料**:
- ❌ JWT / accessToken —— 機密,只在 renderer Pinia,不持久化
- ❌ 公司部門 / 組織架構 —— 屬於業務查詢結果,需要時即時查,不做本機快取
- ❌ 頭像 URL —— 本期不需要

## 4. 後端契約(tmbom)

新增 **1 個 endpoint**:

| Method | Endpoint | 用途 | 資料來源 |
|---|---|---|---|
| GET | `/api/User/info` | 取當前 JWT 用戶的 dingId + unionId + 基本資料 | 公司 LDAP / 釘釘 user.get(對應後端實作決定) |

### 回傳結構

```jsonc
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "userId":      "S2403279",          // 工號,跟 JWT userName 一致(讓 client 二次校驗)
    "dingId":      "manager0xxxxxxx",
    "unionId":     "xxxxxxxxxxxxxxxx",
    "displayName": "張三",
    "email":       "zhang.san@example.com"
  }
}
```

### 後端責任邊界

| 後端做 | 後端不做 |
|---|---|
| 從 JWT 解 userName(工號) | 不從 query / body 信任 userId |
| 查 LDAP / 釘釘 user.get 拼回傳 | 不存使用者業務資料 |
| **可選的短 TTL cache** | 不做永久落地表 |

### 關於後端 cache 設計

`/api/User/info` 對應的後端查詢可能成本不低(打 LDAP / 釘釘 OpenAPI)。本期建議:

- **後端記憶體做短 TTL cache**(建議 10 分鐘 - 1 小時可配),key = userName
- 桌面端登入時拉一次,寫進本機 user_profiles;**後續桌面端不會頻繁打**,主要靠本地讀
- 若 cache TTL 內被打多次只算一次釘釘 API 用量

避免兩個極端:
- 完全不 cache:每次桌面端登入 → 後端打釘釘 → 浪費 quota
- 後端做永久表:多此一舉,釘釘端就是 source of truth,後端只是中轉

## 5. 登入流程整合

### 5.1 接點

| 觸發 | 位置 | 行為 |
|---|---|---|
| 表單登入成功 | `src/views/Login/LoginView.vue` 內 `onLoginSuccess` | 呼叫 `userProfileStore.syncAfterLogin()` |
| AD 自動登入成功 | `src/stores/auth.store.ts` 內 `loginByAd()` 拿到 token 後 | 同上 |
| 配置熱重載 / 一般 router 跳轉 | — | 不觸發(只在登入這個一次性事件) |

### 5.2 store 對外 API

```ts
// src/features/user-profile/store.ts
export const useUserProfileStore = defineStore('userProfile', () => {
  /** 當前 active profile;沒同步過或沒登入時為 null */
  const profile = ref<UserProfile | null>(null)

  /** 同步狀態,UI 用此判斷顯示什麼 */
  const profileState = ref<'idle' | 'syncing' | 'ready' | 'syncError'>('idle')

  /** 同步錯誤訊息,UI 顯示用 */
  const profileError = ref<string | null>(null)

  /**
   * 登入成功後呼叫。內部:
   *  1. 比對既有 active row,不一致 → 觸發 ACCOUNT_CHANGED_CLEAR
   *  2. GET /api/User/info(失敗時降級顯示)
   *  3. IPC USER_PROFILE_UPSERT 寫入 SQLite
   *  4. 更新 store state
   * @param forceRefresh true 時即使 userId 一致也重打後端(預設 false)
   */
  async function syncAfterLogin(forceRefresh = false): Promise<void> { ... }

  return { profile, profileState, profileError, syncAfterLogin }
})
```

### 5.3 失敗不阻塞登入

`syncAfterLogin` 內部 catch 所有錯誤,寫進 `profileError`,**不對 caller 拋例外**。

換言之:LoginView / AD auto-login 對它的呼叫不需要 try/catch,失敗會在後續用到 dingId 的 feature 內被識別(profileState !== 'ready' → 該 feature 顯示「身份未就緒」)。

## 6. AD 帳號變更:偵測與清空

### 6.1 觸發條件

```
登入後 → currentUserId = JWT.userName
   │
   await electronAPI.userProfile.getActive()  // 讀 user_profiles
   │
   ├─ null                                          → 首次登入,什麼都不清,直接寫
   ├─ existing.userId === currentUserId             → 同一個帳號,直接複用 / 視情況刷新
   └─ existing.userId !== currentUserId             → AD 帳號變更!觸發清空
```

### 6.2 清空範圍

**單帳號模型**:整台機器只服務當前 AD 帳號,前一人留下的所有 per-user 資料**都該清掉**。

| 資料 | 為什麼清 | 怎麼清 |
|---|---|---|
| `user_profiles` | 前一人的 dingId / unionId 已無意義 | `DELETE FROM user_profiles`(truncate) |
| `work_records` | 前一人的工作流水線屬於他的隱私,新使用者不該看到 | `DELETE FROM work_records` |
| **未來新增的 per-user 表** | 同理 | 在 `AccountChangeCleaner` 列表加一行 |
| `logs` | 主進程 / renderer 的日誌,跟業務帳號無關 | **不清**(技術日誌跨帳號保留有助 troubleshoot) |
| `app-config.json` | 應用層配置 / 視窗位置 / 開關 | **不清**(屬於機器設定,跨帳號維持) |

### 6.3 實作架構 — `AccountChangeCleaner`

把「需要清哪些表」收斂到單一服務,避免散落:

```ts
// electron/main/db/account-change-cleaner.ts
export class AccountChangeCleaner {
  constructor(private readonly db: DatabaseManager) {}

  /**
   * 偵測到帳號變更時呼叫,清空所有 per-user 表。
   * 加新 per-user 表時,在這函式內加一行 DELETE,單一 source of truth。
   */
  clearAllUserData(): void {
    this.db.run('DELETE FROM user_profiles')
    this.db.run('DELETE FROM work_records')
    // 未來:DELETE FROM <新表>;
    logger.info('AD 帳號變更,已清空所有 per-user 表', 'AccountChangeCleaner')
  }
}
```

### 6.4 為什麼不靠應用層 filter(WHERE userId = ?)

- 「邏輯隔離」聽起來可行,但只要任何一處 query 漏掉 `WHERE userId`,前一人資料就洩漏
- **物理刪除**最簡單也最安全,符合「單帳號模型」設計初衷
- 同時節省 SQLite 空間 —— per-user 表的歷史不會無限累積

### 6.5 失敗處理

清空動作走交易(transaction):

```ts
this.db.transaction(() => {
  this.db.run('DELETE FROM user_profiles')
  this.db.run('DELETE FROM work_records')
})
```

任一個 DELETE 失敗就 rollback,**不會留下半清空狀態**。

清空失敗的話 main → renderer 推 `PUSH_ACCOUNT_CHANGE_CLEAR_FAILED`,UI 顯示警告。

## 7. 主進程模組

### 7.1 新增檔案

```
electron/main/
├── db/features/user-profile/
│   ├── schema.ts                ← user_profiles drizzle 表定義
│   └── service.ts               ← UserProfileService(upsert / getActive / clear)
└── db/account-change-cleaner.ts ← AccountChangeCleaner(跨表清空)

electron/main/ipc-handlers/
└── user-profile.handlers.ts     ← USER_PROFILE_* + ACCOUNT_CHANGED_CLEAR
```

### 7.2 UserProfileService

```ts
export class UserProfileService {
  /** 取當前 active row(單帳號模型只會有 0 或 1 行) */
  getActive(): UserProfile | null

  /** 寫入或更新(upsert) */
  upsert(profile: Omit<UserProfile, 'syncedAt'>): void

  /** 清空整張表(AccountChangeCleaner 呼叫) */
  clearAll(): void
}
```

### 7.3 啟動時不做帳號變更偵測

主進程啟動時**不主動**讀 user_profiles 比對,因為:
- 啟動時還沒登入 → 沒有 JWT → 沒有 currentUserId 可比
- 觸發時機綁定登入事件最自然

App 啟動只做:`DatabaseManager.init()` 內把 user_profiles 表建出來。

export const UserProfileChannels = {
  /** 取當前 active profile(主進程查 SQLite)。invoke。返回:UserProfile | null */
  USER_PROFILE_GET_ACTIVE:        'user-profile:get-active',

  /** 寫入或更新 profile。invoke。payload:UserProfile。返回:boolean(成功/失敗) */
  USER_PROFILE_UPSERT:            'user-profile:upsert',

  /**
   * 偵測到帳號變更,通知主進程清空所有 per-user 表。

  ACCOUNT_CHANGED_CLEAR:          'user-profile:account-changed-clear',

  /**
   * PUSH:帳號變更清空動作失敗,UI 顯示警告(極少發生,通常 DB 故障才會)。
   * send(main → renderer)
   */
  PUSH_ACCOUNT_CHANGE_CLEAR_FAILED: 'push:account-change-clear-failed',
} as const
```

`PUSH_ACCOUNT_CHANGE_CLEAR_FAILED` 加進 preload 的 `ALLOWED_PUSH_CHANNELS` 白名單。

## 9. 渲染端模組

### 9.1 檔案結構

```

├── api.ts                       ← GET /api/User/info(走 createHttpClient,複用 auth 攔截器)

### 9.2 api.ts

```ts
import {createHttpClient} from '@/api/http-client'
import type {UserProfile} from './types'

// 走通用 API host(USER_INFO 不屬於特定子服務,放主後端)
const USER_API_URL: string =
  (import.meta.env.VITE_API_BASE_URL as string | undefined) ?? 'http://localhost:5247'

let _client: ReturnType<typeof createHttpClient> | null = null
const client = () => (_client ??= createHttpClient(USER_API_URL, 30000))

  /** 取當前 JWT 用戶的釘釘 ID + 基本資料 */
  info: () => client().get<UserProfile>('/api/User/info'),
}
```

> ⚠️ 環境變數選擇待定 —— 若 `/api/User/info` 跟 repair / work-collect 同 tmbom 後端,可考慮複用 `VITE_REPAIR_API_URL`;若是 portal 後端,用 `VITE_API_BASE_URL`。實作時跟後端確認。

### 9.3 store.ts

```ts
export const useUserProfileStore = defineStore('userProfile', () => {
  const profile = ref<UserProfile | null>(null)
  const profileState = ref<'idle' | 'syncing' | 'ready' | 'syncError'>('idle')
  const profileError = ref<string | null>(null)

  /** 是否就緒可供其他 feature 用 */
  const isReady = computed(() => profileState.value === 'ready' && profile.value !== null)

  async function syncAfterLogin(forceRefresh = false): Promise<void> {
    profileState.value = 'syncing'
    profileError.value = null

    try {
      const authStore = useAuthStore()
      const currentUserId = authStore.userId   // 從 JWT 解的工號
      if (!currentUserId) {
        profileState.value = 'syncError'
        profileError.value = '無法解析 JWT 中的工號'
        return
      }

      // 1. 比對既有 active row
      const existing = await window.electronAPI.userProfile.getActive()

      if (existing && existing.userId !== currentUserId) {
        // 帳號變更 → 觸發清空
        const cleared = await window.electronAPI.userProfile.accountChangedClear({
          oldUserId: existing.userId,
          newUserId: currentUserId,
        })
        if (!cleared) {
          logger.error('帳號變更清空失敗', 'UserProfileStore')
        }
      }

      // 2. 如果既有資料且帳號一致且不強制刷新,直接沿用
      if (existing && existing.userId === currentUserId && !forceRefresh) {
        profile.value = existing
        profileState.value = 'ready'
        return
      }

      // 3. 打後端拉最新
      const fresh = await userApi.info()
      await window.electronAPI.userProfile.upsert(fresh)
      profile.value = fresh
      profileState.value = 'ready'
    } catch (err) {
      profileState.value = 'syncError'
      profileError.value = err instanceof Error ? err.message : String(err)
      logger.warn('使用者身份同步失敗', 'UserProfileStore', err)
    }
  }

  return {profile, profileState, profileError, isReady, syncAfterLogin}
})
```

### 9.4 哪裡呼叫 `syncAfterLogin`

```ts
// src/views/Login/LoginView.vue 登入成功 callback
await authStore.login(...)
await useUserProfileStore().syncAfterLogin()  // 不 await 也可,看 UX

// src/stores/auth.store.ts 的 loginByAd() 拿到 token 後
authStore.setToken(jwt)
await useUserProfileStore().syncAfterLogin()
```

兩處共用同一個 `syncAfterLogin`,失敗都不阻塞主流程。

## 10. 錯誤處理與 UI 狀態

| 狀態 | profileState | 觸發 | UI 行為 |
|---|---|---|---|
| 未登入 / 未呼叫 | `idle` | 初始 | 任何依賴 dingId 的 feature 顯示「請先登入」 |
| 同步中 | `syncing` | syncAfterLogin 開始 | 該 feature 顯示 loading skeleton |
| 已就緒 | `ready` | /api/User/info 成功 + DB 寫入成功 | 正常顯示 |
| 同步失敗 | `syncError` | /api/User/info 失敗 / DB 寫失敗 / JWT 解析失敗 | 該 feature 顯示「身份同步失敗,部分功能受限」+ 重試按鈕 |

### 10.1 跨 feature 共用模式

未來任何需要 dingId 的 feature 在自己的 store 內:

```ts
const userProfile = useUserProfileStore()

if (!userProfile.isReady) {
  // 顯示降級 UI,引導使用者點重試
  return
}

const dingId = userProfile.profile!.dingId
// 用 dingId 打 feature 接口...
```

### 10.2 重試入口

未來在 SettingsView 或頂部 TitleBar 加「同步身份」按鈕,呼叫 `syncAfterLogin(true)` 強制重抓。
本期 MVP 可以不加 UI,失敗時只在 console 看 log;有實際需求再補。

## 11. 資料邊界

| 資料 | 位置 | 保留期 |
|---|---|---|
| dingId / unionId / displayName / email | userData/app.db / user_profiles | 帳號未變則永久保留 |
| JWT(含 userName) | 渲染端 Pinia(`authStore.accessToken`) | 不持久化,跟現行 auth 一致 |
| 釘釘 corpId + accessToken | **後端記憶體** | 後端內部 cache,前端永不接觸 |
| 後端 user-info 短 TTL cache | **後端記憶體** | 10 分鐘 - 1 小時,可配置 |
