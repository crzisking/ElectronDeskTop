## 2. Pinia Store 設計

### `config.store.ts` — 應用配置 Store

使用 **Setup Store** 寫法（`defineStore('config', () => { ... })`，與 `auth.store.ts`、`ui.store.ts` 一致）。

**State（`ref`）：**

- `appConfig: Ref<AppConfig | null>` — 從 `app-config.json` 讀取的完整配置
- `isLoaded: Ref<boolean>` — 配置是否已載入完成
- `loadError: Ref<string | null>` — 載入失敗時的錯誤信息

**Getters（`computed`）：**

- `sidebarItems` — 過濾 `enabled: false` 後的側邊欄項
- `sidebarDefaultCollapsed` — 應用啟動時側邊欄是否預設折疊
- `floatingBallMenuItems` — 浮球快捷菜單項（已過濾 `enabled: false`）
- `floatingBallConfig` — 浮球完整配置（`size` / `opacity` / `defaultPosition` …）
- `platformSystems` — 統一平台的系統列表
- `functionsConfig` — 內部功能配置（`apiBaseUrl` / `apiTimeout` / `tools`），對應 JSON 的 `internalFunctions`
- `language` — 當前語言設定

**Actions：**

- `loadConfig()` — 調用 `window.electronAPI.config.read()` 載入配置
- `writeConfig(partial)` — 深合併後寫回磁盤，並重新 `loadConfig()` 保持同步

> **命名提醒**：內部功能配置的 getter 叫 `functionsConfig`，對應 JSON 的 `internalFunctions` 鍵
> （歷史上曾叫 `aiConfig` / `aiQuickFunctions`，已統一改名）。

---

### `auth.store.ts` — 認證 Store

使用 **Setup Store** 寫法（`defineStore('auth', () => { ... })`）。

**State：**

- `isAuthenticated: Ref<boolean>` — 是否已登錄
- `user: Ref<UserProfile | null>` — 當前用戶信息
- `accessToken: Ref<string | null>` — Token（僅內存，不持久化到磁盤）
- `isRestoringSession: Ref<boolean>` — 已棄用、保留兼容（過去用於磁盤恢復會話期間放行路由守衛；現在 Token 不持久化，啟動時必走登入頁）

**Actions：**

- `login(userName, password)` — 調用登錄 API，將 Token 存入內存
- `logout()` — 清空所有內存狀態（路由守衛自動跳 `/login`）

> **重要**：`login()` 調用 `authApi.login()` 後直接解構 `{token, user}`，不要讀 `.code` 或 `.data`。原因見 `docs/開發記錄/2026-04-15.md` 的數據流說明。
>
> **Token 不持久化**：應用每次啟動 `accessToken` 均為 `null`，用戶必須重新登入；不再有「啟動恢復會話」流程。

---

### `ui.store.ts` — UI 狀態 Store

使用 **Setup Store** 寫法。存放不需要持久化的界面狀態：

- `sidebarCollapsed: Ref<boolean>` — 側邊欄是否折疊
- `isWindowMaximized: Ref<boolean>` — 窗口是否最大化（由主進程 `push:window-maximized` 推送同步）
- `globalLoading: Ref<boolean>` — 全局加載遮罩（應用初始化期間顯示）
- `toggleSidebar()` / `setSidebarCollapsed(val)` — 切換／設置側邊欄
- `setWindowMaximized(val)` — 更新最大化狀態
- `hideGlobalLoading()` — 隱藏初始加載遮罩

## 3. API 層設計

### HTTP Client 工廠（`http-client.ts`）

調用 `createHttpClient(baseURL, timeout?)` 創建一個帶攔截器的 Axios 實例。每個 API 模塊懶初始化自己的客戶端（第一次調用時創建，後續複用同一個實例）。

`baseURL` 從環境變量讀取，不寫死在代碼裡。

---

### Auth 攔截器（`auth.interceptor.ts`）

同一個攔截器同時處理請求注入和響應錯誤：

**請求攔截（注入 Token）：**

- 從 `authStore.accessToken` 讀取 Token
- 注入 `Authorization: {token}`（注意：直接帶 Token 字串，**不加** `Bearer ` 前綴，與公司 Portal 後端規格一致）

**響應攔截（錯誤處理）：**

- HTTP 200 且業務碼非 200 → `ElMessage.error` 顯示錯誤，但不拋出異常（讓調用方繼續）
- HTTP 401 → `ElMessageBox` 提示「登錄已過期」→ 自動 logout + 跳 `/login`
- HTTP 403 → `ElMessageBox` 提示「無權限」
- HTTP 404 → `ElMessageBox` 提示「資源不存在」
- 網絡錯誤 → `ElMessage.error` 提示「網絡異常」
- 正常響應返回 `response.data`（即後端 JSON body `{code, message, data}`）

---

### API 模塊命名規範

每個 API 模塊對應一組後端接口，文件名格式 `*.api.ts`，使用工廠函數暴露：

- `auth.api.ts` — 登錄接口，使用 `VITE_AUTH_BASE_URL`
- `repair.api.ts` — IT 報修工單（create、list、detail、uploadFile），使用 `VITE_REPAIR_API_URL`
- `ai.api.ts` — AI 通用接口（Dify SSE 共用底層調用）
- `aisop.api.ts` — AiSop 生成上傳接口，使用 `VITE_AI_SOP_URL`

## 4. TypeScript 規範

### 強制規則

| 規則 | 說明 |
|------|------|
| 禁止 `any` | 所有類型必須明確，未知類型用 `unknown` 替代 |
| 嚴格 null 檢查 | 可能為 null/undefined 的地方必須顯式處理 |
| 接口優先 | 數據結構用 `interface`，工具類型用 `type` |
| 枚舉避用 | 用 `const` 對象 + `typeof` 替代 `enum`（防止運行時膨脹問題）|
| 導入類型 | 只用於類型的導入用 `import type` |

### 類型聲明位置

| 文件 | 存放什麼 |
|------|---------|
| `src/types/config.types.ts` | AppConfig 及其所有子類型 |
| `src/types/api.types.ts` | 所有 API 請求/響應類型（彙總 re-export） |
| `src/types/repair.types.ts` | IT 報修工單相關類型（RepairDetail 等） |
| `src/types/auth.types.ts` | 登錄/使用者相關類型 |
| `src/types/ai.types.ts` | AI 流式接口類型 |
| `src/types/electron.d.ts` | `window.electronAPI` 的完整類型聲明 |
| `src/vite-env.d.ts` | `import.meta.env.VITE_*` 的類型聲明 |

## 5. Vue 組件規範

### SFC 文件組織順序

```
<script setup lang="ts">
  // 1. 外部包導入（vue, vue-router, element-plus 等）
  // 2. 內部模塊導入（@/stores, @/api, @/types 等）
  // 3. Store 初始化
  // 4. Props / Emits 定義
  // 5. 響應式數據（ref, reactive）
  // 6. 計算屬性（computed）
  // 7. 業務方法
  // 8. 生命週期鉤子（onMounted, onUnmounted 等）
</script>

<template>
  <!-- 模板：根元素只有一個 -->
</template>

<style scoped>
  /* 所有樣式 scoped，避免全局污染 */
  /* 注意：動態創建 DOM 的第三方庫（如 Quill）的樣式需用非 scoped 全局樣式覆蓋 */
</style>
```

### 命名規範

| 類型         | 規範                    | 示例                                   |
|------------|-----------------------|--------------------------------------|
| 組件文件       | PascalCase            | `SystemCard.vue`                     |
| 視圖文件       | PascalCase + View 後綴  | `LoginView.vue` / `ITRepairView.vue` |
| Composable | camelCase + use 前綴    | `useRepairSubmit.ts`                 |
| Store      | camelCase + .store 後綴 | `auth.store.ts`                      |
| API 模塊     | camelCase + .api 後綴   | `repair.api.ts`                      |
| CSS 類名     | kebab-case            | `.system-card`                       |
| IPC 頻道常量   | SCREAMING_SNAKE_CASE  | `IpcChannels.CONFIG_READ`            |

### Composable 拆分原則

當一個 Vue 組件的 `<script setup>` 超過 150 行時，考慮按業務邏輯職責拆分為 Composable。

拆分示例（IT 報修）：

- `useRepairSubmit.ts` — 表單、Quill 編輯器生命週期、圖片上傳、AI 潤色
- `useRepairTickets.ts` — 工單列表分頁、狀態篩選、詳情彈窗

視圖組件（ITRepairView.vue）只負責組裝，不含業務邏輯。

### 編寫約定：`el-radio-group` 的「全部」哨兵值

當 `el-radio-group` / `el-radio-button` 需要一個「全部／不過濾」選項時，不要用 `undefined`
（Element Plus 會觸發 `label act as value is about to be deprecated` 告警，
且內部會退回到已廢棄的 label-as-value 路徑）。

做法：
1. 使用 `0`（或其他不與合法狀態碼衝突的值）作為哨兵，`:value="0"` 綁到「全部」按鈕。
2. 在呼叫後端前映射回 `undefined`，對後端語義仍是「不傳該篩選參數」。

參考實作：`src/views/InternalFunctions/child/ITRepair/composables/useRepairTickets.ts` 的
`ticketParams.status` 與 `loadTickets()`。

## 7. 打包配置說明

打包配置在 `package.json` 的 `build` 字段（electron-builder 讀取）：

| 配置項 | 值 | 說明 |
|--------|-----|------|
| `appId` | `com.ichia.desktop.client` | 應用唯一標識 |
| `productName` | `ichiaDesktop` | 安裝包和應用顯示名 |
| `directories.output` | `dist/` | 安裝包輸出目錄 |
| `files` | `out/**/*` + `config/app-config.json` | 打包進安裝包的文件 |
| `extraResources` | `app-config.json`、`resources/` | 解包到 app 資源目錄 |
| `asarUnpack` | `[]`（空） | 不需要解包的文件（當前無 Native Addon） |

Windows 打包產出兩種格式：
- **NSIS 安裝程序**：有安裝嚮導，支持選擇安裝目錄，創建桌面快捷方式
- **便攜版（portable）**：無需安裝，直接雙擊運行

> **重要**：`build:win` 打包時使用的是 `.env.production` 裡的環境變量（API 地址等），不是 `.env.development`。打包前請確認 `.env.production` 的地址都是正式環境地址。
