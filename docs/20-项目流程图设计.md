### 0.2 核心设计原则

1. **三个模块各自独立** — 汇报不是"对着节点打卡"，备忘录不是"项目子任务"，每个模块都可以单独使用
2. **关联是可选的** — 汇报可以关联项目节点（"今天完成了 XX 项目的 YY 任务"），也可以不关联
3. **项目编辑权在管理端** — 谁能创建/编辑/删除项目和节点，由管理端统一配置，大部分员工只有查看权
4. **AI 聚合多源数据** — 写汇报/生成建议时，AI 自动拉取 work-collect 数据 + 节点变更 + 备忘录完成情况
5. **主管和员工同端** — 都在桌面端操作，主管通过"团队视图"切换视角

### 0.3 用户角色与权限

| 角色         | 能力                            | 权限来源        |
|------------|-------------------------------|-------------|
| **员工(預設)** | 查看自己被分配的項目和節點、匯報進度、撰寫匯報、使用備忘錄 | 默認          |

## Part 1: Module A — 項目管理

### 1.1 功能定位

項目是員工工作的"計劃視圖"。它用流程圖畫布展示一個項目的所有任務節點及其依賴關係，幫助員工和主管快速了解項目全貌和當前進度。

**誰能編輯**: 只有管理端配置的"項目編輯者"可以創建/修改/刪除項目和節點。

**誰能查看**: 項目負責人（OwnerUserId）、被分配節點的員工、主管都可以查看。普通員工不能查看自己不參與的項目。

關鍵交互：

- **畫布**: @antv/x6 實現，支持拖拽平移、滾輪縮放、節點拖放、連線
- **節點**: 顯示標題 + 狀態色圓點，點擊選中後右側面板顯示屬性
- **工具列**: 僅項目編輯者可見「新增節點」「刪除」按鈕

- **AI 項目總結**: 按鈕觸發，拉取項目所有節點數據，生成結構化進度報告

```

| 狀態          | 顏色        | 說明   |
|-------------|-----------|------|
| not_started | 灰 #909399 | 尚未開始 |

### 1.4 節點類型

| 類型        | 說明   | 用途       |

| task      | 普通任務 | 默認類型     |

| decision  | 決策點  | 需要做判斷的節點 |

## Part 2: Module B — 工作匯報

### 2.1 功能定位

工作匯報是員工向主管匯報工作的**獨立溝通工具**。它不是"對著項目節點打卡"，而是員工自主撰寫的日報/周報。AI
可以從多個數據源輔助生成草稿，員工編輯確認後發送。

### 2.2 與項目的關係

匯報**可以**關聯項目和節點，但不是必須的：

- 員工今天做了項目 A 的工作 → 可以關聯到項目 A 的相關節點
- 員工今天處理了零散的 IT 問題 → 不需要關聯任何項目
- 一條匯報可以關聯多個項目/節點（多對多）

#### AI 輔助生成流程

```
員工點擊「AI 輔助生成」
  → 桌面端收集上下文:
    1. 當日 work-collect 記錄（分類、描述、時間分佈）
    2. 員工參與的項目節點狀態變更（今日內）
    3. 今日已完成的備忘錄
    4. 主管最新的未處理反饋
  → 發送到後端 POST /api/ProjectFlow/ai/report-generate
  → 後端調用 OpenAI SDK + JSON Mode
  → 返回結構化匯報草稿
  → 前端填充到編輯面板
  → 員工檢查、修改、補充
  → 可選關聯項目/節點
  → 保存草稿 或 發送給主管
```

**AI 生成 Prompt 設計要點**:

- 系統提示詞定義角色：專業的工作匯報助手
- 輸入數據包括：work-collect 記錄摘要、節點變更、備忘錄、反饋
- 輸出格式：`{ todayWork, completedItems, issues, tomorrowPlan, aiNote }`
- 強制 JSON 輸出，temperature 0.3
- aiNote 欄位告訴員工"以上內容基於什麼數據生成，請核實"

## Part 3: Module C — 智能備忘錄

### 3.1 功能定位

智能備忘錄是員工的**個人智能待辦清單**。它既可以手動創建，也能由 AI 從多源數據中自動生成建議。備忘錄可以選擇性關聯到項目/節點，但不依賴項目模塊。

### 3.2 數據來源

| 來源               | 說明                | 範例                         |
|------------------|-------------------|----------------------------|
| 手動創建             | 員工自己添加            | "記得週五前提交報銷單"               |
| AI 根據項目 deadline | 節點即將到期            | "項目 A 的部署節點明天截止"           |
| AI 根據主管反饋        | 主管交代的事            | "主管提到要更新安全模組文檔"            |
| AI 根據工作模式        | 從 work-collect 分析 | "本週 code review 時間偏低，建議安排" |

### 3.3 交互流程

```
┌─────────────────────────────────────────────────────────────┐
│  智能備忘錄                            [+ 新建備忘]           │
│                                                              │
│  篩選: [全部 ▼] [待完成] [已完成] [按優先級 ▼]                │
├────────────────────────────────┬────────────────────────────┤
│  我的備忘                       │  AI 建議區                  │
│                                │                            │
│  ┌────────────────────────┐   │  ┌──────────────────────┐  │

### 3.4 備忘錄與其他模塊的關聯

- **可選關聯項目/節點**: 備忘可軟關聯到某個項目的某個節點，方便追蹤上下文

### 4.3 主管反饋機制

主管可以在兩個地方留下反饋：

1. **對項目節點**: 點擊節點 → 右側面板 → 底部反饋區
2. **對匯報**: 展開匯報詳情 → 底部反饋區

反饋發送後：

- 員工在桌面端收到紅點通知（頂部導航欄反饋圖標）
- 員工點擊可查看反饋詳情，跳轉到對應的節點或匯報
- 反饋標記已讀後紅點消失

## Part 6: 模塊串聯設計

### 6.1 數據如何在模塊間流轉

```
場景: 員工的一天

08:00  桌面端啟動，work-collect 開始後台採集
09:00  員工打開備忘錄 → 看到 AI 昨晚生成的建議
       "項目 A 部署節點明天截止，建議今天優先處理" → 接受
10:00  員工打開項目畫布 → 查看節點狀態 → 把「部署」節點從 not_started 改為 in_progress
       (如果員工是項目編輯者，直接修改；否則只能查看)
11:00  員工在備忘錄中勾選完成「確認部署方案」
14:00  員工寫了一個節點進度更新（在節點屬性面板 > 進度時間線記錄）
16:30  員工打開匯報 → 點擊「AI 輔助生成」
       AI 聚合:
       - work-collect: 今天寫了 8 小時代碼，主要用 VS Code
       - 項目節點變更: 「部署」節點狀態變更 not_started → in_progress
       - 備忘錄完成: 「確認部署方案」已完成
       → 生成匯報草稿:
         "今日工作: 1. 開始部署方案實施 [關聯: 項目A > 部署]
          遇到的問題: 部署服務器權限未開通，需要 IT 協助
          明日計劃: 聯繫 IT 開通權限後繼續部署"
16:45  員工檢查草稿，修改補充 → 點擊「發送給主管」
17:00  主管收到通知 → 打開團隊視圖 → 匯報 Tab → 查看
       → 留下反饋: "已聯繫 IT，明天應該能開通。另外注意部署文檔要同步更新。"
17:30  員工在匯報列表中看到紅點 → 點擊查看反饋 → 已讀
       同時，AI 自動分析主管反饋 → 生成備忘建議:
       "更新部署文檔" → 員工第二天看到
```

### 6.2 AI 聚合的數據源

當 AI 需要生成內容時，它從以下來源拉取數據：

```
AI 匯報生成:
  輸入:
    - work-collect 當日記錄（類別分佈、應用使用、時間段）
    - 員工參與的項目節點今日狀態變更
    - 今日完成的備忘錄
    - 主管最新的未讀反饋
  輸出: { todayWork, completedItems, issues, tomorrowPlan }

AI 項目總結:
  輸入:
    - 項目所有節點（狀態、截止日、進度歷史）
    - 節點負責人資訊
  - 最近 N 天的節點進度記錄
  輸出: { overallStatus, summary, progress, highlights, blockers, upcomingDeadlines, suggestedActions }

AI 備忘建議:
  輸入:
    - 所有項目中該員工被分配的節點（特別關注 deadline 臨近的）
    - 主管反饋（未處理的）
    - 近期 work-collect 模式分析（哪些類型工作時間偏高/偏低）
    - 已有備忘錄（避免重複建議）
  輸出: [{ title, description, priority, dueDate, reasoning }]

AI 團隊總結:
  輸入:
    - 所有下屬的項目進度摘要
    - 所有下屬的最新匯報
    - 阻塞節點集合
  輸出: { overallProgress, highlights, blockers, suggestedActions }
```

### 6.3 通知流

| 事件             | 通知對象  | 通知方式             |
|----------------|-------|------------------|
| 主管給項目節點留反饋     | 節點負責人 | 導航欄紅點 + 反饋列表     |
| 主管給匯報留反饋       | 匯報作者  | 導航欄紅點 + 匯報列表標記   |
| 員工發送匯報         | 主管    | 團隊視圖中該下屬標記「有新匯報」 |
| AI 生成新建議       | 員工    | 備忘錄頁 AI 建議區刷新    |
| 節點 deadline 臨近 | 節點負責人 | 項目列表卡片標記         |

## Part 9: 驗證

1. **後端**: SQL Server 2008 建表 → Swagger 測試全部 API
2. **權限**: 普通員工不能建項目 → 項目編輯者可以 → 主管只能查看
3. **匯報**: 手動撰寫 → AI 輔助生成 → 關聯/不關聯項目 → 發送 → 主管查看
4. **備忘錄**: 手動創建 → AI 建議 → 接受/忽略 → 標記完成
5. **團隊視圖**: 主管看下屬項目(只讀)、匯報、備忘 → 留反饋 → 員工收到紅點
6. **端到端**: 編輯者建項目 → 員工匯報進度 → 員工寫匯報(AI輔助) → 主管查看留反饋 → 員工回應 → AI 生成新建議
