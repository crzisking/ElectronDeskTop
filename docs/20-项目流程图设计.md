## TL;DR — 跟既有系統的整合決策

下列是設計層面的「**怎麼跟現有 desktop / tmbom 銜接**」拍板結論,後面章節依此展開:

| 主題                             | 決策                                                                                               |
|--------------------------------|--------------------------------------------------------------------------------------------------|
| **AI 匯報生成**                    | **完全 desktop 本地**(複用既有 `LlmClient`,跟 work-analysis 同路徑)。**後端不提供** `/ai/report-generate` 端點       |
| **AI 項目總結 / 團隊總結**             | **後端**(主管 / 跨人資料聚合,desktop 拿不到別人的)                                                               |
| **AI 備忘建議**                    | **desktop 本地**(資料源全在 desktop:work-collect / 自己備忘 / 主管反饋 cache)                                   |
| **AI 配額**                      | **分配額**:匯報生成 10/天、項目總結 5/天(per project)、團隊總結 5/天、備忘建議 1 次/2h                                     |
| **匯報「發送給主管」**                  | desktop POST 到後端 → 落 DB(支撐 Module D)→ 後端寄 email 通知主管                                             |
| **跨端通知**(反饋紅點 / AI 新建議 / 匯報送達) | **複用 docs/18 的 SignalR Hub**,新增一個 `notification` action 子類型,不另起 channel                          |
| **work-collect 資料給 AI 用**      | desktop 本地直接讀 SQLite,不走 server 也不需上傳                                                             |
| **離線寫入**                       | **不支援**。所有寫操作需線上,離線只能讀本地 cache(SQLite 移除 isDirty 欄位避免誤導)                                         |
| **編輯權限**                       | 單欄 `EnumLevel`(0=none/1=own/2=dept/3=global),`(UserId)` UNIQUE,程式碼 `if (level >= required)`      |
| **匯報 schema**                  | 拆兩張:`Project_Reports`(meta)+ `Project_ReportItems`(work/issue/plan 各為一行)— 統計可走 SQL,不用 parse JSON |
| **反饋表**                        | 統一 `Project_Feedback (TargetType, TargetId, ...)` — 不為每個目標類型各開一張表                                |

### 0.2 核心设计原则

1. **三个模块各自独立** — 汇报不是"对着节点打卡"，备忘录不是"项目子任务"，每个模块都可以单独使用
2. **关联是可选的** — 汇报可以关联项目节点（"今天完成了 XX 项目的 YY 任务"），也可以不关联
3. **项目编辑权在管理端** — 谁能创建/编辑/删除项目和节点，由管理端统一配置，大部分员工只有查看权
4. **AI 聚合多源数据** — 写汇报/生成建议时，AI 自动拉取 work-collect 数据 + 节点变更 + 备忘录完成情况
5. **主管和员工同端** — 都在桌面端操作，主管通过"团队视图"切换视角

### 0.3 用户角色与权限

| 角色         | 能力                                     | 权限来源                |
|------------|----------------------------------------|---------------------|
| **員工(預設)** | 查看自己被分配的項目和節點、匯報節點進度、撰寫匯報、使用備忘錄        | 默認(EditLevel=0)     |
| **項目編輯者**  | 員工權限 + 創建項目 + 編輯/刪除「Owner==self」的項目和節點 | 管理端配置 EditLevel=1   |
| **部門編輯者**  | 員工權限 + 編輯所屬部門所有項目                      | 管理端配置 EditLevel=2   |
| **全局編輯者**  | 員工權限 + 編輯所有項目(跨部門)                     | 管理端配置 EditLevel=3   |
| **主管**     | 上述任一 + 查看下屬的項目/匯報/備忘錄 + 留下反饋           | Project_Supervisors |
| **管理員**    | 配置主管關係 + 配置編輯權限 + 管理模板                 | tmbomweb 操作         |

## Part 1: Module A — 項目管理

### 1.1 功能定位

項目是員工工作的"計劃視圖"。它用流程圖畫布展示一個項目的所有任務節點及其依賴關係，幫助員工和主管快速了解項目全貌和當前進度。

**誰能編輯**: 只有管理端配置的"項目編輯者"可以創建/修改/刪除項目和節點。

**誰能查看**: 項目負責人（OwnerUserId）、被分配節點的員工、主管都可以查看。普通員工不能查看自己不參與的項目。

## Part 2: Module B — 工作匯報

### 2.1 功能定位

工作匯報是員工向主管匯報工作的**獨立溝通工具**。它不是"對著項目節點打卡"，而是員工自主撰寫的日報/周報。AI
可以從多個數據源輔助生成草稿，員工編輯確認後發送。

### 2.2 與項目的關係

匯報**可以**關聯項目和節點，但不是必須的：

- 員工今天做了項目 A 的工作 → 可以關聯到項目 A 的相關節點
- 員工今天處理了零散的 IT 問題 → 不需要關聯任何項目
- 一條匯報可以關聯多個項目/節點（多對多）

#### AI 輔助生成流程(**desktop 本地,不走後端**)

```
員工點擊「AI 輔助生成」
  → 桌面端 ReportEditor 直接收集上下文(全部本地,無 IPC 跨進程聚合):
    1. 當日 work-collect 記錄(從本地 SQLite work_records 撈)
    2. 員工參與的項目節點狀態變更(本地 project_cache.json 加上 server 同步的最新狀態)
    3. 今日已完成的備忘錄(本地 memo_cache)
    4. 主管最新的未處理反饋(feedback store)
  → 呼叫 desktop 端既有的 LlmClient(走 OpenAI / Qwen 配置,跟 work-analysis 同一個 LlmClient 實例)
  → JSON Mode 輸出結構化匯報草稿(schema 見 §7.4)
  → 前端填充到編輯面板
  → 員工檢查、修改、補充
  → 可選關聯項目/節點
  → 保存草稿 或 發送給主管
```

**為什麼 desktop 本地**:

- work-collect 原始記錄在 desktop 本地 SQLite,不在 server。後端做 = 還要先把資料上傳,多一個延遲跟流量
- 跟 work-analysis 已建立的 LlmClient 路徑一致,不重複造輪子
- API Key / Provider config 已經在 desktop 端 agent_configs 表

**配額管理**:跟 work-analysis 共用 LlmClient,但**獨立配額計數**(避免一個吃光另一個)

- 匯報生成 **10 次/天**(寫日報 / 周報 + 修改 / 重生需求)
- 後端在 `Project_AiQuota` 表(per UserId / 動作類型)維護計數;desktop 呼叫前先 IPC 拿配額

**AI 生成 Prompt 設計要點**:

- 系統提示詞定義角色:專業的工作匯報助手
- 輸入數據:work-collect 記錄摘要 + 節點變更 + 備忘錄完成情況 + 主管反饋
- 輸出格式:`{ todayWork, completedItems, issues, tomorrowPlan, aiNote }`
- 強制 JSON 輸出,temperature 0.3
- aiNote 欄位告訴員工「以上內容基於什麼數據生成,請核實」

**「發送給主管」**(這一步走後端):

- desktop 編輯完成 → POST `/api/ProjectFlow/reports` 落 Project_Reports + Project_ReportItems
- POST `/api/ProjectFlow/reports/{id}/submit` 標記為已送出
- server 收到後:
    1. 把 status 設 'submitted'
    2. 透過 SignalR Hub 推給主管 desktop(若在線),主管立刻看到紅點
    3. 同時寄 email 通知主管(離線兜底)

## Part 3: Module C — 智能備忘錄

### 3.1 功能定位

智能備忘錄是員工的**個人智能待辦清單**。它既可以手動創建，也能由 AI 從多源數據中自動生成建議。備忘錄可以選擇性關聯到項目/節點，但不依賴項目模塊。

### 3.2 數據來源

| 來源               | 說明                | 範例                         |

| 手動創建             | 員工自己添加            | "記得週五前提交報銷單"               |
| AI 根據項目 deadline | 節點即將到期            | "項目 A 的部署節點明天截止"           |
| AI 根據主管反饋        | 主管交代的事            | "主管提到要更新安全模組文檔"            |
| AI 根據工作模式        | 從 work-collect 分析 | "本週 code review 時間偏低，建議安排" |

### 6.1 數據如何在模塊間流轉


### 6.2 AI 聚合的數據源 + 執行位置

| AI 功能       | 執行位置                        | 配額                 | 資料源                                                |
|-------------|-----------------------------|--------------------|----------------------------------------------------|
| **AI 匯報生成** | **desktop 本地**(`LlmClient`) | 10/天               | 全本地:work-collect / project cache / memo / feedback |
| **AI 項目總結** | 後端                          | 5/天 per project    | 後端 Project_Nodes / NodeProgress                    |
| **AI 備忘建議** | desktop 本地                  | 1 次/2h             | 本地:project cache + memo + work-collect 模式          |
| **AI 團隊總結** | 後端                          | 5/天 per supervisor | 後端聚合下屬匯報 / 節點狀態                                    |

```
AI 匯報生成 (desktop 本地):
  輸入(全部本地讀,無 server 往返):
    - work-collect 當日記錄(本地 SQLite work_records WHERE capturedAt >= 今日 0:00)
    - 員工參與的項目節點今日狀態變更(project_cache 內 NodeProgress 時間線)
    - 今日完成的備忘錄(memo_cache)
    - 主管最新的未讀反饋(feedback store)
  輸出: { todayWork, completedItems, issues, tomorrowPlan, aiNote }

AI 項目總結 (後端):
  輸入:
    - 項目所有節點(狀態、截止日、進度歷史) — Project_Nodes / Project_NodeProgress
    - 節點負責人資訊 — Project_Nodes.AssigneeUserId
    - 最近 N 天的節點進度記錄
  輸出: { overallStatus, summary, progress, highlights, blockers, upcomingDeadlines, suggestedActions }
  cache: 同 ProjectId 4 小時內無節點狀態變更 → 直接回 Project_AiSummaries cache,不重打 AI

AI 備忘建議 (desktop 本地):
  輸入(全部本地):
    - 員工被分配節點(本地 project_cache 內所有 AssigneeUserId == self 的節點)
      特別關注 Deadline <= 今日 + 7 天 且 Status != completed 的
    - 主管反饋(未處理) — feedback store
    - 近期 work-collect 模式分析(哪些類別偏高 / 偏低)
    - 已有備忘錄(避免重複建議)
  輸出: [{ title, description, priority, dueDate, reasoning }]

AI 團隊總結 (後端):
  輸入:
    - 所有下屬的項目進度摘要 — 跨 user join Project_Projects / Nodes
    - 所有下屬的最新匯報 — Project_Reports + Project_ReportItems
    - 阻塞節點集合 — Project_Nodes WHERE Status='blocked'
  輸出: { overallProgress, highlights, blockers, suggestedActions }
  cache: 4 小時 + 沒有新匯報 → 命中
```

### 6.3 通知流

**傳輸機制:複用 docs/18 既有 SignalR Hub**(`/hubs/notifications`),不另起新 channel。

server 對接點:後端 `Project_FeedbackService` / `Project_ReportService` 在資料變更時呼叫既有
`INotificationSender.SendTaskToConnectionsAsync()`,payload 多帶一個 `action` 子類型(對齊 docs/18 ScriptRunner action
概念):

| 事件             | 通知對象   | SignalR action                  | desktop 端處理                           |
|----------------|--------|---------------------------------|---------------------------------------|
| 主管給節點留反饋       | 節點負責人  | `project-flow.feedback-new`     | 導航欄紅點 +1,儲存到 feedbackStore.unread     |
| 主管給匯報留反饋       | 匯報作者   | `project-flow.feedback-new`     | 同上(payload.targetType 區分 node/report) |
| 員工發送匯報         | 該員工的主管 | `project-flow.report-submitted` | 團隊視圖紅點 + 該下屬卡片標「有新匯報」                 |
| 節點 deadline 臨近 | 節點負責人  | 不走 SignalR                      | desktop 本地 cron 檢查 cache,本地計算紅點       |
| AI 備忘建議刷新      | 員工自己   | 不需 — desktop 本地觸發               | UI 自動更新                               |

**為什麼 deadline 走本地計算**:

- deadline 是純時間判定,desktop 本地有 project_cache,自己定時掃比 server 推省事
- 避免 server 跑個 cron 半夜推訊息打擾(時區還要考慮)

**為什麼 AI 備忘建議不走 SignalR**:

- 它是 desktop 本地生成的,當下 UI 直接刷新 state,沒有跨進程訊號需要

### 7.4 AI 集成方案

兩條路徑共存:

**路徑 A — desktop 本地 LLM**(匯報生成 / 備忘建議)

```
桌面端 → 既有 LlmClient(走 agent_configs 內配置的 OpenAI / Qwen)→ JSON Mode
         → 返回結構化 JSON → 桌面端直接 fill 表單

呼叫前先 POST /ai/quota/consume 扣配額,server 端維護 Project_AiQuota
```

**路徑 B — server 後端 LLM**(項目總結 / 團隊總結)

```
桌面端 → 後端 API → server-side OpenAI SDK → JSON Mode → server 落 Project_AiSummaries cache
         → 返回給桌面端
```

**為什麼分兩路**:

- 路徑 A 的輸入資料(work-collect / project cache / memo / 反饋)**全在 desktop**,server 拿不到 → 必須本地跑
- 路徑 B 的輸入資料(跨節點、跨人匯報)**跨 user**,desktop 個人拿不到別人資料 → 必須 server 跑
- **API Key 分離**:server LLM 用 server 端 OpenAI Key(放 appsettings);desktop LLM 用 agent_configs 內個人 / 公司配的
  Key(已有機制)
- 配額**統一在 server 計**(Project_AiQuota),即使 desktop 端 LLM 也要先扣再跑 → 避免 desktop 端被竄改繞過配額

### 7.7 本地 SQLite (桌面端緩存 — **純讀加速,不支援離線寫**)

**設計決策**:寫操作(改節點、發匯報、新增備忘)一律要線上。離線時 UI 提示「網路未就緒」,等使用者上線再操作。本地 cache
純為「讀加速 / AI 本地拿資料」存在。

故 schema 移除 `isDirty` 欄位(會誤導以為支援離線寫):

```typescript
// electron/main/db/features/project-flow/schema.ts

// 項目 + 節點 cache(每次進畫布頁先讀本地 → 同時打 server 更新)
export const projectCache = sqliteTable('project_cache', {
    projectId: integer('projectId').primaryKey(),
    userId: text('userId').notNull(),                    // 多帳號切換時隔離
    dataJson: text('dataJson').notNull(),                // 整份 project + nodes + edges
    serverUpdatedAt: integer('serverUpdatedAt'),         // server 的 Project.UpdatedAt;判斷 cache 是否過期
    cachedAt: integer('cachedAt').notNull(),             // 本地落地時間 ms
})

// 備忘錄 cache(AI 備忘建議要本地讀)
export const memoCache = sqliteTable('memo_cache', {
    memoId: integer('memoId').primaryKey(),
    userId: text('userId').notNull(),
    dataJson: text('dataJson').notNull(),
    serverUpdatedAt: integer('serverUpdatedAt'),
    cachedAt: integer('cachedAt').notNull(),
})

// 匯報草稿暫存(僅本地;真正發送走 server POST)
// 用途:寫到一半關掉 desktop,下次打開恢復;不參與跨機同步
export const reportDrafts = sqliteTable('report_drafts', {
    draftId: integer('draftId').primaryKey({autoIncrement: true}),
    userId: text('userId').notNull(),
    dataJson: text('dataJson').notNull(),                // {type, title, items: [...]}
    updatedAt: integer('updatedAt').notNull(),
})
```

**Cache 命中規則**:

- 進項目畫布頁:先讀 projectCache → 立刻顯示 → 同時呼叫 `GET /projects/{id}` → 回應後比對 serverUpdatedAt,新就刷新 UI + 寫
  cache
- 離線:只有 cache 有就顯示(右上角灰色「離線模式」標)
- 寫操作:不嘗試 cache,直接打 server;沒網 → 跳錯 toast(不 queue 重試)

## Part 9: 驗證

1. **後端**: SQL Server 2008 建表 → Swagger 測試全部 API

6. **端到端**: 編輯者建項目 → 員工匯報進度 → 員工寫匯報(AI輔助) → 主管查看留反饋 → 員工回應 → AI 生成新建議
