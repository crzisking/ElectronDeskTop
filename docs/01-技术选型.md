# 技術選型文檔

## 1. 桌面框架選型

### 1.1 候選方案對比

| 維度 | Electron | Tauri | NW.js |
|------|----------|-------|-------|
| **語言** | Node.js + Chromium | Rust + WebView | Node.js + Chromium |
| **包體大小** | 較大（~80MB） | 極小（~5MB） | 較大（~80MB） |
| **性能** | 中等 | 優秀 | 中等 |
| **生態系統** | 極豐富 | 成長中 | 一般 |
| **Node.js 訪問** | 完整支持 | 通過 command 限制 | 完整支持 |
| **系統 API** | 完整 | 完整 | 完整 |
| **多窗口支持** | 原生支持 | 支持 | 支持 |
| **透明窗口** | 完整支持 | 支持（部分平台限制） | 支持 |
| **社區活躍度** | 極高 | 高 | 低 |
| **企業案例** | VSCode, Slack, Discord | 無大規模案例 | 少 |
| **自動更新** | electron-updater 成熟 | 需自行集成 | 複雜 |
| **調試工具** | Chromium DevTools | 有限 | Chromium DevTools |

### 1.2 選擇 Electron 的理由

1. **透明浮球窗口需求**：浮球需要 `transparent: true` + `alwaysOnTop` 跨應用置頂，Electron 對此支持最成熟
2. **Node.js 完整訪問**：需要操作系統鑰匙串（keytar）、本地文件讀寫，Electron 主進程可直接使用 Node.js
3. **生態豐富**：electron-store、electron-updater、keytar 等企業常用庫都是 Electron 生態的一等公民
4. **團隊熟悉度**：前端開發者幾乎無需學習新語言
5. **調試便利**：Chromium DevTools 調試 UI，主進程也可用 VS Code 調試

## 2. 前端框架選型

### 2.1 Vue 3 vs React vs Angular

| 維度 | Vue 3 | React | Angular |
|------|-------|-------|---------|
| **TypeScript 支持** | 一等公民 | 需配置 | 原生支持 |
| **學習曲線** | 低 | 中 | 高 |
| **組件寫法** | SFC（`.vue`） | JSX | 模板 |
| **狀態管理** | Pinia（官方） | Redux/Zustand | NgRx |
| **包體大小** | 小 | 中 | 大 |
| **Element UI 生態** | Element Plus | Ant Design | NG-Zorro |

**選擇 Vue 3 + TypeScript 理由**：
- Composition API 對 TypeScript 類型推斷支持最好
- Element Plus 是 Vue 生態最成熟的企業 UI 庫
- SFC 文件格式讓模板、邏輯、樣式內聚，維護成本低

---

## 3. UI 組件庫

### 3.1 Element Plus

- Vue 3 官方推薦組件庫，原 Element UI 的 Vue 3 版本
- 組件齊全：表格、表單、彈窗、佈局、圖標均可開箱即用
- 主題定制：CSS 變量支持亮/暗色主題切換
- 按需引入：配合 `unplugin-vue-components` + `unplugin-auto-import` 自動按需引入，減小包體

## 4. 構建工具

### 4.1 electron-vite vs 手動配置 Vite

| 維度 | electron-vite | 手動多入口 Vite |
|------|---------------|-----------------|
| **配置複雜度** | 單一配置文件 | 需要 3 套 Vite 配置 |
| **`__dirname` 支持** | 自動處理 | 需手動 polyfill |
| **HMR 支持** | 原生支持渲染進程熱更新 | 需額外配置 |
| **預加載腳本構建** | 自動監聽並重啟 | 手動管理 |
| **外部包處理** | 自動標記 electron 為 external | 需手動配置 |
| **TypeScript 路徑別名** | 三個進程統一配置 | 各自配置 |

**選擇 electron-vite**：配置最少，進程間構建協調問題全部由框架處理。

### 5.2 關鍵依賴說明

| 包名 | 用途 | 進程 |
|------|------|------|
| `electron` | 桌面框架核心 | Main |
| `electron-vite` | 構建工具 | 構建時 |
| `electron-builder` | 打包分發 | 構建時 |
| `electron-store` | 本地配置持久化（非敏感） | Main |
| `keytar` | OS 系統鑰匙串，存儲 Auth Token | Main |
| `vue` | UI 框架 | Renderer |
| `element-plus` | UI 組件庫 | Renderer |
| `pinia` | 狀態管理 | Renderer |
| `vue-router` | 前端路由 | Renderer |
| `axios` | HTTP 請求 | Renderer |
| `unplugin-auto-import` | Element Plus 按需自動導入 | 構建時 |
| `unplugin-vue-components` | Vue 組件自動導入 | 構建時 |

> **注意**：`keytar` 是原生 Node.js 模塊，打包時需在 `electron-builder` 的 `extraResources` 或 `asar.unpack` 中特別處理。

## 6. electron-vite 核心配置

```typescript
// electron.vite.config.ts
import { resolve } from 'path'
import { defineConfig, externalizeDepsPlugin } from 'electron-vite'
import vue from '@vitejs/plugin-vue'
import AutoImport from 'unplugin-auto-import/vite'
import Components from 'unplugin-vue-components/vite'
import { ElementPlusResolver } from 'unplugin-vue-components/resolvers'

export default defineConfig({
  main: {
    plugins: [externalizeDepsPlugin()],
    resolve: {
      alias: { '@main': resolve('electron/main') }
    }
  },
  preload: {
    plugins: [externalizeDepsPlugin()]
  },
  renderer: {
    resolve: {
      alias: { '@': resolve('src') }
    },
    plugins: [
      vue(),
      AutoImport({
        resolvers: [ElementPlusResolver()],
        imports: ['vue', 'vue-router', 'pinia'],
        dts: 'src/types/auto-imports.d.ts'
      }),
      Components({
        resolvers: [ElementPlusResolver()],
        dts: 'src/types/components.d.ts'
      })
    ]
  }
})
```
