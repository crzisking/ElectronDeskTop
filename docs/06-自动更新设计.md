# 自動更新設計文檔（electron-updater 方案）

> 本方案使用 `electron-updater` + 內網 IIS/Nginx，支援差量更新與自動版本比對。
> 流程：**開發機打包 → 產出 `Installer + latest.yml + blockmap` → 上傳到內網 HTTP 服務 → 客戶端自動檢查/下載/安裝**。

---

## 1. 整體流程

```
┌──────────────────────────────────────────────────────────────────┐
│  開發機                                                           │
│    1. 改 package.json → "version": "1.0.1"                       │
│    2. npm run release:win                                        │
│       （= electron-builder --win --publish never）               │
│    3. 產物：dist/                                                 │
│         ├── ichiaDesktop-Setup-1.0.1.exe                         │
│         ├── ichiaDesktop-Setup-1.0.1.exe.blockmap                │
│         └── latest.yml                                           │
└────────────────────────────────┬─────────────────────────────────┘
                                 │ 上傳（手動或 CI）
                                 ▼
┌──────────────────────────────────────────────────────────────────┐
│  發布服務器（內網 IIS / Nginx）                                    │
│    http://192.168.120.135:505/                                   │
│        ├── latest.yml                                            │
│        ├── ichiaDesktop-Setup-1.0.1.exe                          │
│        └── ichiaDesktop-Setup-1.0.1.exe.blockmap                 │
└────────────────────────────────┬─────────────────────────────────┘
                                 │ HTTP GET（定時輪詢）
                                 ▼
┌──────────────────────────────────────────────────────────────────┐
│  客戶端                                                           │
│    electron-updater                                              │
│      • checkForUpdatesAndNotify()                                │
│      • 比對 latest.yml.version vs app.getVersion()                │
│      • 下載差量或完整安裝包（blockmap 加速）                       │
│      • 校驗 sha512                                               │
│      • 提示用戶重啟，執行 quitAndInstall()                        │
└──────────────────────────────────────────────────────────────────┘
```

**複雜度**：中等。核心程式碼不多（一個 manager 檔 + 幾個 IPC），重點在配置和發布服務器的小坑。

## 3. 構建配置（`package.json`）

### 3.1 新增 `build.publish`

在現有 `build` 區塊結尾加 `publish`：

```jsonc
{
  "build": {
    "appId": "com.ichia.desktop.client",
    "productName": "ichiaDesktop",
    // ... 既有配置不變 ...

    "publish": [
      {
        "provider": "generic",
        "url": "http://192.168.120.135:505/",
        "channel": "latest"
      }
    ]
  }
}
```

- `provider: "generic"`：任意 HTTP 靜態服務都支持，不綁雲廠商
- `url` 結尾的 `/` **不能省**，否則路徑拼接會錯
- `channel: "latest"` → 會生成 `latest.yml`；改為 `beta` 則生成 `beta.yml`，用於灰度

### 3.2 新增 `release` 腳本

在 `scripts` 區塊加：

```jsonc
{
  "scripts": {
    "release:win": "npm run build && electron-builder --win --publish never",
    "release:mac": "npm run build && electron-builder --mac --publish never",
    "release:linux": "npm run build && electron-builder --linux --publish never"
  }
}
```

> **為何用 `--publish never` 而不是 `always`**：
> `always` 會讓 `electron-builder` 嘗試自動把產物上傳到 publish.url 對應的位置。但 generic provider 沒有統一上傳 API（它只是個靜態目錄），所以 `always` 實際上不會工作 — 寫 `never` 明確表示「只生成文件，上傳我自己處理」。
> 產物放在 `dist/` 目錄下，手動或用腳本複製到服務器即可。

### 3.3 打包後產物清單（`dist/`）

```
dist/
├── ichiaDesktop-Setup-1.0.1.exe             ← NSIS 安裝包
├── ichiaDesktop-Setup-1.0.1.exe.blockmap    ← 差量更新用
├── latest.yml                               ← 版本元信息
├── win-unpacked/                            ← 解包的內容（不用上傳）
└── builder-effective-config.yaml            ← 調試用（不用上傳）
```

需要上傳的只有 3 個：`.exe` / `.exe.blockmap` / `latest.yml`。

### 3.4 `latest.yml` 範例（自動生成，不用手改）

```yaml
version: 1.0.1
files:
  - url: ichiaDesktop-Setup-1.0.1.exe
    sha512: hGmN3...（Base64 編碼的 SHA-512）
    size: 78123456
path: ichiaDesktop-Setup-1.0.1.exe
sha512: hGmN3...
releaseDate: '2026-04-24T10:00:00.000Z'
```

## 4. 發布服務器配置

### 4.1 目錄結構

```
http://192.168.120.135:505/
    ├── latest.yml                               ← 必須
    ├── ichiaDesktop-Setup-1.0.1.exe             ← 必須
    └── ichiaDesktop-Setup-1.0.1.exe.blockmap    ← 建議（差量更新）
```

歷史版本可保留（例如 `1.0.0.exe`），但 `latest.yml` 永遠指向最新版。

### 4.2 IIS 配置（Windows 服務器）

**關鍵坑**：IIS 默認不認識 `.yml` 和 `.blockmap` 的 MIME type，會回 404。

在站點根目錄加 `web.config`：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>
    <staticContent>
      <!-- 移除可能已存在的，避免重複 -->
      <remove fileExtension=".yml" />
      <remove fileExtension=".blockmap" />
      <!-- 加入 MIME 對應 -->
      <mimeMap fileExtension=".yml" mimeType="text/yaml" />
      <mimeMap fileExtension=".blockmap" mimeType="application/octet-stream" />
    </staticContent>
    <!-- 禁用緩存 latest.yml，否則客戶端可能看不到新版 -->
    <caching>
      <profiles>
        <add extension=".yml" policy="DontCache" />
      </profiles>
    </caching>
  </system.webServer>
</configuration>
```

### 4.3 Nginx 配置（如果用 Nginx）

```nginx
server {
  listen 505;
  server_name 192.168.120.135;
  root /var/www/ichia-update;

  location / {
    autoindex off;
  }

  # 禁止快取 latest.yml（核心）
  location = /latest.yml {
    add_header Cache-Control "no-cache, no-store, must-revalidate";
    add_header Pragma "no-cache";
    expires 0;
  }

  # 大檔下載
  location ~ \.(exe|blockmap)$ {
    add_header Cache-Control "public, max-age=31536000";  # 檔名帶版本號，可長期緩存
  }

  types {
    text/yaml          yml;
    application/octet-stream  blockmap exe;
  }
}
```

### 4.4 驗證服務器正常

服務器部署後用 curl 測試：

```bash
curl -I http://192.168.120.135:505/latest.yml
# 應返回 200 + Content-Type: text/yaml

curl http://192.168.120.135:505/latest.yml
# 應返回 yml 內容

curl -I http://192.168.120.135:505/ichiaDesktop-Setup-1.0.1.exe
# 應返回 200 + Content-Length
```

## 5. 客戶端配置

在 `config/app-config.json` 頂層新增 `update` 節點（保留，方便不重新打包就切換環境）：

```jsonc
{
  "version": "1.0.0",
  // ... 既有節點 ...

  "update": {
    "enabled": true,
    "feedUrl": "http://192.168.120.135:505/",
    "channel": "latest",
    "checkOnStartup": true,
    "checkIntervalMinutes": 120,
    "autoDownload": true,
    "autoInstallOnAppQuit": false
  }
}
```

| 字段 | 默認 | 說明 |
|------|------|------|
| `enabled` | `true` | 總開關 |
| `feedUrl` | `http://192.168.120.135:505/` | 服務器地址（結尾必須帶 `/`） |
| `channel` | `latest` | 對應 `latest.yml`；灰度時改 `beta` |
| `checkOnStartup` | `true` | 啟動 3 秒後檢查 |
| `checkIntervalMinutes` | `120` | 每 2 小時再檢查一次；`0` = 只啟動時檢查 |
| `autoDownload` | `true` | 發現新版自動下載（否則等用戶點「下載」） |
| `autoInstallOnAppQuit` | `false` | 下載完等用戶主動重啟；`true` = 下次退出時靜默安裝 |

型別（加到 `src/types/config.types.ts`）：

```typescript
export interface UpdateConfig {
  enabled: boolean
  feedUrl: string
  channel: 'latest' | 'beta' | 'alpha'
  checkOnStartup: boolean
  checkIntervalMinutes: number
  autoDownload: boolean
  autoInstallOnAppQuit: boolean
}

// AppConfig 追加 update 字段
export interface AppConfig {
  // ... 既有 ...
  update: UpdateConfig
}
```

### 6.2 `electron/main/update-manager.ts`

```typescript
/**
 * UpdateManager — 自動更新管理器（electron-updater 封裝）
 *
 * 職責：
 *   1. 依 configManager.getUpdateConfig() 配置 autoUpdater
 *   2. 綁定 autoUpdater 事件，廣播到渲染進程
 *   3. 暴露 check/download/quitAndInstall 給 IPC 使用
 *
 * 架構參考既有的 config-manager / tray-manager / floating-ball。
 */
import { autoUpdater, type UpdateInfo, type ProgressInfo } from 'electron-updater'
import { app, BrowserWindow } from 'electron'
import log from 'electron-log'
import { configManager } from './config-manager'
import { IpcChannels } from '../shared/ipc-channels'

export class UpdateManager {
  private mainWindow: BrowserWindow | null = null
  private timer: NodeJS.Timeout | null = null

  init(mainWindow: BrowserWindow) {
    this.mainWindow = mainWindow
    const cfg = configManager.getUpdateConfig()
    if (!cfg.enabled) return

    // ── 日誌：寫到 %APPDATA%/ichiaDesktop/logs/main.log ──
    log.transports.file.level = 'info'
    autoUpdater.logger = log

    // ── 基礎配置 ──
    autoUpdater.autoDownload = cfg.autoDownload
    autoUpdater.autoInstallOnAppQuit = cfg.autoInstallOnAppQuit
    autoUpdater.channel = cfg.channel
    autoUpdater.setFeedURL({
      provider: 'generic',
      url: cfg.feedUrl,
      channel: cfg.channel
    })

    // dev 模式下默認不檢查；如需本地驗證打開下面這行
    // autoUpdater.forceDevUpdateConfig = !app.isPackaged

    // ── 綁定事件 → 廣播到渲染進程 ──
    autoUpdater.on('checking-for-update', () => {
      this.send(IpcChannels.PUSH_UPDATE_CHECKING)
    })
    autoUpdater.on('update-available', (info: UpdateInfo) => {
      this.send(IpcChannels.PUSH_UPDATE_AVAILABLE, info)
    })
    autoUpdater.on('update-not-available', () => {
      this.send(IpcChannels.PUSH_UPDATE_NOT_AVAILABLE)
    })
    autoUpdater.on('download-progress', (p: ProgressInfo) => {
      this.send(IpcChannels.PUSH_UPDATE_PROGRESS, p)
    })
    autoUpdater.on('update-downloaded', (info: UpdateInfo) => {
      this.send(IpcChannels.PUSH_UPDATE_DOWNLOADED, info)
    })
    autoUpdater.on('error', (err) => {
      this.send(IpcChannels.PUSH_UPDATE_ERROR, {
        message: err?.message ?? String(err)
      })
    })

    // ── 啟動檢查 + 定期輪詢 ──
    if (cfg.checkOnStartup) {
      setTimeout(() => this.check(), 3_000)
    }
    if (cfg.checkIntervalMinutes > 0) {
      this.timer = setInterval(
        () => this.check(),
        cfg.checkIntervalMinutes * 60_000
      )
    }
  }

  /** 檢查是否有更新。autoDownload=true 時會自動開始下載 */
  async check() {
    try {
      return await autoUpdater.checkForUpdates()
    } catch (err) {
      log.error('[UpdateManager] check failed', err)
    }
  }

  /** autoDownload=false 時手動觸發下載 */
  async download() {
    return autoUpdater.downloadUpdate()
  }

  /** 下載完成後呼叫：立即重啟並安裝 */
  quitAndInstall() {
    autoUpdater.quitAndInstall()
  }

  private send(channel: string, payload?: unknown) {
    if (this.mainWindow && !this.mainWindow.isDestroyed()) {
      this.mainWindow.webContents.send(channel, payload)
    }
  }

  dispose() {
    if (this.timer) clearInterval(this.timer)
    autoUpdater.removeAllListeners()
  }
}

export const updateManager = new UpdateManager()
```

### 6.3 `electron/main/ipc-handlers/update.handlers.ts`

```typescript
import { ipcMain } from 'electron'
import { updateManager } from '../update-manager'
import { IpcChannels } from '../../shared/ipc-channels'

export function registerUpdateHandlers() {
  ipcMain.handle(IpcChannels.UPDATE_CHECK, () => updateManager.check())
  ipcMain.handle(IpcChannels.UPDATE_DOWNLOAD, () => updateManager.download())
  ipcMain.handle(IpcChannels.UPDATE_QUIT_AND_INSTALL, () => updateManager.quitAndInstall())
}
```

- `electron/main/ipc-handlers/index.ts` 統一入口加 `registerUpdateHandlers()`

## 8. 渲染層（UI）

### 8.1 Preload 擴展（`electron/preload/index.ts`）

```typescript
update: {
  check: () => ipcRenderer.invoke(IpcChannels.UPDATE_CHECK),
  download: () => ipcRenderer.invoke(IpcChannels.UPDATE_DOWNLOAD),
  quitAndInstall: () => ipcRenderer.invoke(IpcChannels.UPDATE_QUIT_AND_INSTALL),

  onChecking: (cb: () => void) =>
    ipcRenderer.on(IpcChannels.PUSH_UPDATE_CHECKING, cb),
  onAvailable: (cb: (_: unknown, info: { version: string; releaseNotes?: string }) => void) =>
    ipcRenderer.on(IpcChannels.PUSH_UPDATE_AVAILABLE, cb),
  onNotAvailable: (cb: () => void) =>
    ipcRenderer.on(IpcChannels.PUSH_UPDATE_NOT_AVAILABLE, cb),
  onProgress: (cb: (_: unknown, p: { percent: number; bytesPerSecond: number }) => void) =>
    ipcRenderer.on(IpcChannels.PUSH_UPDATE_PROGRESS, cb),
  onDownloaded: (cb: (_: unknown, info: { version: string }) => void) =>
    ipcRenderer.on(IpcChannels.PUSH_UPDATE_DOWNLOADED, cb),
  onError: (cb: (_: unknown, err: { message: string }) => void) =>
    ipcRenderer.on(IpcChannels.PUSH_UPDATE_ERROR, cb)
}
```

### 8.2 最小 UI 範例

在 `src/App.vue` 或專門的 `composables/useUpdate.ts` 中：

```typescript
import { ElNotification, ElMessageBox } from 'element-plus'

// 發現新版本 → 提示
window.electronAPI.update.onAvailable((_e, info) => {
  ElNotification({
    title: `發現新版本 ${info.version}`,
    message: '正在後台下載...',
    type: 'info',
    duration: 5000
  })
})

// 下載進度（可做成右下角進度條組件）
window.electronAPI.update.onProgress((_e, p) => {
  console.log(`下載中 ${p.percent.toFixed(1)}%`)
})

// 下載完成 → 彈窗問是否重啟
window.electronAPI.update.onDownloaded(async (_e, info) => {
  const confirmed = await ElMessageBox.confirm(
    `新版本 ${info.version} 已下載完成，是否立即重啟安裝？`,
    '更新就緒',
    { confirmButtonText: '立即重啟', cancelButtonText: '稍後', type: 'success' }
  ).catch(() => false)

  if (confirmed) {
    window.electronAPI.update.quitAndInstall()
  }
  // 用戶選「稍後」→ autoInstallOnAppQuit=true 時下次退出自動安裝
})

window.electronAPI.update.onError((_e, err) => {
  console.error('[Update]', err.message)
  // 更新錯誤不打擾用戶，只記日誌
})
```

## 9. 發布流程（你的日常操作）

```bash
# 1. 改版本號
# 手動編輯 package.json: "version": "1.0.0" → "1.0.1"

# 2. 本地打包
npm run release:win

# 3. 把產物複製到服務器
# 需要上傳的 3 個檔案：
#   dist/ichiaDesktop-Setup-1.0.1.exe
#   dist/ichiaDesktop-Setup-1.0.1.exe.blockmap
#   dist/latest.yml

# 可以寫個一鍵腳本，例如 scripts/release.ps1：
```

PowerShell 發布腳本範例：

```powershell
# scripts/release.ps1
$version = (Get-Content package.json | ConvertFrom-Json).version
$serverPath = "\\192.168.120.135\ichia-update"   # 映射的 UNC 路徑

# 或用 SCP / curl 上傳
Write-Host "Publishing version $version..."
Copy-Item "dist/ichiaDesktop-Setup-$version.exe" $serverPath -Force
Copy-Item "dist/ichiaDesktop-Setup-$version.exe.blockmap" $serverPath -Force
Copy-Item "dist/latest.yml" $serverPath -Force
Write-Host "Done. Clients will auto-update within $checkIntervalMinutes minutes."
```

發布節奏：
- 緊急 bug 修復：客戶端最多等 `checkIntervalMinutes`（2 小時）拿到更新
- 重大變更前：可通知用戶手動點「檢查更新」按鈕立即拉取

## 11. 常見問題與排查

| 問題 | 原因 / 解法 |
|------|------------|
| 客戶端日誌：`HttpError: 404 Not Found latest.yml` | 服務器 MIME 未配置，IIS 返回 404；檢查 §4.2 的 `web.config` |
| `Error: net::ERR_CONTENT_LENGTH_MISMATCH` | Nginx `sendfile` 與動態壓縮衝突；在 `.exe` 的 location 關掉 `gzip off;` |
| 下載完成但沒觸發 `update-downloaded` | sha512 校驗失敗 → 服務器上的 exe 與 yml 不匹配（很可能是上傳一半就覆蓋了） |
| 客戶端一直檢查不到新版 | yml 被 IIS/Nginx 或代理緩存；確認 §4.2/§4.3 的 no-cache 設定 |
| dev 模式下想本地測試 | 在 `init()` 中加 `autoUpdater.forceDevUpdateConfig = !app.isPackaged`；本機起 `http-server dist -p 505` 模擬服務器 |
| Windows SmartScreen 警告 | 安裝包未代碼簽名；內網環境可忽略；對外發布建議購買 EV 憑證 |
| macOS 自動更新失敗 | macOS 必須代碼簽名 + Apple 公證，否則 `electron-updater` 直接拒絕 |

### 11.1 日誌位置

更新過程的詳細日誌（包括每次 HTTP 請求、下載進度、錯誤堆疊）：

- Windows：`%APPDATA%\ichiaDesktop\logs\main.log`
- macOS：`~/Library/Logs/ichiaDesktop/main.log`
- Linux：`~/.config/ichiaDesktop/logs/main.log`

客戶端反饋更新問題時，第一件事就是撈這個檔。

## 12. 代碼簽名

> 當前用戶全為 Windows，所以本節主要講 Windows 簽名；macOS / Linux 作為參考附在後面。

### 12.1 為什麼要簽名？

| 不簽 | 簽了 |
|------|------|
| Windows SmartScreen 彈紅色警告「不安全的應用」 | 警告消失或大幅降低 |
| 部分防毒軟體誤殺 | 可信任來源，誤殺率降低 |
| 企業內網組策略可能直接禁止執行 | 通過白名單 |
| `electron-updater` 能跑 | 一樣能跑（Windows 下不強制） |

**結論**：
- 內網部署 + 自己的員工機 → 簽不簽都行，簽了體驗好一點
- 公網發布 / 陌生用戶 → **強烈建議簽**

### 12.2 Windows 簽名流程（主要）

#### 步驟 1：取得代碼簽名憑證

| 類型 | 價格（約） | SmartScreen 行為 | 建議 |
|------|-----------|-----------------|------|
| **OV（普通）憑證** | USD $100~200/年 | 首次下載仍會警告，需累積「聲譽」（數百次下載後消失） | 內網可用 |
| **EV（擴展）憑證** | USD $300~500/年 | 即時通過，無警告 | 對外發布推薦 |
| **Azure Trusted Signing**（雲端） | 按次計費（約 USD $10/月起） | 同 OV | 不想管硬體 token 首選 |
| **SSL.com eSigner**、**DigiCert KeyLocker** | 月費制 | 同 OV/EV（視方案） | 替代品 |

常見販售商：SSL.com、DigiCert、Sectigo、Comodo、GlobalSign。OV 憑證審核 1-3 天，EV 要寄硬體 USB token（2-7 天）。

#### 步驟 2：準備憑證檔

憑證通常以 `.pfx`（含私鑰）形式拿到，也有硬體 USB token 形式。

```
build/
├── cert.pfx              ← 憑證檔（千萬別 commit 進 git！）
└── ...
.gitignore                ← 務必加上 cert.pfx 和 *.pfx
```

#### 步驟 3：修改 `package.json` 的 `build.win`

**方式 A：設定檔路徑（本地打包用）**

```jsonc
{
  "build": {
    "win": {
      "target": [...既有設定不變...],
      "icon": "resources/icons/icon.ico",
      "certificateFile": "./build/cert.pfx",
      "certificatePassword": "YOUR_PFX_PASSWORD",
      "signingHashAlgorithms": ["sha256"],
      "publisherName": "Ichia Co., Ltd."
    }
  }
}
```

> **警告**：不要把 `certificatePassword` 明文寫進 git！本地打包可以暫時寫著，提交前刪除或改用方式 B。

**方式 B：環境變數（CI / 安全）推薦**

```jsonc
{
  "build": {
    "win": {
      "target": [...],
      "icon": "resources/icons/icon.ico",
      "signingHashAlgorithms": ["sha256"],
      "publisherName": "Ichia Co., Ltd."
      // certificateFile / certificatePassword 由環境變數提供
    }
  }
}
```

執行打包前設環境變數：

```powershell
# PowerShell
$env:CSC_LINK = "C:\path\to\cert.pfx"       # 或 base64 字串 / https URL
$env:CSC_KEY_PASSWORD = "YOUR_PFX_PASSWORD"
npm run release:win
```

```bash
# CMD
set CSC_LINK=C:\path\to\cert.pfx
set CSC_KEY_PASSWORD=YOUR_PFX_PASSWORD
npm run release:win
```

`electron-builder` 會自動偵測 `CSC_LINK` / `CSC_KEY_PASSWORD`，打包時：
1. 簽 `.exe`（主程序、安裝包、uninstaller）
2. 簽 `.blockmap`（差量更新檔）
3. 把 sha512 寫進 `latest.yml`（與簽名獨立，自動處理）

#### 步驟 4：硬體 USB Token 簽名（EV 憑證）

EV 憑證通常是硬體 token，私鑰無法匯出，只能透過 `signtool.exe` 呼叫：

```jsonc
{
  "build": {
    "win": {
      "signtoolOptions": {
        "sign": "./build/sign-with-token.js"
      }
    }
  }
}
```

然後寫個自定義簽名腳本 `build/sign-with-token.js`：

```javascript
// build/sign-with-token.js
// electron-builder 會以 (configuration) => Promise<void> 呼叫此函數
exports.default = async function (configuration) {
  const { execSync } = require('child_process')
  const filePath = configuration.path

  // 需先裝好 USB token 的驅動和管理工具（如 SafeNet Authentication Client）
  // 密碼可透過環境變數或運行時彈窗輸入
  execSync(
    `signtool sign /tr http://timestamp.digicert.com /td sha256 /fd sha256 ` +
    `/a /csp "eToken Base Cryptographic Provider" ` +
    `/k "[{{${process.env.TOKEN_PIN}}}]=YourCertContainerName" "${filePath}"`,
    { stdio: 'inherit' }
  )
}
```

> 硬體 token 不適合 CI，一般在本地機或固定的簽名伺服器執行。

#### 步驟 5：Azure Trusted Signing（雲端，推薦新項目）

沒有硬體 token、想免維護 → 考慮 Microsoft 官方的 [Azure Trusted Signing](https://learn.microsoft.com/en-us/azure/trusted-signing/)：

```jsonc
{
  "build": {
    "win": {
      "azureSignOptions": {
        "endpoint": "https://eus.codesigning.azure.net/",
        "certificateProfileName": "ichia-profile",
        "codeSigningAccountName": "ichia-signing"
      }
    }
  }
}
```

並設定環境變數：`AZURE_TENANT_ID` / `AZURE_CLIENT_ID` / `AZURE_CLIENT_SECRET`。

#### 步驟 6：驗證簽名

打包完成後：

```powershell
# 方法 A：右鍵 exe → 屬性 → 數位簽章 頁籤
# 應看到簽署者名稱、時間戳、摘要演算法 sha256

# 方法 B：用 signtool 查詢
signtool verify /pa /v "dist\ichiaDesktop-Setup-1.0.1.exe"
# 成功輸出會有 "Successfully verified"
```

檢查清單：
- [ ] `.exe` 屬性對話框的「數位簽章」頁籤存在且顯示簽署者
- [ ] `signtool verify /pa` 返回成功
- [ ] `.exe.blockmap` 也已簽名（electron-builder 會自動簽）
- [ ] 在乾淨的 Windows 上安裝時 SmartScreen 警告消失（OV 首次下載可能仍有）

### 12.3 macOS 簽名流程（參考）

> 目前無 macOS 用戶，以下僅記錄備用。若將來增加 macOS，**必須**簽名 + 公證，否則 `electron-updater` 會直接拒絕更新。

#### 準備

1. 加入 [Apple Developer Program](https://developer.apple.com/programs/)（USD $99/年）
2. 在 Apple Developer 後台創建 **Developer ID Application** 憑證（用於 app 本體）和 **Developer ID Installer** 憑證（用於 .pkg，若需要）
3. 從 Keychain 匯出成 `.p12` 檔
4. 創建 App-specific Password（[appleid.apple.com](https://appleid.apple.com) → 登入 → App-Specific Passwords）供 notarization 使用

#### 配置

```jsonc
{
  "build": {
    "mac": {
      "target": ["dmg"],
      "icon": "resources/icons/icon.icns",
      "category": "public.app-category.business",
      "hardenedRuntime": true,
      "gatekeeperAssess": false,
      "entitlements": "build/entitlements.mac.plist",
      "entitlementsInherit": "build/entitlements.mac.plist",
      "notarize": {
        "teamId": "YOUR_TEAM_ID"
      }
    }
  }
}
```

`build/entitlements.mac.plist`（Hardened Runtime 必需）：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>com.apple.security.cs.allow-jit</key><true/>
  <key>com.apple.security.cs.allow-unsigned-executable-memory</key><true/>
  <key>com.apple.security.cs.disable-library-validation</key><true/>
</dict>
</plist>
```

#### 環境變數

```bash
export CSC_LINK="/path/to/developer-id.p12"
export CSC_KEY_PASSWORD="P12_PASSWORD"
export APPLE_ID="your@apple.id"
export APPLE_APP_SPECIFIC_PASSWORD="xxxx-xxxx-xxxx-xxxx"
export APPLE_TEAM_ID="YOUR_TEAM_ID"
npm run release:mac
```

`electron-builder` 會自動：
1. 用 `codesign` 簽 `.app` bundle
2. 打包成 `.dmg`
3. 上傳到 Apple 公證服務，等待結果（5-30 分鐘）
4. `stapler staple` 把公證票據釘到 .dmg

### 12.4 Linux 簽名流程（參考）

> Linux 不強制簽名，`electron-updater` 對 AppImage 不執行簽名驗證。
> 若有需求可用 GPG 簽名，但通常跳過。

```jsonc
{
  "build": {
    "linux": {
      "target": ["AppImage"],
      "icon": "resources/icons/icon.png"
      // 一般不需要簽名配置
    }
  }
}
```

若要 GPG 簽名 AppImage：

```bash
# 打包後對 AppImage 簽名
gpg --detach-sign --armor "dist/ichiaDesktop-1.0.1.AppImage"
# 會產出 ichiaDesktop-1.0.1.AppImage.asc
```

分發時把 `.asc` 一併放上服務器，用戶自行用 `gpg --verify` 檢查。

### 12.5 常見簽名問題

| 問題 | 解法 |
|------|------|
| `signtool.exe` 找不到 | 裝 [Windows SDK](https://developer.microsoft.com/en-us/windows/downloads/windows-sdk/)，signtool 在 `C:\Program Files (x86)\Windows Kits\10\bin\<version>\x64\` |
| 簽名後 SmartScreen 仍警告 | OV 憑證需累積聲譽（幾百到幾千次下載後自動消失）；要即時通過請用 EV 或 Azure Trusted Signing |
| 時間戳伺服器連不上 | 換另一個時間戳 URL，常用：`http://timestamp.digicert.com`、`http://timestamp.sectigo.com`、`http://tsa.starfieldtech.com` |
| macOS 公證超時 / 失敗 | 檢查 `entitlements.mac.plist` 是否完整；用 `xcrun notarytool log <submission-id>` 看錯誤原因 |
| 憑證密碼洩漏到 git | 立即作廢憑證 → 向 CA 申請重新簽發 → 清理 git 歷史（`git filter-repo`） |

## 13. 安全提示

1. **內網 HTTP 可用，但公網必須 HTTPS**。目前 `http://192.168.120.135:505/` 是內網地址，中間人風險可控；若將來暴露到公網，務必改 HTTPS。
2. **sha512 完整性校驗是框架自動行為**，無需配置。`electron-builder` 打包時寫入 yml，`electron-updater` 下載後自動比對，篡改會被拒絕。這保證了「就算沒代碼簽名，至少下到的 exe 是服務器上那個」。
3. **代碼簽名 ≠ sha512 校驗**：
   - sha512 防「傳輸中被篡改」（完整性）
   - 代碼簽名防「安裝包被惡意仿冒」+ 過 SmartScreen（身份驗證）
   - 兩者獨立，各解決不同問題
4. **服務器訪問控制**：若擔心惡意員工替換安裝包，在 IIS/Nginx 設只允許運維 IP 上傳（或用 `scp` + 部署密鑰）。
5. **憑證保護**：`.pfx` 和 `.p12` 絕不進 git，密碼只透過環境變數或密鑰管理服務注入。

## 14. 與輕量輪詢方案的對比

| 項目 | 本方案（electron-updater） | 純輪詢 version.json |
|------|--------------------------|---------------------|
| 依賴 | `electron-updater` + `electron-log` | 無 |
| 服務器文件 | `latest.yml` + `.exe` + `.blockmap` | `version.json` + `.exe` |
| 差量更新 | ✅ blockmap 省頻寬 | ❌ 每次下完整包 |
| sha512 完整性校驗 | ✅ 自動 | 需自己實現 |
| macOS 支持 | ✅ 成熟 | DMG 需自己處理掛載/覆蓋 |
| 代碼量 | 中（~150 行 + 配置） | 中（~200 行，全手寫下載/執行） |
| 社區穩定性 | 成熟（百萬應用使用） | 自行維護 |

**結論**：內網部署也推薦 electron-updater，省維護成本且有差量更新加成。
