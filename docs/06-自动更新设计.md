# 自動更新設計文檔（electron-updater 方案）

> 本文檔對應 ichiaDesktop 專案內已實作的自動更新邏輯，與代碼一一對應。
> 流程：**開發機打包 → 產出 `Installer + latest.yml + blockmap` → 上傳到內網 HTTP 服務 → 客戶端每天 11:00 自動檢查/下載/安裝**

---

## 1. 整體流程

```
┌──────────────────────────────────────────────────────────────────┐
│  開發機                                                           │
│    1. 改 package.json → "version": "1.0.1"                       │
│    2. npm run release:win                                        │
│    3. 產物：dist/                                                 │
│         ├── ichiaDesktop-Setup-1.0.1.exe                         │
│         ├── ichiaDesktop-Setup-1.0.1.exe.blockmap                │
│         └── latest.yml                                           │
└────────────────────────────────┬─────────────────────────────────┘
                                 │ 手動上傳
                                 ▼
┌──────────────────────────────────────────────────────────────────┐
│  發布服務器（內網 IIS / Nginx）                                    │
│    http://192.168.120.135:10001/                                   │
│        ├── latest.yml                                            │
│        ├── ichiaDesktop-Setup-1.0.1.exe                          │
│        └── ichiaDesktop-Setup-1.0.1.exe.blockmap                 │
└────────────────────────────────┬─────────────────────────────────┘
                                 │ HTTP GET（每天 11:00）
                                 ▼
┌──────────────────────────────────────────────────────────────────┐
│  客戶端                                                           │
│    UpdateManager（electron/main/update-manager.ts）              │
│      • 啟動時計算「下一次到 11:00 的時間差」並 setTimeout         │
│      • 觸發 autoUpdater.checkForUpdates()                         │
│      • 比對 latest.yml.version vs app.getVersion()                │
│      • autoDownload=true → 下載差量包 + sha512 校驗               │

│      • 用戶確認 → quitAndInstall（NSIS oneClick 靜默安裝）        │
└──────────────────────────────────────────────────────────────────┘
```

## 3. 構建配置（`package.json`）

### 3.1 `build.publish`（已加入）

```jsonc
{
  "build": {
    "publish": [
      {
        "provider": "generic",
        "url": "http://192.168.120.135:10001/",
        "channel": "latest"
      }
    ]
  }
}
```

- `provider: "generic"`：任意 HTTP 靜態服務皆可，不綁雲廠商
- `url` 結尾的 `/` **不能省**
- `channel: "latest"` → 生成 `latest.yml`

### 3.2 NSIS 改為一鍵更新模式（已加入）

```jsonc
{
  "build": {
    "nsis": {
      "oneClick": true,             // 改為 true：自動更新時不彈安裝精靈
      "perMachine": false,           // false：安裝到當前用戶目錄，無需管理員權限
      "allowElevation": true,        // 用戶權限不足時允許提權
      "deleteAppDataOnUninstall": false,
      "createDesktopShortcut": true,
      "createStartMenuShortcut": true
    }
  }
}
```

**為什麼從 `oneClick: false` 改為 `true`**：
- `oneClick: false` 模式下 `quitAndInstall()` 會彈出完整安裝精靈（讓用戶選路徑、按 Next…），對自動更新體驗很差
- `oneClick: true` 配合 `perMachine: false`：完全靜默安裝，下次啟動就是新版本

> **取捨**：用戶失去「自選安裝路徑」的能力，但自動更新體驗順暢。內網企業客戶端推薦 oneClick。

### 3.3 release 腳本（已加入）

```jsonc
{
  "scripts": {
    "release:win": "npm run build && electron-builder --win --publish never",
    "release:mac": "npm run build && electron-builder --mac --publish never",
    "release:linux": "npm run build && electron-builder --linux --publish never"
  }
}
```

`--publish never` 明確表示「只生成文件，上傳我自己處理」（generic provider 沒有統一上傳 API）。

### 3.4 打包產物

```
dist/
├── ichiaDesktop-Setup-1.0.1.exe             ← NSIS 安裝包
├── ichiaDesktop-Setup-1.0.1.exe.blockmap    ← 差量更新用
├── latest.yml                               ← 版本元信息
├── win-unpacked/                            ← 解包內容（不用上傳）
└── builder-effective-config.yaml            ← 調試用（不用上傳）
```

### 5.2 UpdateManager 設計（依賴注入）

**關鍵點**：與專案其他 manager（FloatingBallManager、TrayManager）一致，採用「依賴注入 + 不導出 module 級單例」風格。

```typescript
// electron/main/update-manager.ts（核心結構）
export class UpdateManager {
  private mainWindow: BrowserWindow | null = null
  private firstCheckTimer: NodeJS.Timeout | null = null
  private dailyTimer: NodeJS.Timeout | null = null
  private listenersBound = false

  // 注入完整依賴：除 configManager 外，安裝完成後需要關閉浮球、托盤等才能觸發 quitAndInstall
  constructor(
    private readonly configManager: ConfigManager,
    private readonly windowManager: WindowManager,
    private readonly floatingBallMgr: FloatingBallManager,
    private readonly trayManager: TrayManager
  ) {}

  init(mainWindow: BrowserWindow): void {
    const cfg = this.configManager.getUpdateConfig()
    if (!cfg.enabled) return

    // 配置 autoUpdater
    autoUpdater.autoDownload = cfg.autoDownload
    autoUpdater.autoInstallOnAppQuit = cfg.autoInstallOnAppQuit
    autoUpdater.channel = cfg.channel
    autoUpdater.setFeedURL({
      provider: 'generic',
      url: cfg.feedUrl,
      channel: cfg.channel
    })

    // 把 autoUpdater 日誌接到專案 logger
    autoUpdater.logger = {
      info: (m) => logger.info(String(m), 'UpdateManager'),
      warn: (m) => logger.warn(String(m), 'UpdateManager'),
      error: (m) => logger.error(String(m), 'UpdateManager'),
      debug: (m) => logger.debug(String(m), 'UpdateManager')
    }

    // 綁定生命週期事件 → 全部廣播到主窗口
    this.bindAutoUpdaterListeners()

    // 排程每日定時檢查
    if (cfg.dailyCheckTime) {
      this.scheduleDailyCheck(cfg.dailyCheckTime)
    }

    // 退出前清理
    app.once('before-quit', () => this.dispose())
  }

  // check / download / quitAndInstall 對外 API ...
  // dispose 清 timer ...
}
```

### 5.3 每日定時檢查實作（重點）

**為什麼不用簡單的 `setInterval(check, 24h)`**：
應用啟動時間隨用戶習慣不同，會導致檢查時刻每天偏移，最終可能變成「凌晨檢查」。

**正確做法**：
1. 啟動時計算「現在 → 下一次 HH:MM 的毫秒差」
2. `setTimeout(check, delayMs)` 觸發第一次
3. 第一次觸發後 `setInterval(check, 24h)` 重複

```typescript
private scheduleDailyCheck(hhmm: string): void {
  const { hour, minute } = this.parseHHMM(hhmm)!
  const now = new Date()
  const next = new Date(now)
  next.setHours(hour, minute, 0, 0)
  if (next.getTime() <= now.getTime()) {
    next.setDate(next.getDate() + 1)  // 已過今日 → 排到明天
  }
  const delayMs = next.getTime() - now.getTime()

  this.firstCheckTimer = setTimeout(() => {
    this.check()
    this.dailyTimer = setInterval(() => this.check(), 24 * 60 * 60 * 1000)
  }, delayMs)
}
```

**邊界情況**：夏令時切換、用戶調整系統時鐘等會造成單次偏差，內網企業環境可忽略。

### 5.4 IPC 入口（`update.handlers.ts`）

### 5.6 主進程入口呼叫順序（`electron/main/index.ts`）

```typescript
// Step 1: 配置
configManager = new ConfigManager()
await configManager.load()

// Step 2: 窗口
windowManager = new WindowManager()
windowManager.createMainWindow()
windowManager.createFloatingBallWindow()

// Step 3: 浮球
floatingBallMgr = new FloatingBallManager(windowManager, ...)

// Step 4: 托盤實例（先建構，init() 延後到 IPC 註冊後才呼叫）
trayManager = new TrayManager(windowManager, configManager)

// Step 5: 自動更新管理器（依賴注入 4 個 manager，必須在 registerAllHandlers 前建構）
updateMgr = new UpdateManager(configManager, windowManager, floatingBallMgr, trayManager)

// Step 6: IPC handler 統一註冊
registerAllHandlers(windowManager, configManager, floatingBallMgr, updateMgr)

// Step 7: 托盤 init（IPC 註冊完成後才能正確處理菜單點擊）
trayManager.init()

// Step 8: 啟動模式判斷（startMinimized）

// Step 9: 啟用自動更新（在主窗口就緒後）
const mainWindow = windowManager.getMainWindow()
if (mainWindow) updateMgr.init(mainWindow)
```

> **為什麼 update 放最後**：UpdateManager 需要把事件推送到主窗口的渲染進程，必須等主窗口創建+IPC 註冊完成。

### 8.1 目錄結構

### 8.2 IIS 配置

**關鍵坑**：IIS 默認不認識 `.yml` 和 `.blockmap` MIME，會回 404。

在站點根目錄加 `web.config`：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>
    <staticContent>
      <remove fileExtension=".yml" />
      <remove fileExtension=".blockmap" />
      <mimeMap fileExtension=".yml" mimeType="text/yaml" />
      <mimeMap fileExtension=".blockmap" mimeType="application/octet-stream" />
    </staticContent>
    <caching>
      <profiles>
        <add extension=".yml" policy="DontCache" />
      </profiles>
    </caching>
  </system.webServer>
</configuration>
```

### 8.3 Nginx 配置（替代方案）

```nginx
server {
  listen 505;
  server_name 192.168.120.135;
  root /var/www/ichia-update;

  location = /latest.yml {
    add_header Cache-Control "no-cache, no-store, must-revalidate";
    expires 0;
  }

  location ~ \.(exe|blockmap)$ {
    add_header Cache-Control "public, max-age=31536000";
  }

  types {
    text/yaml          yml;
    application/octet-stream  blockmap;
  }
}
```

## 11. 常見問題

| 問題 | 解法 |
|------|------|
| 客戶端日誌 `404 Not Found latest.yml` | IIS MIME 未配置（§8.2） |
| `ERR_CONTENT_LENGTH_MISMATCH` | Nginx gzip 動態壓縮衝突，在 `.exe` location 加 `gzip off` |
| 下載完無 `update-downloaded` 事件 | sha512 校驗失敗 → 服務器 exe 與 yml 不匹配（檔案上傳一半被覆蓋） |
| 一直檢查不到新版 | yml 被 IIS/Nginx/代理緩存（§8.2/§8.3 的 no-cache 設定） |
| dev 模式下想本地測試 | `update-manager.ts` 中放開 `autoUpdater.forceDevUpdateConfig = !app.isPackaged`，本機起 `http-server dist -p 505` 模擬服務器 |
| Windows SmartScreen 警告 | 安裝包未代碼簽名；內網可忽略，外發建議 EV 憑證（§13） |
| 11:00 沒觸發檢查 | 看主進程日誌「首次定時檢查將於 ... 觸發」是否打印；若沒有則 `dailyCheckTime` 格式錯誤 |
| 下載完用戶選「稍後」後再也不提示 | `update-downloaded` 事件本次應用週期內已觸發過；下次重啟應用會重新流程 |

### 11.1 日誌位置

更新流程的詳細日誌走專案 logger，並由 `electron/main/utils/log-file-writer.ts` **自動落地到文件**：

| 模式            | 路徑                                                         |
|---------------|------------------------------------------------------------|
| 開發模式          | 專案根目錄 `logs/main-YYYY-MM-DD.log` + `electron-vite dev` 控制台 |
| 生產模式（Windows） | `%APPDATA%\ichiaDesktop\logs\main-YYYY-MM-DD.log`          |

主進程 `app.whenReady()` 一進來就呼叫 `initLogFileWriter()`，後續 `logger.info/warn/error` 會同時寫到控制台和對應文件。autoUpdater
的日誌也通過 `logger` 接出來（見 §5.2），所以更新事件會出現在同一份日誌中。

## 12. NSIS oneClick 模式說明

從 `oneClick: false` → `true` 的影響：

| | oneClick: false（原） | oneClick: true（現在） |
|---|---|---|
| 首次安裝 | 用戶看精靈、選路徑 | 一鍵裝到 `%LOCALAPPDATA%/Programs/ichiaDesktop` |
| 自動更新 | **彈完整安裝精靈** | 完全靜默 |
| 用戶可選裝路徑 | ✓ | ✗（固定到 LocalAppData） |
| 需要管理員權限 | 看路徑 | 不需要（perMachine: false） |
| 企業批量部署 | 適合 MSI/SCCM | 適合用戶自助 |

**改動的取捨**：犧牲首次安裝的可定制性，換取自動更新的順暢體驗。對企業內部桌面客戶端是合理選擇。

如果未來要兼容「IT 統一部署」場景，可以保留 `oneClick: true` 的同時提供 `--per-machine` 命令列選項給 IT，或另出一個 MSI 版本。

## 13. 代碼簽名

> 當前用戶全為 Windows，本節以 Windows 為主；macOS / Linux 簡述參考。

### 13.1 為什麼要簽名

| 不簽 | 簽了 |
|------|------|
| Windows SmartScreen 彈紅色警告 | 警告消失或大幅降低 |
| 部分防毒軟體誤殺 | 可信任來源，誤殺率降低 |
| `electron-updater` 能跑 | 一樣能跑（Windows 不強制） |

**結論**：
- 內網部署 + 自家員工 → 簽不簽都可以
- 公網發布 / 陌生用戶 → **強烈建議簽**

### 13.2 Windows 簽名流程

#### 取得憑證

| 類型 | 價格 | SmartScreen 行為 | 建議 |
|------|-----|-----------------|------|
| **OV** 憑證 | USD $100~200/年 | 首次仍警告，需累積聲譽 | 內網可用 |
| **EV** 憑證 | USD $300~500/年 | 即時通過 | 對外推薦 |
| **Azure Trusted Signing** | 月費約 $10 | 同 OV | 不想管 USB token 首選 |

販售商：SSL.com、DigiCert、Sectigo、Comodo、GlobalSign。

#### 配置（環境變數方式，推薦）

```jsonc
{
  "build": {
    "win": {
      "signingHashAlgorithms": ["sha256"],
      "publisherName": "Ichia Co., Ltd."
    }
  }
}
```

打包前設環境變數：

```powershell
# PowerShell
$env:CSC_LINK = "C:\path\to\cert.pfx"
$env:CSC_KEY_PASSWORD = "YOUR_PFX_PASSWORD"
npm run release:win
```

`electron-builder` 會自動偵測並簽 `.exe` / `.blockmap`。

#### 驗證

```powershell
signtool verify /pa /v "dist\ichiaDesktop-Setup-1.0.1.exe"
# 成功會看到 "Successfully verified"
```

> 詳細的 EV USB token / Azure Trusted Signing / macOS notarization 流程，請參考 electron-builder 官方文檔，本專案目前不需要。

### 13.3 安全提示

1. **內網 HTTP 可用，公網必須 HTTPS**。`http://192.168.120.135:10001/` 是內網地址，中間人風險可控；公網部署務必改 HTTPS
2. **sha512 完整性校驗是 electron-updater 框架自動行為**，無需配置。打包時寫入 yml，下載後自動比對，篡改會被拒絕。這保證了「就算沒簽名，至少下到的 exe 與服務器一致」
3. **代碼簽名 ≠ sha512 校驗**：
   - sha512 防「傳輸中被篡改」（完整性）
   - 代碼簽名防「假冒安裝包」+ 過 SmartScreen（身份驗證）
   - 兩者獨立，各解決不同問題
4. **服務器訪問控制**：在 IIS/Nginx 設只允許運維 IP 上傳，或用 SCP + 部署密鑰
5. **憑證保護**：`.pfx` 絕不進 git，密碼只透過環境變數注入

## 14. 與「純輪詢」方案對比

| 項目 | electron-updater（採用） | 純輪詢 version.json |
|------|--------------------------|---------------------|
| 依賴 | `electron-updater` | 無 |
| 服務器文件 | `latest.yml` + `.exe` + `.blockmap` | `version.json` + `.exe` |
| 差量更新 | ✅ blockmap 省頻寬 | ❌ 每次下完整包 |
| sha512 完整性校驗 | ✅ 自動 | 需自己實現 |
| macOS 支持 | ✅ 成熟 | DMG 需自己處理 |
| 代碼量 | 中（~250 行 + 配置） | 中（~200 行手寫下載/執行） |
| 社區穩定性 | 成熟 | 自行維護 |

**結論**：內網部署也採用 electron-updater，省維護成本且有差量更新加成。
