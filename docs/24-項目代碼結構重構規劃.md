# 項目代碼結構重構規劃

> **目的**:全項目代碼經過多輪迭代後出現臃腫、歸類混亂、業務 / 執行耦合。本文做一次徹底盤點,
> 出一份**有優先級的重構計劃**,**先看完規劃確認方向再開工**。
>
> **範圍**:整個 `DeskTop/` Electron 專案(main / preload / shared / renderer)。不含 server / web admin。
>
> **承接**:[23-工作採集重構規劃.md](./23-工作採集重構規劃.md)(模板移到 client + work-collect 模組整理)。
> 本文涵蓋更廣 —— 整個專案結構,work-collect 那塊不再贅述。

---

## 0. TL;DR — 三件事

1. **拆 mega 文件**:5 個檔 > 400 行,最大的 1500 行,讀起來像考古
2. **業務 / 執行分離**:很多 manager 同時做「驅動 OS API」+「跑業務規則」,改一邊動到另一邊
3. **歸類重組**:`src/composables/` 只有 2 個檔但 features/* 內到處是 composables;`src/api/modules/` 半空,各 feature 自己有
   api.ts;類型分散在 `types/` 和 `types/config/`

下面按「**有問題的具體位置**」一一給拆法。

## 2. 主要痛點

### 2.1 業務 / 執行耦合(最嚴重)

「執行代碼」= 跟 OS / 框架 API 直接打交道(timer、BrowserWindow、ipcMain)
「業務代碼」= 跑業務規則(工時判定、idle 判定、勾選哪個模板)

當前耦合的典型例子:

#### `electron/main/work-collector.ts`(407 行)

```
class CaptureService          // 執行:呼叫 desktopCapturer / 算 dHash
class IdleDetector            // 業務:idle 判定規則
class WorkSyncCoordinator     // 業務 + 執行:推 IPC + 管 pending 狀態
class WorkCollectorScheduler  // 編排:timer + capture + sync + idle + config gate
+ 北京時間工具函式 (getBeijingHour / getBeijingDate)
```

4 個 class + 工具函式擠一檔,改 idle 算法要在 IPC push 邏輯旁邊看半天。

#### `electron/main/window-manager.ts`(502 行)

```
class WindowManager
  - mainWindow / floatingBallWindow / logViewerWindow / agentWindow
  - 各窗口的 create / show / hide / position / close
```

四種窗口的開窗策略全在一個 class —— 浮球的「貼邊吸附」邏輯跟 agent 窗口的「resize 後保存位置」混在 18 個 method 裡。

#### `electron/main/db/features/config/repository.ts`(553 行)

```
assembleAppConfig()    // 讀 7 張表 → 組裝 AppConfig
applyPartial()         // 寫 7 張表 + KV upsert
resyncDevOwnedConfig() // 啟動時 reseed
USER_OWNED_KEYS Set    // dev/user-owned 區分
+ deepMerge / boolToInt 等 ad-hoc helpers
```

7 張表(KV + 6 collections)的 read / write / reseed 邏輯都在一個 repository。改一張 collection 表的 schema 要在 553
行內找對位置。

### 2.2 Mega 組件(讀不下去)

`AgentWindow.vue` **1549 行** —— 一個 Vue SFC 包:

- onMounted 初始化(讀 config / providers / 對話列表 / 自動彈設定)
- 訊息渲染(v-for + scroll-to-bottom)
- 對話側邊欄(列表 / 刪除 / 切換)
- 輸入框(send / abort / Enter 處理)
- 設定 dialog
- Prompt dialog
- 鍵盤快捷鍵
- 5 個 watch + 10+ functions + 100+ refs

`LogViewerView.vue` **536 行** —— 同樣堆疊 filter / table / 健康卡 / 模組下拉等多個關注點。

### 2.3 歸類混亂

#### `src/composables/` vs `features/*/composables/`

`src/composables/` 只有 2 檔(`useConfigText` / `useLanguage`),feature 內又散著:

- `features/agent/composables/{useAgentChat, fetch-models, fetch-providers}`
- `features/repair/composables/{useRepairPolish, useRepairSubmit, useRepairTickets}`
- `features/work-collect/composables/useChartOptions`

**規則不一致** —— 全局共用走 `src/composables/`,feature 私用走 `features/*/composables/`,但目前邊界沒人遵守。

#### `src/api/modules/` 半空

`src/api/modules/` 只有 `ai.api.ts` / `auth.api.ts`,
而 `features/work-collect/api.ts` / `features/repair/api.ts` 自帶 API 檔。

兩種風格並存,沒明文規定哪個對。

#### `types/` 雙軌

- `src/types/config/` —— 按 feature 拆好(8 個檔)
- `src/types/config.types.ts` —— 老的 monolith,內容跟 `config/index.ts` 重疊

不知道引哪個。`grep` 顯示兩個都被 import,**等於有兩個 source of truth**。

### 2.4 子窗口入口分散

- 主窗口:`src/main.ts`(常規)
- Agent:`src/entries/agent/{main.ts, index.html}` ← 在 entries 下
- 浮球:`src/floating-ball/{main.ts, App.vue, FloatingBall.vue}` ← 直接平鋪
- 日誌:`src/log-viewer/{LogViewerView.vue, App.vue, main.ts}` ← 直接平鋪

三個子窗口三種放法。

## 3. 重構總原則

### 3.1 業務 / 執行三層

```
┌──────────────────────────────────────────────────────────┐
│  Orchestrator / Manager                                  │
│  (生命週期、注入依賴、event wiring)                       │
└──────────────────────────────────────────────────────────┘
        ▲                  ▲                    ▲
        │                  │                    │
┌──────────────┐   ┌──────────────┐   ┌──────────────────┐
│  Domain      │   │  Domain      │   │  Infrastructure  │
│  (純業務規則) │   │  (純業務規則) │   │  (OS / 框架 API)  │
│  無 IO       │   │  無 IO       │   │  純技術側         │
└──────────────┘   └──────────────┘   └──────────────────┘
```

**規則**:

- Domain 層只接受**值**,只回傳**值**;不持 IO、不發 IPC、不開窗
- Infrastructure 層只跟 OS / 框架 API 交互,**不知道業務規則**
- Orchestrator 層編排,可以調 Domain + Infrastructure,但**自己不寫業務 / 技術細節**

例:`IdleDetector` 不該知道 `powerMonitor`;`isInWorkHours()` 應該是純函式接收 hour 而非從 Config 拿。

### 3.2 文件大小目標

| 類型                 | 目標上限  | 超過怎辦                       |
|--------------------|-------|----------------------------|
| Vue SFC            | 300 行 | 拆 components / composables |
| TS class / service | 250 行 | 拆策略 / 拆責任                  |
| TS 工具函式集           | 150 行 | 拆主題                        |
| Migration / schema | 不限    | 不拆                         |

### 3.3 目錄歸類規則(新)

```
src/
├── api/                       # HTTP 客户端 + 全局拦截器(共用基础)
│   ├── http-client.ts
│   └── interceptors/
│
├── shared/                    # 全 renderer 共用,無 feature 屬性
│   ├── composables/           # ← 原 src/composables/ 改名
│   ├── utils/                 # ← 原 src/utils/
│   └── types/                 # ← 原 src/types/ 内非 config 部分
│
├── stores/                    # 全局 Pinia store(auth / config / ui)
│
├── features/                  # **每個 feature 自治**
│   └── <feature>/
│       ├── api.ts             # 該 feature 的 HTTP 呼叫
│       ├── store.ts           # 該 feature 的 Pinia(如有)
│       ├── types.ts           # 該 feature 內部型別
│       ├── composables/       # 該 feature 私用 composables
│       ├── components/        # 該 feature 私用 components
│       ├── <FeatureView>.vue  # 主入口
│       └── README.md          # **新增** 簡介這個 feature 做什麼
│
├── views/                     # 純 route view(無 feature 邏輯,薄殼引 features/*)
│
├── windows/                   # ← 整合 entries/ / floating-ball/ / log-viewer/
│   ├── main/                  # 主窗口 entry
│   ├── agent/                 # AgentWindow
│   ├── floating-ball/
│   └── log-viewer/
│
├── components/                # **跨 feature 共用** components
│   ├── common/                # 通用元件(IframeContainer 等)
│   └── layout/                # 框架(SidebarNav / TitleBar)
│
├── router/
├── styles/
├── locales/
└── App.vue / main.ts
```

`src/api/modules/` 撤掉,各 feature 自己管 api.ts。

### 3.4 Backend 中的取消廢棄類

舊 8 類分類 `WorkCategory` 寫死 union、`CATEGORY_LABEL_KEY / CATEGORY_COLOR / CATEGORY_ORDER` 等已棄用,
模板化後動態派生(docs/23 §3.6 列了清單)。本文再加一條:

- `src/types/config.types.ts` legacy monolith **整檔刪掉**,全部走 `src/types/config/` 的拆分版
