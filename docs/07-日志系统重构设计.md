## 2. 痛點分析

### 2.1 「只記 ERROR」太激進

當前策略是「沒錯誤就不該有檔案」，但實際排查問題時，**錯誤前後的上下文同樣重要**。例：

- 用戶反映「無法登錄」，後端在 401 前其實有一系列 `INFO` / `WARN`（自動填入帳號、攔截器拋出、Token 刪除等）— 全部不在文件，只能讓用戶開
  DevTools 截圖。
- `autoUpdater` 從「開始檢查」到「下載失敗」中間會打十幾條 `INFO`（HTTP 請求 yml、解析、進度），現在文件裡只有最後一條 ERROR，*
  *幾乎無法回溯**。
- 配置加載失敗時 `WARN` 提示「使用默認配置」 — 這是常見 bug 起點，但不寫文件。

### 2.2 沒有按大小輪轉

只按日期切分（`main-2026-04-29.log`）。
平常一天可能 0 條，啟用更廣的寫入策略後（見下文），高頻錯誤日可能讓單檔達到幾十 MB，文本編輯器打開卡死。

### 2.3 策略硬編碼，無法調整

「寫哪些級別」「保留幾天」全部寫在代碼裡。技術支援想暫時把 `INFO` 也記下來排查，必須改源碼重打包。

### 2.4 缺少跨進程關聯

主、渲染各自一份文件，**沒有 traceId 或會話 ID 把兩邊串起來**。
例：用戶按下「保存配置」後失敗，渲染端寫了一條 ERROR，主進程的 `config.handlers.ts` 也寫了一條，但無法判斷是同一次操作。

### 2.5 沒有 PII 自動脫敏

源碼註釋警告「不要把 token、密碼放進 args」，但無運行時保護。一旦有人寫
`logger.error('登錄失敗', 'auth', { username, password })`，明文密碼就進文件了。

### 2.6 用戶側回報體驗差

`LogSection` 目前只能：

- 打開資料夾（用戶不熟 Windows 檔案總管，常找不到 `.log`）
- 複製路徑

沒有：

- **一鍵打包當日 main + renderer 為 `.zip`**
- 在彈窗內預覽尾部 N 行
- 過濾級別查看

### 2.7 渲染端格式與主進程不對齊

主進程 console 輸出單行 `[ts] [LEVEL][module] msg JSON`，渲染端 console 輸出是 `[ts] [LEVEL][module]` + 散參數（瀏覽器
console 友好，但跟文件格式不一致，使用者切換看會混亂）。

### 2.8 IPC 寫入無批量

每條 renderer 日誌一次 IPC。一般 OK，但 `unhandledrejection` 風暴時可能瞬間數百條，無背壓控制。

### 2.9 部分代碼仍走 `console`

見第 7 節。這些 `console.*` 在生產環境只進 DevTools，**永遠不會落地到文件**，使排查殘缺。

## 3. 設計目標

按優先級排列：

1. **P0 — 生產可排查**：用戶回報問題時，當天 `.log` 應該包含足夠上下文（不只 error）。
2. **P0 — 統一入口**：所有 `console.*` 改走 `logger`；CI 可加 lint 規則禁止新增 `console`。
3. **P1 — 用戶側自助回報**：設定彈窗一鍵導出當日日誌包。
4. **P1 — 大小輪轉**：單檔不超過 5MB，自動切片 `-1.log`、`-2.log`。
5. **P1 — PII 脫敏**：固定關鍵字（`password`、`token`、`authorization`、`secret`）自動打碼。
6. **P2 — 跨進程 traceId**：HTTP 請求、用戶操作可選帶 traceId 關聯。
7. **P2 — 配置可調**：`app-config.json` 加 `logging` 區段，控制級別與保留。
8. **P2 — autoUpdater 獨立文件**：避免大量 updater 日誌淹沒主流程。

## 4. 新方案總覽

### 4.1 寫文件策略改為「文件級別 + 控制台級別」雙旋鈕

| 級別    | console（dev） | console（prod） |  文件（默認）  |
|-------|:------------:|:-------------:|:--------:|
| DEBUG |      ✓       |       ✗       |    ✗     |
| INFO  |      ✓       |       ✗       | ✓（**新**） |
| WARN  |      ✓       |       ✓       |    ✓     |
| ERROR |      ✓       |       ✓       |    ✓     |

`INFO` 也寫文件後，當天的「應用啟動 → 配置加載 → 主窗口創建 → 用戶登錄 → 自動更新檢查」整條時間線都能在文件裡看到。

> 用 `app-config.json` 的 `logging.fileLevel` 可以微調（預設 `info`，遇到問題可臨時調 `debug`）。

### 4.2 文件分檔

| 檔案                        | 寫入來源                                          |
|---------------------------|-----------------------------------------------|
| `main-YYYY-MM-DD.log`     | 主進程業務日誌                                       |
| `renderer-YYYY-MM-DD.log` | 渲染進程業務日誌（含全局錯誤鉤子）                             |
| `updater-YYYY-MM-DD.log`  | **新增**：autoUpdater 的全部 info/warn/error，與主進程隔離 |

### 4.3 大小輪轉

單檔 ≥ 5MB（`logging.rotateMaxBytes`，可配置）→ 切為 `main-YYYY-MM-DD-1.log`，繼續寫 `main-YYYY-MM-DD-2.log`。
保留策略：保留最近 14 天內的所有切片（不額外限切片數）。

### 4.4 結構化（雙模式）

- **dev**：保留現有 plain text 格式（人看），方便終端 grep。
- **prod**：寫入 **JSON Lines**（每行一個 JSON），便於：
    - 自動解析（未來想做 ELK / Loki 收集無痛接入）
    - 安全脫敏（pretty-print 容易遺漏，JSON 結構化字段更可靠）
    - 跨進程關聯（traceId 直接是字段）

```jsonc
// main-2026-04-29.log（生產 JSON Lines 範例）
{"ts":"2026-04-29T03:35:22.123Z","lvl":"INFO","src":"main","mod":"App","msg":"配置加載完成","ctx":{"version":"1.0.2"}}
{"ts":"2026-04-29T03:35:23.001Z","lvl":"ERROR","src":"main","mod":"IPC:auth","msg":"讀取 Token 失敗","err":{"name":"Error","message":"safeStorage unavailable","stack":"..."},"trace":"sess-a1b2"}
```

### 4.5 PII 自動脫敏

寫入前對 `args` 遞迴遍歷，匹配 `redactFields`（默認
`['password', 'token', 'accessToken', 'authorization', 'secret', 'apiKey']`）的鍵值替換為 `'***'`。

```typescript
// 默認規則
logger.error('登錄失敗', 'auth', {username: 'S2403279', password: 'p@ss'})
// 文件裡：
// {..., "ctx": {"username":"S2403279","password":"***"}}
```

### 4.6 渲染端 IPC 批量化

`src/utils/logger.ts` 內部維護一個 buffer：

- 攢滿 32 條 / 1 秒 flush 一次，用單條 IPC 帶 entries 數組
- ERROR 級別立即 flush（不等攢滿）
- 應用退出前最後一次 flush

對應主進程改 `LOG_WRITE` 接收 `entry | entry[]`。

### 4.7 LogSection UI 增強

新增三個交互：

| 功能          | 行為                                                                                                                                  |
|-------------|-------------------------------------------------------------------------------------------------------------------------------------|
| **導出當日日誌包** | 主進程把當日 `main-*.log` + `renderer-*.log` + `updater-*.log` 打包成 `ichiaDesktop-logs-YYYY-MM-DD.zip` 到桌面，並調用 `shell.showItemInFolder` 高亮 |
| **預覽尾部**    | 在彈窗內顯示最近 200 行（只讀），帶級別過濾下拉框                                                                                                         |
| **清空當日日誌**  | 給 QA 用：清空後重現 bug，確保導出的包乾淨；含二次確認                                                                                                     |

### 4.8 跨進程 traceId（可選使用）

提供 `logger.withTrace(traceId)` 方法返回一個帶 traceId 的子 logger。典型用例：

- HTTP 攔截器：每個請求生成 `req-<8字符>`，請求日誌、響應日誌、錯誤日誌都帶同一個 trace
- 用戶操作：登錄、保存配置等操作開始時生成 `op-<8字符>`，渲染端 → 主進程 IPC 帶上，主進程 logger 也用這個 trace

非強制，現有調用點可繼續不帶 trace。

### 5.4 大小輪轉實作

`writeLine(source, line)` 流程：

1. 計算當前活躍檔案路徑：先嘗試 `<source>-<date>.log`，若 size ≥ rotateMaxBytes 則探測 `-1`、`-2`…直到找到 size <
   rotateMaxBytes 的檔。
2. 為避免每次寫入都 `statSync`，**內存緩存**每個 source 的當前檔案路徑與已知大小，每寫一行累加 line.length；累積到一定量時
   stat 校對一次。
3. 文件不存在自動創建。

> 注意：`appendFile` 是異步的，並發 `writeLine` 可能交錯。實際上 Node 對單個 fd 的 appendFile 在 < 4KB
> 寫入是原子的（PIPE_BUF），普通日誌行遠小於這個閾值，可以接受。

### 5.5 格式器

```typescript
// log-formatter.ts
interface LogRecord {
    ts: string         // ISO 8601
    lvl: 'DEBUG' | 'INFO' | 'WARN' | 'ERROR'
    src: 'main' | 'renderer' | 'updater'
    mod?: string
    msg: string
    ctx?: Record<string, unknown>  // args 序列化後（已脫敏）
    err?: { name: string; message: string; stack?: string }
    trace?: string
}

function formatPlain(rec: LogRecord): string  // dev
function formatJson(rec: LogRecord): string   // prod
```

`dev` 模式產出與當前一致的可讀格式；`prod` 產出 JSON Lines。**dev/prod 切換完全由 `app.isPackaged` 決定**，不暴露給用戶。

### 5.6 脫敏

```typescript
// log-redactor.ts
const DEFAULT_REDACT_KEYS = [
    'password', 'token', 'accessToken', 'refreshToken',
    'authorization', 'cookie', 'secret', 'apiKey'
]

export function redact(obj: unknown, keys = DEFAULT_REDACT_KEYS): unknown {
    // 遞迴遍歷對象 / 數組
    // 鍵名匹配（不分大小寫）→ 值替換為 '***'
    // Error 對象只保留 name/message/stack（不會遞迴它的自定義屬性）
    // 字串值若整體看起來像 Bearer token、JWT 也整體打碼（可選）

```

### 5.7 渲染端批量化

```typescript
// src/utils/logger.ts 內部
const BUFFER: LogEntry[] = []
const FLUSH_INTERVAL_MS = 1000
const BUFFER_MAX = 32

function flush(): void {
    if (BUFFER.length === 0) return
    const batch = BUFFER.splice(0, BUFFER.length)
    window.electronAPI.log.write(batch)  // 一次 IPC，數組
}

setInterval(flush, FLUSH_INTERVAL_MS)
window.addEventListener('beforeunload', flush)

// error 級別立即 flush
function enqueue(entry: LogEntry): void {
    BUFFER.push(entry)
    if (entry.level === 'ERROR' || BUFFER.length >= BUFFER_MAX) flush()
}
```

主進程 `LOG_WRITE` 處理改為兼容單筆與數組：

```typescript
ipcMain.on(IpcChannels.LOG_WRITE, (_e, payload) => {
    const entries = Array.isArray(payload) ? payload : [payload]
    for (const entry of entries) writeRendererLog(...entry)
})
```

### 5.8 autoUpdater 獨立文件

`update-manager.ts` 中 `autoUpdater.logger` 接到一個專用實例：

```typescript
import {createLogger} from '../utils/logger'

const updaterLog = createLogger('updater')  // src='updater'

autoUpdater.logger = {
    info: (m) => updaterLog.info(String(m), 'autoUpdater'),
    warn: (m) => updaterLog.warn(String(m), 'autoUpdater'),
    error: (m) => updaterLog.error(String(m), 'autoUpdater'),
    debug: (m) => updaterLog.debug(String(m), 'autoUpdater')
}
```

`createLogger(source)` 返回的 logger 走 `writeLine(source, ...)`，落到 `updater-YYYY-MM-DD.log`。

UpdateManager 自己的業務日誌（如「未配置 dailyCheckTime」）仍走默認 `logger`（src='main'），不混進 updater 文件。

│  日誌資料夾                  [打開] [複製路徑] │

│  ─────  進階  ─────                         │

└───────────────────────────────────────────────┘
```

新增 IPC：

| 頻道                 | 模式     | 行為                                               |
|--------------------|--------|--------------------------------------------------|
| `LOG_EXPORT_TODAY` | invoke | 主進程打包當日日誌為 zip 到桌面，返回 zip 絕對路徑                   |
| `LOG_TAIL`         | invoke | 接 `{ source, lines, levelFilter }`，返回最近 N 行（已過濾） |
| `LOG_CLEAR_TODAY`  | invoke | 刪除當日所有 `*-YYYY-MM-DD*.log` 檔，返回刪除數量              |

打包用 Node 內建沒有 zip 庫，方案二選一：

- **A**：增加 `archiver` 依賴（成熟、API 簡單，~30KB gzip）
- **B**：用 PowerShell 的 `Compress-Archive` 命令（無新依賴，Windows-only 可接受）

> 推薦 **A**，避免外部依賴 PowerShell 路徑與權限。

    "retentionDays": 14,
    "redactFields": ["password", "token", "accessToken", "authorization", "secret", "apiKey"],
    "updaterSeparateFile": true     // false 時 updater 日誌混回 main
  }
}
```

對應類型加在 [src/types/config.types.ts](../src/types/config.types.ts)：

```typescript
export interface LoggingConfig {
    fileLevel: 'debug' | 'info' | 'warn' | 'error'
    consoleLevelDev: 'debug' | 'info' | 'warn' | 'error'
    consoleLevelProd: 'debug' | 'info' | 'warn' | 'error'
    rotateMaxBytes: number
    retentionDays: number
    redactFields: string[]
    updaterSeparateFile: boolean
}

export interface AppConfig {
    // ... 既有 ...
    logging?: LoggingConfig
}
```

`ConfigManager` 補 `getLoggingConfig()` fallback。
**logger 必須在 ConfigManager 加載前能用**（極早期錯誤要記下來），所以 logger 啟動時用內建默認，等 ConfigManager 加載完呼叫
`logger.applyConfig(loggingCfg)` 覆蓋。

## 9. 驗證方式

### 9.1 本地手動

| 場景                                               | 預期                                                                        |
|--------------------------------------------------|---------------------------------------------------------------------------|
| 啟動應用                                             | 當天 `main-*.log` 立即出現「應用初始化完成」等 INFO 行                                     |
| 觸發 401                                           | `renderer-*.log` 有 ERROR + 攔截器上下文；同時段的 `main-*.log` 有 IPC:auth 的 token 操作 |
| 寫入 5MB 後再寫一行                                     | 自動切片到 `-1.log`                                                            |
| 修改 `logging.fileLevel` 為 `warn` 重啟               | INFO 不再進文件                                                                |
| 故意 `logger.error('x', 'y', { password: '123' })` | 文件中對應字段為 `***`                                                            |
| 導出當日日誌包                                          | 桌面出現 `ichiaDesktop-logs-2026-04-29.zip`，解壓含 3 個 `.log`                    |
| 連續 100 條 renderer.error 風暴                       | 主進程 IPC 收到 ≤4 次（批量），文件無丟失                                                 |
| 14 天前的日誌                                         | 下次啟動清理，包括 `-N.log` 切片                                                     |

### 9.2 自動化（建議）

- **redactor 單元測試**：覆蓋嵌套對象、數組、Error、循環引用
- **formatter 單元測試**：plain / json 兩種模式產出對齊
- **rotation 集成測試**：mock fs，驗證大小達標時切片

## 10. 兼容性與回滾

- **API 100% 向後兼容**：`logger.{debug,info,warn,error}` 簽名不變，現有調用點不需動。
- **配置選填**：`logging` 區段缺失時走內建默認，舊 `app-config.json` 直接可用。
- **文件命名兼容**：保留 `main-YYYY-MM-DD.log` 與 `renderer-YYYY-MM-DD.log`，僅新增 `updater-*` 與 `-N` 切片。
- **回滾策略**：階段 B/C 出問題時，恢復「僅 ERROR 落地 + 單檔不切片」需要回滾 `logger.ts` 與 `log-file-writer.ts` 兩個檔；UI
  與配置可獨立保留不影響邏輯。
