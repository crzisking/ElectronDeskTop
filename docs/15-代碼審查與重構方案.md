# 代碼審查與重構方案

> **目的**:在**不改變任何業務行為**的前提下,梳理現有架構問題、給出可分階段執行的重構計畫。
> **範圍**:`electron/`(主進程 / preload / IPC / DB)+ `src/`(4 個獨立 Vue 渲染入口)。
> **不改的**:後端 API 契約、UI 文案、路由結構、業務流程、外部依賴版本。

---

## 0. TL;DR

| 維度          | 健康度      | 主要缺陷                                                              |
|-------------|----------|-------------------------------------------------------------------|
| 主進程分層       | 7.5 / 10 | Agent 表跳過 drizzle migration、IPC 無 input schema、handler context 過大 |
| 渲染端結構       | 6.5 / 10 | `features/` 內部模板不統一、Agent 窗口完全游離、跨進程型別重複定義                        |
| HTTP / 狀態管理 | 8 / 10   | 拦截器跟 Element Plus 硬耦合、Config store 雙向同步契約模糊                       |
| 一致性         | 5.5 / 10 | Agent 窗口 UI / 目錄 / store 全部走自己一套,與其它 feature 斷裂                   |

**整體判斷**:基礎設施(IPC / DB / auth 攔截器 / config 熱更新)設計成熟,**穩定可維護 1–2 年**。
**核心痛點**:Agent feature 是後加的,沒對齊既有模式,放大了「模板不一致」這個既存問題。

**建議執行順序**:

1. **P0(本週)**:Agent feature 對齊既有約定(目錄 / DB / 型別)
2. **P1(本月)**:確立 feature 內部標準形態 + 抽離跨進程共用型別
3. **P2(本季)**:IPC payload schema 驗證 + 攔截器解耦 + Composable 邊界規範化

### 1.2 設計亮點(這些是好的,不要動)

- **IPC channel 集中管理**:`electron/shared/ipc-channels/*.ts` 是 single source of truth,主窗 preload 走 bridge factory
  注入,channel 字串不內聯
- **DB 失敗降級**:Service 寫入失敗只 `console.error`,絕不擴散到 caller(避免「寫日誌失敗 → 又寫日誌 → 又失敗」迴圈)
- **依賴順序嚴格編排**:`main/index.ts` 一處集中 wire-up,DBManager → ConfigManager → WindowManager → ... →
  registerAllHandlers
- **Config 三層體系**:KV singleton + 6 張 collection 表 + dev-owned vs user-owned 區分(`USER_OWNED_KEYS`)
- **401 並發保護**:`auth.interceptor.ts` 用 Promise 級聯,並發 401 全部 await 同一個彈窗 Promise
- **graceful shutdown 統一**:`gracefulShutdown()` 一處清理,tray / update / quit-and-install 三個入口都呼叫它

## 3. 重構方案

> **執行原則**:
> 1. **零業務變更** — 所有改動只動「組織方式」和「型別契約」,不動行為
> 2. **可分階段** — 每個階段獨立可發佈,出問題只回滾該階段
> 3. **可驗證** — 每階段交付前跑完整 `npm run typecheck && npm run build`,並手動驗證關鍵路徑

### 3.1 階段劃分總覽

| 階段         | 範圍                                      | 預估工作量 | 風險 |
|------------|-----------------------------------------|-------|----|
| **P0**     | Agent feature 對齊既有約定                    | 1-2 天 | 低  |
| **P1**     | 確立 feature 標準形態 + 跨進程型別契約               | 3-5 天 | 中  |
| **P2**     | IPC schema 驗證 + 攔截器解耦 + Composable 規範化  | 5-7 天 | 中  |
| **P3**(選做) | 補測試 + magic number 文檔化 + 拆 useAgentChat | 3-5 天 | 低  |

### 4.2 把 Agent DB 表納入 drizzle migration

**現狀**:`electron/main/db/features/agent/service.ts` constructor 內呼叫 `ensureTables()` 用
`CREATE TABLE IF NOT EXISTS` 建表。原因是當初擔心動 drizzle snapshot 會跟既有 6 張 config 表的 migration 衝突。

**目標**:Agent 兩張表(agent_configs / agent_messages)走標準 drizzle migration 流程,跟 logs / work_records /
user_profiles 一致。

**步驟**:

1. 確認 `electron/main/db/features/agent/schema.ts` 的 drizzle table 定義沒問題(已存在)
2. 跑 `npm run db:generate` 讓 drizzle-kit 生成新的 migration SQL
3. 檢查生成的 migration **不會** drop / 重建既有 config 表(對 drizzle snapshot 不熟的話這步要細看 diff)
4. 刪掉 `AgentService.ensureTables()` 方法和 constructor 內的呼叫
5. 既有使用者(本機已有 `agent_configs` / `agent_messages` 表)的情況:drizzle migration 用 `CREATE TABLE IF NOT EXISTS` 也是
   idempotent 的,不會炸

**回滾方案**:萬一 drizzle snapshot 真的把既有表也納入新 migration,直接保留 `ensureTables()`,本步驟跳過。

### 4.3 抽出跨進程共用型別

**現狀**:

- `electron/main/db/features/agent/service.ts` 定義了 `AgentConfig` / `AgentMessage`
- `src/agent/types.ts` 又定義了一份(註解寫「因為 contextBridge 是 plain object 所以重定義」)

實際上型別是 **plain interface**,可以跨進程共用,不會帶 runtime 物件,沒有 contextBridge 序列化問題。

**目標**:

```
electron/shared/types/
└── agent.types.ts        # AgentConfig / AgentMessage / OpenAIToolCall / ToolExecResult / ConversationSummary
```

主進程 `service.ts` 和渲染端 `features/agent/types.ts` 都從 `@shared/types/agent.types` 匯入。

**改動範圍**:

- `src/features/agent/types.ts`:同上,並把 `Window.agentAPI` 的全域宣告留在這

## 6. P2 — IPC 驗證 + 攔截器解耦 + Composable 邊界

### 6.1 引入 zod 做 IPC payload schema 驗證

**現狀**:Handler 接收 payload 後只做字串 trim / 空值檢查。例:

```ts
ipcMain.handle(IpcChannels.AGENT_EXEC_TOOL, async (_e, payload) => {
    const {name, args} = payload || {name: '', args: {}}
    // 假設 payload 是合法的...
})
```

**目標**:每個 handler 用 zod schema 驗證 payload,失敗時統一 reject。

```ts
const ExecToolSchema = z.object({
    name: z.string().min(1),
    args: z.record(z.unknown()),
})

ipcMain.handle(IpcChannels.AGENT_EXEC_TOOL, async (_e, raw) => {
    const payload = ExecToolSchema.parse(raw)  // 失敗自動 throw
...
})
```

**範圍**:從風險最大的 handler 起手 — `agent:exec-tool`(系統工具)/ `repair:upload`(檔案上傳)/ `config:write`(配置寫入)
。其它低風險的 handler 不強制。

**依賴**:`npm install zod`(輕量,~12KB gzipped)。

### 6.2 拦截器解耦

**現狀**:`src/api/interceptors/auth.interceptor.ts` 直接 `import { ElMessage, ElMessageBox } from 'element-plus'`,因此:

- 攔截器無法被單元測試(必須 mock ElMessage)
- 浮球 / log-viewer / Agent 窗口若想復用攔截器,被迫拉進 Element Plus

**目標**:

```ts
// auth.interceptor.ts 改造
export interface InterceptorUiHooks {
    showError(msg: string): void

    showConfirm(msg: string): Promise<void>
}

export function setupAuthInterceptor(instance: AxiosInstance, ui: InterceptorUiHooks) { ...
}
```

呼叫方在 `src/main.ts` 注入 Element Plus 的實作,Agent 窗口可以注入 toast 風格的實作。

### 6.3 Composable 邊界規範化

**規範**:

- 跨 feature 通用 → `src/composables/`(目前 `useAiStream` / `useConfigText` / `useLanguage`)
- 單 feature 內部 → `src/features/{name}/composables/`
- **單檔超過 200 行**或**單函式超過 100 行**就必須拆

**首批拆分對象**:

- `useAgentChat.ts`(280+ 行):拆成 `useAgentChat`(while 循環)+ `useAgentStream`(SSE 解析)+ `useAgentToolDispatch`(
  工具路由)
- `useRepairEditor.ts`(140 行,paste handler 寫死):paste 處理抽到 `useQuillPasteImage` composable
- `useAiStream.ts`:評估是否該下沉到 `features/work-collect/composables/`

### 6.4 拦截器內 `ERROR_KEY_MAP` 廢除

**現狀**:`auth.interceptor.ts` 內有一張 `ERROR_KEY_MAP`(8 項)把舊錯誤碼映射到 i18n key。

**目標**:直接用 i18n key 呼叫,廢掉這張表。一次性 search-replace 工程,改完即可。

### 6.5 Quill 型別補完

**現狀**:`useRepairEditor.ts` 用 `let quillInstance: any = null`。

**目標**:

```ts
import type {Quill} from 'quill'

let quillInstance: Quill | null = null
```

或退而求其次用 JSDoc 標註,至少讓 IDE 補全。

## 8. 不做的事(明確劃線)

避免重構過度膨脹,以下事項**本次方案外**,將來另案處理:

- ❌ 不改業務邏輯(報修流程、工作採集流程、Agent 工具行為)
- ❌ 不換技術棧(Vue / Pinia / Element Plus / Quill / better-sqlite3 / drizzle 全保留)
- ❌ 不引入大型新依賴(zod 是唯一例外,~12KB)
- ❌ 不改 UI 文案(i18n 字典結構可調整,但顯示內容不動)
- ❌ 不動 `app.db` 既有資料結構(只新增 migration,不 alter / drop 既有表)
- ❌ 不改路由名稱(雖然 `views/` vs `features/` 邊界模糊,但改路由 name 會影響埋點 / 文檔)
- ❌ 不改 `app-config.json` 已棄用的相關邏輯(舊版相容代碼留著)
