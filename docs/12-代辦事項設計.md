# 代辦事項 設計

> 同步釘釘待辦 → 本機 SQLite 快取 → 桌面視窗顯示 / 操作 + 定時同步。
> 後端只做 proxy 不存業務待辦資料(但允許短 TTL 快取使用者身份映射)。
> 視窗**懶創建** —— App 啟動不建,使用者第一次點入口才建,之後 close 只 hide 復用。
> **單帳號模型**:一台電腦同時只服務一個 AD 帳號;切換帳號時清空舊紀錄(含工作採集)。

## 設計原則(綁定 MVP 範圍)

- **不過度設計** —— `localDirty` / 離線寫入等狀態機留到實際需要時再加
- **同步流程明確 ack** —— renderer 完成後回報 main,main 持有 lastSyncStatus
- **錯誤可見化** —— SQLite 寫失敗 / 同步失敗 都要進 UI,不靜默吞
- **scope-based cleanup** —— 全量 sync 結束、且所有分頁都成功,才清「scope 內早於 sync 開始時間」的紀錄

## 1. 整體流程

```
登入成功(LoginView / AD auto-login)
   │
   1. 渲染端 → todoStore.syncUserInfo()
      ├─ GET /api/Todo/user-info
      └─ IPC USER_PROFILE_UPSERT → 主進程 UserProfileService.upsert
   ▼
App 啟動完成
   │
   2. 主進程 TodoWindowManager.create()        ← 視窗建出來,但 hidden
   3. 主進程 TodoSyncScheduler.start()         ← timer 啟動(讀 config.todo.syncIntervalMinutes)
   ▼
使用者點「代辦事項」入口(浮球右鍵 / 內部功能卡片 / Tray)
   │
   4. IPC TODO_WINDOW_TOGGLE → 主進程 win.show() + focus
   5. 渲染端 mounted(視窗第一次顯示時)
      ├─ store.refresh()  ← 從本機 SQLite 立刻讀出 cache,UI 秒開
      └─ store.sync()     ← 同時觸發一次背景 sync
   ▼
定時觸發(每 N 分鐘,N 由 config 決定)
   │
   6. TodoSyncScheduler tick → webContents.send(PUSH_TODO_SYNC_TICK)
   7. 渲染端 store 接到事件 → store.sync()
      a. GET /api/Todo/list                       一次
      b. 若 resp.nextToken 非空 → 帶 nextToken 再請求,直到 next 為空
      c. for each todoCards[i] → IPC TODO_UPSERT
      d. cleanupNotIn:本機多出的 taskId(釘釘端已刪)清掉
      e. PUSH_TODO_UPDATED → UI 反應式更新
   ▼
使用者關閉視窗(✕ / ─ 按鈕)
   │
   8. IPC TODO_WINDOW_HIDE → win.hide()  ← 視窗不 destroy,timer 繼續
   ▼
使用者「結束應用程式」
   │
   9. gracefulShutdown → TodoSyncScheduler.stop() → TodoWindowManager.destroy()
```

**為什麼快取在本機 SQLite**:
- 離線 / 弱網狀態仍可看待辦(只是不能改)
- 視窗 mount 不必等釘釘 HTTP,先讀 cache 0.1 秒呈現
- 後端不存業務資料,符合「每人自己的事自己存」隱私邊界

**為什麼前端改不直接打釘釘**:
- 釘釘 OpenAPI 需要 corpId + accessToken,token 是機密不能進 client
- 走 tmbom 中轉:client 帶 JWT → tmbom 驗 JWT + 取釘釘 token + 呼叫釘釘 → 回傳 → client 更新本機 cache

## 2. 釘釘契約對齊

### 列表回傳(已知)

```jsonc
{
  "nextToken": "...",            // string,空表示已到最後一頁
  "todoCards": [
    {
      "taskId":       "string",
      "subject":      "string",
      "dueTime":      1779155657742,    // Long,Unix ms
      "detailUrl":    { /* object */ }, // 詳情頁鏈接,釘釘原樣 object
      "priority":     10,               // Integer,釘釘規範 10/20/30/40
      "createdTime":  1779000000000,    // Long
      "modifiedTime": 1779100000000,    // Long
      "creatorId":    "string",         // 創建者 dingId
      "sourceId":     "string",
      "bizTag":       "string",         // 所屬應用
      "isDone":       false,
      "todoType":     "TODO"            // "TODO" | "READ"
    }
  ]
}
```

### priority 對應(釘釘官方 4 級)

| 釘釘值 | 我方語義 | UI 顯示 |
|---|---|---|
| 10 | 低 | 灰色 dot |
| 20 | 普通 | 藍色 dot |
| 30 | 高 | 橘色 dot |
| 40 | 緊急 | 紅色 dot |

### `detailUrl` 結構

釘釘文檔給 object 形狀,實際打過接口後補完整 schema。本期先存整包 JSON 字串。

## 3. SQLite Schema

新增 **2 張表**,放在 `electron/main/db/features/todo/schema.ts`。

### user_profiles

存登入用戶的釘釘 ID 等資訊。**單機應用只會有 1 筆 active 行**,但用 userId 為 PK 為未來「切換帳號」場景預留。

```ts
user_profiles
  userId          TEXT PRIMARY KEY     -- 公司工號 / JWT subject
  dingId          TEXT NOT NULL        -- 釘釘員工 ID
  unionId         TEXT NOT NULL        -- 釘釘 unionid
  displayName     TEXT                 -- 顯示名,可空
  email           TEXT                 -- email,可空
  lastSyncAt      INTEGER              -- 上次 user-info 更新時間 Unix ms
```

每次登入都 upsert → 釘釘 ID 永遠最新(對應「每次登錄都要重新獲取」需求)。

### todo_items

對齊釘釘 todoCard,加幾個本機欄位。

```ts
todo_items
  taskId          TEXT PRIMARY KEY     -- 直接用釘釘 taskId
  userId          TEXT NOT NULL        -- 屬於哪個登入用戶
  subject         TEXT NOT NULL
  dueTime         INTEGER              -- Unix ms,可空
  detailUrl       TEXT                 -- 釘釘 object 序列化 JSON 存
  priority        INTEGER NOT NULL DEFAULT 20
  createdTime     INTEGER NOT NULL
  modifiedTime    INTEGER NOT NULL
  creatorId       TEXT
  sourceId        TEXT
  bizTag          TEXT
  isDone          INTEGER NOT NULL DEFAULT 0     -- SQLite 沒 boolean,用 0/1
  todoType        TEXT NOT NULL                  -- 'TODO' | 'READ'

  -- 本機額外欄位
  syncedAt        INTEGER NOT NULL               -- 上次從釘釘 pull 下來的 Unix ms
  localDirty      INTEGER NOT NULL DEFAULT 0     -- 0=已同步 / 1=本機有未推送變更

INDEXES:
  idx_todo_userId        (userId)                              -- 多用戶切換查詢
  idx_todo_isDone_due    (userId, isDone, dueTime)             -- 「未完成 + 按截止排序」UI 主查詢
```

**`localDirty` 欄位**:離線 / 網路失敗時本地先打標,下次 sync 時優先推送。MVP 可以先不實作離線寫入,但欄位保留。

## 4. 後端契約(tmbom)

純 proxy,本身不存業務資料。所有接口都從 JWT 拿 userName,**禁止信任前端傳的 userId / dingId**。

| Method | Endpoint | 用途 | 釘釘對應 |
|---|---|---|---|
| GET | `/api/Todo/user-info` | 拿當前 JWT 用戶的 dingId + unionId | 公司 LDAP / 釘釘 user.get |
| GET | `/api/Todo/list?nextToken={token}&isDone={bool}` | 拉待辦列表(分頁) | `dingtalk.todo.list` |
| POST | `/api/Todo/create` | 新增待辦 | `dingtalk.todo.create` |
| PUT | `/api/Todo/{taskId}` | 修改待辦 | `dingtalk.todo.update` |
| PATCH | `/api/Todo/{taskId}/done` | 標記完成 / 取消完成 | `dingtalk.todo.updateExecutorStatus` |
| DELETE | `/api/Todo/{taskId}` | 刪除待辦(如釘釘支援) | `dingtalk.todo.delete` |

### user-info 回傳

```jsonc
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "dingId":      "manager0xxxxxxx",
    "unionId":     "xxxxxxxxxxxxxxxx",
    "displayName": "張三",
    "email":       "zhang.san@example.com"
  }
}
```

### list 回傳(透傳釘釘格式 + unified response wrapper)

```jsonc
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "nextToken": "...",
    "todoCards": [ /* 同第 2 節 */ ]
  }
}
```

### 後端責任邊界

| 後端做 | 後端不做 |
|---|---|
| 從 JWT 解 userName | 不從 body 信任 userId |
| 查 LDAP / 釘釘 user.get 取 dingId/unionId | 不快取使用者 → 釘釘對應(每次重查) |
| 持有釘釘 corpId + accessToken(secret) | 不存待辦業務資料 |
| 把釘釘 API 響應透傳給前端 | 不對 todoCards 做業務邏輯 |
| 日誌記錄呼叫釘釘 API 結果 | 不對待辦做審計(屬於釘釘那邊) |

## 5. 使用者資訊獲取流程

**觸發時機**:登入成功(LoginView 表單登入 / authStore.loginByAd 自動登入)後立刻呼叫。

```
authStore.setToken(jwt) 成功
   │
   1. 渲染端 → workCollectStore.bootstrap()(現有)
   2. 渲染端 → todoStore.syncUserInfo()             ← 新增
      ├─ GET /api/Todo/user-info
      ├─ 拿到 dingId + unionId
      └─ IPC USER_PROFILE_UPSERT → 主進程 UserProfileService.upsert
   3. 失敗:console.warn,不阻塞登入(代辦視窗開啟時顯示「未取得釘釘資訊,請聯絡管理員」)
```

**為什麼每次登入都重抓**:
- 使用者可能在釘釘端被重新分配 dingId(極少數,但有發生)
- 顯示名 / email 變動就要同步
- 簡化邏輯 —— 不用判斷「上次抓多久了該不該重抓」,登入 = 重抓

## 6. 同步策略 + 定時設定

### 6.1 三種觸發時機

| 時機 | 行為 |
|---|---|
| **手動** | 視窗標題列「🔄 重新整理」按鈕,任何時候可觸發 |
| **視窗顯示時** | hide → show 轉換時自動 sync 一次(如果上次 sync 超過 5 分鐘) |
| **定時** | TodoSyncScheduler 每 N 分鐘自動觸發,N 由 config 決定 |

### 6.2 定時間隔可配置

```ts
// config.todo.syncIntervalMinutes
// 預設 30(分鐘),允許範圍 5-180,使用者在代辦視窗「設定」彈窗內調整
```

**UI**:代辦視窗右上角齒輪 → 彈出小設定面板:
```
┌─ 同步設定 ──────────────┐
│ 自動同步間隔            │
│  ○ 30 分鐘              │
│  ○ 1 小時               │
│  ● 自訂  [60] 分鐘      │
│                         │
│ ☑ 視窗顯示時自動同步    │
└─────────────────────────┘
```

寫入走既有的 `CONFIG_WRITE` IPC,主進程 `TodoSyncScheduler` 監聽 config 變動後重啟 timer(類似 work-collector 的 restart)。

### 6.3 全量 sync 演算法

```ts
async function syncTodos(userId: string) {
  let nextToken: string | null = null
  const seen = new Set<string>()
  do {
    const resp = await todoApi.list(nextToken)
    for (const card of resp.todoCards) {
      todoSvc.upsertFromDingTalk(card, userId)
      seen.add(card.taskId)
    }
    nextToken = resp.nextToken || null
  } while (nextToken)

  // 本地「不在這次 list 內」的紀錄 → 釘釘端已刪 / 已歸檔
  todoSvc.cleanupNotIn(seen, userId)
}
```

### 6.4 修改 / 完成 / 新增:write-through

```
UI 操作(例如點「完成」)
   │
   1. 渲染端 store.markDone(taskId)
   2. PATCH /api/Todo/{taskId}/done
   3. 成功 → 本地 SQLite isDone=1 + syncedAt 更新
              webContents.send(PUSH_TODO_UPDATED)
   4. 失敗 → toast 提示,本地不變(MVP);
            未來可改成本地 isDone=1 + localDirty=1(離線寫入)
```

### 6.5 視窗 hidden 時 timer 行為

**timer 繼續跑**(只要 app 沒退出)。理由:
- 開啟視窗瞬間想看到最新資料,不希望「卡 5 秒等同步」
- 釘釘端有新事項可以在 Tray 圖示 / 通知區顯示「您有 N 筆未讀待辦」(後續功能,MVP 不做)

如果發現「視窗整天沒打開但 timer 一直打釘釘 API」消耗太大,可改成「視窗 hide 後 N 分鐘停 timer,show 時恢復」,但不在 MVP 範圍。

### 7.2 TodoWindowManager 職責(常駐視窗)

```ts
class TodoWindowManager {
  private win: BrowserWindow | null = null

  /** App 啟動時呼叫:把視窗建出來但 hidden,後續 toggle 才 show */
  create(): void

  /** 切換顯示;若視窗未建(理論不該發生)會自動 create */
  toggle(): void

  /** 顯示視窗並 focus */
  show(): void

  /** 隱藏視窗(✕ 按鈕觸發);不 destroy */
  hide(): void

  /** 取得視窗引用,主進程 scheduler push 事件用 */
  getWindow(): BrowserWindow | null

  /** App 退出時呼叫,完全銷毀 */
  destroy(): void
}
```

**為什麼不寫進 WindowManager**:WindowManager 已經管 main / floating-ball / log-viewer 三個視窗。Todo 視窗自身行為(always-on-top + dock 到右側 + 定時 sync 配合)夠 specialized,拆獨立 class 後續好維護。**邏輯上跟 FloatingBallManager 同層級**。

### 7.4 TodoSyncScheduler 職責(定時器)

```ts
class TodoSyncScheduler {
  private timer: NodeJS.Timeout | null = null

  start(): void          // 讀 config.todo.syncIntervalMinutes,setInterval
  stop(): void           // clearInterval
  restart(): void        // config 變動後重新 setInterval
  dispose(): void        // gracefulShutdown 內呼叫
}
```

每次 tick:`win.webContents.send(PUSH_TODO_SYNC_TICK)`。渲染端 store 接到 push → 呼叫 store.sync()。

**為什麼推播給渲染端 sync,而不是主進程直接打 HTTP**:跟 work-collect 一樣的理由 —— renderer 有 createHttpClient 攔截器、auth、unified error handling。token 不必 IPC 同步給 main。

## 8. IPC 通道

新增 `electron/shared/ipc-channels/todo.ts`:

```ts
export const TodoChannels = {
  // ── 視窗控制 ──
  /** 開啟 / 切換代辦視窗顯示。send(渲染端 → 主進程) */
  TODO_WINDOW_TOGGLE:        'todo-window:toggle',
  /** 隱藏代辦視窗(✕ 按鈕,不 destroy)。send */
  TODO_WINDOW_HIDE:          'todo-window:hide',

  // ── 業務 IPC ──
  /** 寫 user_profiles(登入後同步釘釘 ID 用)。invoke。返回:boolean */
  USER_PROFILE_UPSERT:       'todo:upsert-user',

  /** 查 todo 列表(本機 SQLite)。invoke。
   *  payload: { userId, isDone?: boolean, limit?, offset? }
   *  返回:TodoItem[] */
  TODO_LIST:                 'todo:list',

  /** upsert 一筆 todo(sync 拉下來時 / write-through 後本地落地)。invoke */
  TODO_UPSERT:               'todo:upsert',

  /** 刪除一筆 todo(本地;釘釘端 by caller proxy) */
  TODO_DELETE:               'todo:delete',

  /** PUSH:scheduler tick,渲染端接到後呼叫 store.sync() */
  PUSH_TODO_SYNC_TICK:       'push:todo-sync-tick',

  /** PUSH:本地 todo 更新完成,UI 刷新 */
  PUSH_TODO_UPDATED:         'push:todo-updated',
} as const
```

加進 `ipc-channels/index.ts` 的 spread。

`PUSH_TODO_*` 加進 preload 的 `ALLOWED_PUSH_CHANNELS` 白名單。

### 9.4 api.ts

```ts
import {createHttpClient} from '@/api/http-client'

const BASE_URL = (import.meta.env.VITE_TODO_API_URL as string | undefined) ?? 'http://localhost:5247'

let _client: ReturnType<typeof createHttpClient> | null = null
const client = () => (_client ??= createHttpClient(BASE_URL, 30000))

export const todoApi = {
  userInfo:  ()                           => client().get('/api/Todo/user-info'),
  list:      (nextToken?: string | null)  => client().get('/api/Todo/list', { params: { nextToken } }),
  create:    (payload)                    => client().post('/api/Todo/create', payload),
  update:    (taskId, p)                  => client().put(`/api/Todo/${taskId}`, p),
  markDone:  (taskId, done)               => client().patch(`/api/Todo/${taskId}/done`, { isDone: done }),
  remove:    (taskId)                     => client().delete(`/api/Todo/${taskId}`),
}
```

新增 `VITE_TODO_API_URL` 環境變數(同 tmbom host,獨立變數,跟 repair / work-collect 分開語意)。

## 11. 視窗生命週期(常駐 + toggle)

### 11.1 行為總覽(跟浮球同模型)

| 階段 | 行為 |
|---|---|
| **App 啟動** | TodoWindowManager.create() 把視窗建出來,**show: false 隱藏狀態**;TodoSyncScheduler.start() 開始定時 |
| **使用者開啟** | IPC TODO_WINDOW_TOGGLE → win.show() + focus + 觸發一次 sync |
| **使用者關閉(✕)** | IPC TODO_WINDOW_HIDE → win.hide() **不銷毀**,timer 繼續跑 |
| **使用者再開啟** | win.show() —— 視窗狀態 / 滾動位置都還在(沒重建) |
| **主視窗關到 tray** | 代辦視窗獨立於主視窗,主視窗藏起來不影響代辦視窗顯示狀態 |
| **完全退出 app** | gracefulShutdown → TodoSyncScheduler.stop() → TodoWindowManager.destroy() |

### 11.2 進入點

| 進入方式 | 觸發 |
|---|---|
| 浮球右鍵 →「代辦事項」 | quickMenu 新加一項,action: `{type: 'open-todo-window'}` |
| **個人功能** 主功能頁 →「代辦事項」卡片 | 卡片 `openMode: 'native-window'`,PersonalFunctionsView 派發 IPC `TODO_WINDOW_TOGGLE` |
| Tray 右鍵菜單(可選) | — |

### 11.3 alwaysOnTop 例外

- 全螢幕應用(看影片 / 開會)時自動降為普通層級(`win.setAlwaysOnTop(false)`),退出全螢幕恢復

### 11.4 多螢幕 / 位置記憶

- 視窗位置 / 大小記在 `app-config.json` 的 `todo.lastBounds`(類似浮球)
- 下次開窗從記憶位置開
- 螢幕拔掉時若位置在不存在的螢幕 → fallback 到主螢幕右側

## 12. 「個人功能」主功能(與內部功能 / 統一平台同層級)

### 12.1 動機與定位

現有 sidebar 兩個主功能:
- **統一平台** — 公司各系統入口(ERP / BPM / 泛微 等),走外部 URL
- **內部功能** — 流程工具卡片(報修 / AI SOP / BPM 查詢 等),走應用內路由

「**工作自動採集**」和「**代辦事項**」性質都不同:
- 不是「打開外部系統」也不是「執行單次流程」
- 而是**為使用者個人服務、會持續產生資料**的功能
- 跟現有兩個主功能語意不對齊,塞進「內部功能」會造成卡片性質混亂

**解法**:在 sidebar 加第三個**主功能**「**個人功能**」,跟「統一平台」「內部功能」**同層級**。
工作採集、代辦事項作為「個人功能」下的子功能(類似內部功能下有 BPM 查詢 / AiSop 那樣的結構)。

## 14. 資料邊界 & 錯誤處理

### 資料邊界

| 資料 | 位置 | 保留期 |
|---|---|---|
| dingId / unionId | userData/app.db / user_profiles | 登出時保留;切換帳號時 upsert |
| todo cache | userData/app.db / todo_items | 永久(視窗開啟時全量 sync,差集會被 cleanup) |
| accessToken(釘釘 corp token)| **後端記憶體** | 後端內部 cache,前端永不接觸 |
| JWT | 渲染端 Pinia | 不持久化,跟現行 auth 一致 |

### 錯誤處理

| 錯誤 | 行為 |
|---|---|
| sync 過程 list 接口失敗 | UI 上方顯示「同步失敗」+ 重試按鈕;本地 cache 仍可瀏覽 |
| user-info 接口失敗 | todoStore.syncError 設訊息;打開視窗時提示「未取得釘釘資訊」 |
| markDone 失敗(網路) | toast 紅色提示,本地不變(MVP)|
| markDone 失敗(釘釘拒絕,業務 4xx) | toast 顯示釘釘錯誤訊息 |
| 釘釘 token 過期 | 後端負責 refresh,前端不關心 |
| SQLite 寫入失敗 | console.error,不擴散到 UI(下次 sync 自然修復) |
| 配置 syncIntervalMinutes 改動 | scheduler.restart() 重設 timer,即時生效 |
