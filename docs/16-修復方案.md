## 1. 真問題修復方案(逐條列)

### 1.1 M1 — Agent DB 表納入 drizzle migration

**現狀**:`AgentService` constructor 內呼叫 `ensureTables()` 用 `CREATE TABLE IF NOT EXISTS` 建表。

**目標**:跟 logs / work_records / user_profiles 一致,走 drizzle migration。

**步驟**:

1. 確認 `electron/main/db/features/agent/schema.ts` 的 drizzle table 定義完整(已存在)
2. 跑 `npm run db:generate` 讓 drizzle-kit 生成 migration SQL
3. **檢查生成的 migration diff**:
    - ✅ 預期:只新增 `agent_configs` / `agent_messages` 兩張表 + `idx_agent_messages_conv` index
    - ❌ 警告:若 drizzle 想 drop / 重建既有 6 張 config 表 → 立即放棄本步驟,保留 `ensureTables`
4. 確認 OK 後,刪掉 `AgentService.ensureTables()` 方法和 constructor 內呼叫
5. 既有使用者升級:drizzle migration 用 `CREATE TABLE IF NOT EXISTS`,跑過 `ensureTables` 的本機資料不受影響

**驗收**:

- 全新環境(無 `app.db`)啟動 → 兩張表自動建好
- 既有環境(有 `app.db` + 已建表)啟動 → 不報錯,對話歷史保留

**回滾**:若 step 3 看到危險 diff,本整節跳過。

### 1.2 M3 — 高風險 handler 加 zod 驗證

**現狀**:Handler 接收 payload 只做字串 trim / 空值檢查。

**目標**:三組高風險 handler 加 zod schema 驗證。

**依賴**:`npm install zod`(~12KB gzipped)。

**範圍**:
| Handler | 風險 | Schema 要點 |
|---|---|---|
| `agent:exec-tool` | 系統工具(可能執行任意命令) | `name: enum(['open_app','read_file',...])`、`args: record(unknown)` |
| `repair:upload`(若有) | 檔案上傳 | `file: instanceof(File)`、size 上限、mime 白名單 |
| `config:write` | 配置寫入(影響應用行為) | `partial: deepPartial(AppConfigSchema)` |

**實作模板**:

```ts
// electron/main/ipc-handlers/agent.handlers.ts
import {z} from 'zod'

const ExecToolSchema = z.object({
    name: z.string().min(1),
    args: z.record(z.unknown()),
})

ipcMain.handle(IpcChannels.AGENT_EXEC_TOOL, async (_e, raw) => {
    const result = ExecToolSchema.safeParse(raw)
    if (!result.success) {
        logger.warn('agent:exec-tool payload 驗證失敗', 'Agent', result.error)
        return {ok: false, content: '', error: 'payload schema invalid'}
    }
    return await executeTool(result.data.name, result.data.args)
})
```

**灰度策略**:

- 上線前 1 週用 `console.warn` 模式(schema 失敗只記 log,仍放行原 payload)
- 確認沒有誤殺合法請求後,切到 `result.error` 直接 reject

**不做**:其它低風險 handler(`window:*` / `log:*` / `floating-ball:*`)維持現狀,避免過度工程化。

### 1.7 R5 — useAgentChat 拆分

**現狀**:`src/agent/useAgentChat.ts` 接近 300 行單一函式,while 循環 + stream 解析 + tool dispatch + 看門狗全混。

**目標**:拆 3 個 composable:

```
features/agent/composables/
├── useAgentChat.ts        # 對外入口:sendMessage / abort / status
│                          # 內部 while 循環、user msg 加入、錯誤分類(401/abort/其它)
├── useAgentStream.ts      # OpenAI stream 解析:text delta + tool_calls 累積 + 看門狗
└── useAgentTools.ts       # 工具執行 dispatch + 結果截斷 + 注入 tool message
```

**關鍵**:**邏輯 100% 保留**,只是拆檔。每個 composable 內部行為跟現在一致。

**拆分原則**:

- `useAgentStream` 接收一個 stream iterator,返回 `{contentBuf, toolCalls, finishReason}` — 純函式,容易測
- `useAgentTools` 接收 toolCalls 陣列,執行並產生 tool messages — 純函式
- `useAgentChat` 負責協調:呼叫 stream → 拿結果 → 跑工具 → 下一輪

### 1.8 R6 — 攔截器解耦

**根據決策**:加注入式 UI hooks。

**現狀**:

```ts
// src/api/interceptors/auth.interceptor.ts
import {ElMessage, ElMessageBox} from 'element-plus'
// ... 直接呼叫 ElMessage.error / ElMessageBox.alert
```

**目標**:

```ts
// src/api/interceptors/auth.interceptor.ts
export interface InterceptorUiHooks {
    /** 顯示一條錯誤訊息(對應 ElMessage.error) */
    showError(msg: string): void

    /** 顯示確認彈窗(對應 ElMessageBox.alert) */
    showAlert(msg: string, title?: string): Promise<void>
}

export function setupAuthInterceptor(
    instance: AxiosInstance,
    ui: InterceptorUiHooks,
) { ...
}
```

呼叫方:

```ts
// src/api/http-client.ts
import {ElMessage, ElMessageBox} from 'element-plus'

const elementPlusUiHooks: InterceptorUiHooks = {
    showError: (msg) => ElMessage.error(msg),
    showAlert: (msg, title) => ElMessageBox.alert(msg, title ?? 'Warning'),
}

export function createHttpClient(baseURL, timeout) {
    const instance = axios.create({...})
    setupAuthInterceptor(instance, elementPlusUiHooks)
    return wrap(instance)
}
```

**好處**:

- 攔截器可單元測試(mock InterceptorUiHooks)
- 將來 Agent 窗口若直連後端,可注入 toast 風格的 hooks
- 主窗呼叫方零改動

**驗收**:現有 401 / 403 / 404 / 業務碼錯誤行為一致。

```ts
// src/utils/uuid.ts
export function uuid(): string {
```

### 2.2 R3 — features / views 邊界

**決策依據**:`views/` 是「頁面骨架 + 容器頁」,`features/` 是「業務模塊本體」。

**文檔化**:在 `docs/05-開發規範.md` 加一段:
> ### `views/` vs `features/` 邊界
> - `views/`:路由直接掛載的頁面,負責**佈局、tab 切換、子模塊編排**。不寫核心業務邏輯。
    >

- 例:`UnifiedPlatform/UnifiedPlatformView.vue` 是統一平台入口,內部編排平台卡片(`SystemCard.vue`)

> - 例:`InternalFunctions/InternalFunctionsView.vue` 列出 `internalFunctions.tools`,點擊跳到 feature 內部
> - `features/{name}/`:業務模塊,包含 api / types / store / composables / components / view
    >

- 例:點「IT 報修」卡片 → 路由到 `features/repair/ITRepairView.vue`

>
> **判斷準則**:打開一個檔案,問自己「這裡的邏輯只有本頁需要,還是其它頁也可能需要?」— 只本頁需要 → `features/`,跨頁通用 →
`components/` 或 `composables/`。

### 2.3 X2 — Config 同步契約

**現狀**:`config.store.ts` 透過 `electronAPI.config` 讀寫,主進程 `PUSH_CONFIG_CHANGED` 觸發後渲染端重新拉。但這套契約沒文檔。

**目標**:新增 `docs/17-跨進程契約.md`,描述:

```markdown
## Config 雙向同步

**來源真理**:`app.db` 的 7 張 config 表(KV + 6 collection)是 single source of truth。
`ConfigManager.config` 是 in-memory cache,渲染端 `configStore.appConfig` 也是 cache。

**讀取路徑**(渲染端 → 主進程):

1. 渲染端 `configStore.loadConfig()` 呼叫 `electronAPI.config.read()`
2. 主進程 `CONFIG_READ` handler 返回 `configManager.getConfig()`(in-memory cache)
3. 渲染端 cache 同步

**寫入路徑**(渲染端 → 主進程 → 廣播):

1. 渲染端 `configStore.writeConfig(partial)` 呼叫 `electronAPI.config.write(partial)`
2. 主進程 `CONFIG_WRITE` handler:
   a. `applyPartial(db, partial)` 落地 DB
   b. 重新 `assembleAppConfig(db)` 更新 in-memory cache
   c. 廣播 `PUSH_CONFIG_CHANGED` 給所有渲染端
3. 渲染端收到 push 後重新 `loadConfig()`

**衝突解決**:無多寫者場景。所有寫入都走主進程 transaction,序列化保證。
渲染端的 `appConfig` 是 read-only 投影,絕對不在渲染端 mutate。
```

## 3. 已決定不動的(明確列出 + 理由)

| ID     | 描述                              | 為何不動                                                          |

| **M2** | `IpcHandlerContext` 注入 11 個依賴   | 用 context object 是刻意選擇,優於位置參數;新增 handler 拿全部依賴是輕微缺陷,改動 ROI 太低 |
