# 功能設計文檔

## 1. 登錄功能

### 設計說明

登錄功能已完整實現，對接公司 Portal OAuth 登錄接口：

- **接口**：`POST {VITE_AUTH_BASE_URL}/api/portal/oauth/login`
- **請求體**：`{ username, password }`
- **後端回應**：`{ code: 200, message: "...", data: { user: UserProfile, token: string } }`

登錄成功後：
1. 後端回傳的 Token 直接寫入 `authStore.accessToken`（Pinia `ref`，僅內存）
2. 後續 Axios 請求由攔截器自動帶上該 Token
3. **不做任何磁盤持久化**，App 關閉即清除；下次啟動會被路由守衛導回 `/login`

### 功能狀態

| 功能項 | 文件位置 | 狀態 |
|--------|----------|------|
| 路由 `/login` | `src/router/index.ts` | ✅ 已實現 |
| 登錄表單頁面 | `src/views/Login/LoginView.vue` | ✅ 工號 + 密碼 + 錯誤提示 |
| 認證 Store | `src/stores/auth.store.ts` | ✅ login / logout（僅內存） |
| 登錄接口 | `src/api/modules/auth.api.ts` | ✅ POST /api/portal/oauth/login |
| 自動注入 Token | `src/api/interceptors/auth.interceptor.ts` | ✅ 所有請求自動加 Authorization 頭 |

### 路由守衛行為

每次路由跳轉前執行，邏輯如下：

1. 若 `isRestoringSession === true` → 直接放行（App 啟動恢復會話期間）
2. 若目標路由 `requiresAuth === true` && 未登錄 → 強制跳轉 `/login`
3. 若已登錄 && 試圖訪問 `/login` → 跳轉 `/unified-platform`

### 開發環境自動登錄

開發時不想每次手動輸入帳號密碼，在 `.env.development` 配置：

```bash
VITE_DEV_AUTO_LOGIN=true
VITE_DEV_USERNAME=S2403279
VITE_DEV_PASSWORD=yourpassword
```

實現方式：`LoginView.vue` 的 `onMounted` 把帳號密碼填入 `form` 後調用 `handleLogin()`，走正常的表單驗證 + 登錄流程。

> **注意**：是「填入 form 再觸發 handleLogin()」，不是直接繞過表單調用 `authStore.login()`。這樣保留了表單驗證，UI 上也能看到填入的帳號，方便調試。自動登錄代碼有 `import.meta.env.DEV` 守衛，不會進入生產包。

### 重要：HTTP 回應資料層提取

理解這一點可避免踩坑（曾經有 Bug 就出在這裡）：

```
後端回傳原始 JSON
{ code:200, message:"...", data:{ user:{...}, token:"eyJ..." } }
        ↓
Axios 響應攔截器（auth.interceptor.ts）
return response.data
        ↓ 攔截器返回整個 JSON body
{ code:200, message:"...", data:{ user, token } }
        ↓
auth.api.ts: const { data } = await client.post(...)
        ↓ 再解構一層 data
{ user:{...}, token:"eyJ..." }   ← authApi.login() 的真實返回值
```

所以調用 `authApi.login()` 後**直接解構 `{user, token}`**，不要再讀 `.code` 或 `.data`。業務碼的錯誤（非 200）已由攔截器統一通過 `ElMessage.error` 顯示。

## 2. 統一平台頁面

### 功能概述

為用戶提供公司各內部系統的統一入口，支持：
- 系統卡片展示（名稱、描述）
- 點擊後按 `openMode` 決定打開方式
- SSO 直通：自動將 Auth Token 注入 URL，實現單點登錄

### 三種打開方式

| `openMode` | 行為 |
|------------|------|
| `iframe` | 在主窗口內嵌入 iframe，左上角有「返回列表」按鈕 |
| `electron-window` | 打開一個新的 Electron 子窗口加載系統 URL |
| `external-browser` | 調用 `shell.openExternal(url)`，在系統默認瀏覽器打開 |

### SSO 直通邏輯

當系統配置了 `ssoEnabled: true` 時：

1. 從 `authStore` 讀取當前 `accessToken`
2. 根據 URL 是否已有查詢字串，用 `?` 或 `&` 拼接 `{ssoTokenParam}={token}`
3. 用拼接後的 URL 按 `openMode` 打開

當 `ssoEnabled: false` 時，直接使用原始 URL，不注入 Token。

### 核心組件

- **`UnifiedPlatformView.vue`** — 卡片列表頁，讀取 `configStore.platformSystems`
- **`SystemCard.vue`** — 單個系統卡片，展示名稱和描述，點擊觸發 open 事件
- **`SystemLauncher.vue`** — 處理打開邏輯（iframe 模式時也負責渲染 iframe 容器）

## 3. 內部功能頁面

### 功能概述

內部功能是一個**工具卡片入口頁**，頁面本身不直接提供功能，而是展示各工具的入口卡片。

顯示哪些工具卡片，完全由 `app-config.json` 的 `internalFunctions.tools` 決定，不需要改代碼就能增減卡片。

| 層級 | 實現 |
|------|------|
| 入口頁 | `InternalFunctionsView.vue`（路由：`/internal-functions`） |
| 配置驅動 | `app-config.json → internalFunctions.tools` |
| 子功能頁 | 各自獨立路由，卡片點擊後跳轉 |

### 當前已有的子功能

**BPM 負責人查詢**（路由：`/ai-bpm-finder`）

- 嵌入 Dify chatbot 的 iframe，URL 由 `app-config.json` 配置
- 用戶可以用自然語言查詢業務流程負責人

**AiSop 生成**（路由：`/ai-sop`）

- 嵌入 Dify chatbot 的 iframe，並提供「上傳生成的 SOP」入口
- 後端服務地址透過 `VITE_AI_SOP_URL` 環境變量切換（測試 / 正式）
- 對應 API 模塊 `src/api/modules/aisop.api.ts`

**IT 報修工單**（路由：`/it-repair`）

- 見下方第 4 節

## 4. IT 報修工單

### 功能概述

員工可以通過此頁面提交設備故障或 IT 問題，IT 人員接收後處理。

功能分兩個 Tab：

| Tab | 功能 |
|-----|------|
| 提交報修 | 填寫工單標題 + 富文本描述（可直接貼圖），提交後進入工單列表 |
| 我的工單 | 查看自己提交的工單列表，可按狀態篩選，點擊行查看詳情 |

頁面左上角另有「返回」按鈕（`router.back()`；若歷史為空則回退到 `internal-functions`），
避免此頁為 session 首個路由時，返回停留在空白路由。

### 代碼結構

IT 報修的代碼按職責拆分：

```
ITRepairView.vue              ← 主視圖，只負責組裝子模塊
├── composables/
│   ├── useRepairSubmit.ts    ← 表單、富文本編輯器、圖片上傳、AI 潤色
│   └── useRepairTickets.ts   ← 工單列表分頁、狀態篩選、詳情載入
└── components/
    ├── RepairPolishDialog.vue ← AI 整理彈窗（純 UI）
    └── RepairDetailDialog.vue ← 工單詳情彈窗（純 UI）
```

### 提交表單

- 標題：純文字，最多 100 字
- 問題描述：Quill 富文本編輯器，支持直接貼圖（圖片自動上傳到 OSS 並插入）
- 工具欄隱藏（只用貼圖功能，不需要格式工具欄），字數限制 2000 字

### 圖片上傳流程

```
用戶 Ctrl+V 貼圖
    ↓
攔截 paste 事件（阻止 Quill 默認的 base64 嵌入行為）
    ↓
調用 repairApi.uploadFile(file) → OSS 存儲
    ↓
拿到 URL → quillInstance.insertEmbed(range.index, 'image', url)
    ↓
圖片顯示在編輯器中
    ↓
每次 text-change 掃描 <img> 標籤，同步 uploadedAttachments 數組
（提交時一起傳給後端）
```

### AI 潤色功能

「問題描述」旁有「使用 AI 整理」按鈕：

1. 取出描述的純文字（去掉 HTML 標籤），發送給 Dify SSE 接口
2. 結果以流式方式顯示在彈窗中（打字機效果）
3. 用戶可手動微調結果文字，點「使用此版本」後回填到編輯器
4. 如果原描述中有圖片，圖片會自動附在回填內容的末尾（不丟失）
5. 每份描述最多整理 5 次（避免濫用）

SSE 使用原生 `fetch` + `ReadableStream` 實現，不用 axios（axios 不支持 ReadableStream），支持 `AbortController` 中途取消。

### 工單狀態

| 狀態碼 | 含義 | Tag 顏色 |
|--------|------|----------|
| 1 | 已提交 | 橙色（warning） |
| 2 | 已分配 | 藍色（primary） |
| 3 | 已關閉 | 灰色（info） |

### 狀態篩選的「全部」哨兵值

「我的工單」Tab 頂部的狀態篩選使用 `el-radio-button`，「全部」選項使用 `0` 作為哨兵值
（避免 `undefined` 觸發 Element Plus 的「label act as value is about to be deprecated」告警）。
`useRepairTickets.loadTickets()` 在呼叫 `repairApi.list` 前會將 `0` 映射回 `undefined`，
對後端仍為「不傳 status」的語義。

### 詳情彈窗（RepairDetailDialog）

彈窗純 UI 展示，所有資料由父層（ITRepairView + `useRepairTickets.currentDetail`）傳入，
props 僅 3 個：`modelValue` / `loading` / `detail`（`RepairDetail | null`）。

彈窗結構：
- 提問信息（工單號、標題、提交人、提交時間）以 `el-descriptions` 展示
- 問題描述用 `v-html` 渲染後端富文本
- IT 匯報回覆同樣富文本渲染；`resultContent` 為 `null` 時顯示占位「IT 尚未提交匯報回覆」
- 匯報附件以**檔名列表**呈現（`<a target="_blank">`），點擊由瀏覽器新分頁開啟／下載，
  不再使用圖片縮圖預覽（附件可能是任意類型，不一定是圖片）

### API 接口

| 方法 | 路徑 | 說明 |
|------|------|------|
| `repairApi.create(data)` | `POST /api/repair/create` | 提交新工單 |
| `repairApi.list(params)` | `GET /api/repair/list` | 查詢工單列表（分頁 + 狀態篩選） |
| `repairApi.detail(id)` | `GET /api/repair/user-report/{id}` | 查詢用戶端工單匯報詳情 |
| `repairApi.uploadFile(file)` | `POST /api/repair/upload` | 上傳附件到 OSS，返回 `{ fileUrl }` |

`repairApi.detail()` 返回 `RepairDetail`，對應後端 `RepairUserReportResponse`：

```typescript
interface RepairDetail {
  id: number
  requestNo: string
  title: string
  description: string                // 富文本 HTML
  userName: string                   // 提交人工號
  createTime: string
  resultContent: string | null       // 用戶可見匯報回覆（IsUserSee=1），未匯報為 null
  resultTime: string | null          // 匯報時間；未匯報為 null
  attachments: RepairResultAttachment[]  // 匯報附件列表
}
```

後端只返回 `IsUserSee=1` 的那筆匯報；內部匯報（`IsUserSee=0`）不會傳到用戶端。

API 基礎地址從 `VITE_REPAIR_API_URL` 環境變量讀取。

## 5. 系統托盤

托盤圖標始終存在（即使浮球也隱藏），是最後的訪問入口。

右鍵托盤圖標彈出菜單，包含：打開主窗口、統一平台、AI 快捷功能、退出應用。

其中功能子菜單項由 `config.sidebar.items` 動態生成，與側邊欄菜單保持一致。

Windows 使用 `resources/icons/tray-icon.png` 作為托盤圖標；`tray-iconTemplate.png` 為跨平台保留，本專案以 Windows
為主，無需特別關注亮/暗色適配。
