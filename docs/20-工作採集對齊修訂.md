# 工作採集對齊修訂(承接 doc 19)

> **觸發**:你在 doc 19 之上提了 4 點補充:
>   1. 配置每天 8 點(工時起點)前同步一次,不是 polling
>   2. 採集結果不是即時上報,而是 17 點工時結束後**批次上傳當天全部**
>   3. 管理頁面要支援**單人 / 多人**配置 + 查詢
>   4. DB 是 **SQL Server 2008**,單秒 20 req/s 即可
>
> 這幾條把 doc 19 的 Phase B/C「異步隊列 + Worker pool + Redis + SignalR」**整個砍掉**,因為峰值需求降了 25 倍。本文就是修訂後的最終定案。

## 0. TL;DR — 跟 doc 19 的關鍵差異

| 維度           | doc 19 原案                            | 你的修訂                         | 結論                   |
|--------------|--------------------------------------|------------------------------|----------------------|
| **配置同步時機**   | 啟動 + 10 分鐘輪詢                         | 每天 8 點前拉一次                   | ✅ 採用,簡單省             |
| **採集上報時機**   | 每次 tick 即時上報                         | 17 點工時結束後批次上傳當天              | ✅ 採用,削峰一刀切到底         |
| **AI 分析位置**  | server 異步隊列                          | 工時內 server 同步(不落庫)+ 批次上傳時只寫庫 | ✅ 採用                 |
| **後端隊列**     | Phase 1 `Channel<T>` → Phase 2 Redis | **完全不需要**                    | ❌ 砍                  |
| **結果回推**     | SignalR / Polling                    | desktop 自己用 server 同步返回      | ❌ 砍                  |
| **資料庫**      | Postgres(Phase 3)                    | SQL Server 2008              | ✅ 採用,schema 全部重設計    |
| **並發承壓目標**   | 500 同秒                               | 1 秒 20 req                   | 大幅降低,單實例輕鬆 hold      |
| **管理操作粒度**   | 單人                                   | **批次多人**                     | ✅ 採用,API 帶 userId 陣列 |
| **管理員看當天數據** | 即時                                   | T-1(昨天及之前)                   | 簡化代價,你接受             |

## 1. 修訂後的整體架構

```
─────────── 每天 08:00 前 ───────────
Desktop 啟動 / 8:00 前喚醒:
  GET /api/work/my-config        ──▶ 拉一次配置
  GET /api/work/pending-sync     ──▶ 檢查昨天有沒沒傳完的本地紀錄,有就先補傳

─────────── 工時 08:00-17:00 ─────────
Desktop tick(每 N 分鐘,N = config.intervalMinutes)
  ├─ idle 判斷:本地直接寫 SQLite,**不打 server**
  └─ 非 idle:POST /api/work/analyze(jpeg + 元數據)
              server 同步調 Qwen-VL → 返回 {category, ...}
              **server 不落庫**,只透傳結果
              Desktop 拿到結果 → 寫本地 SQLite

─────────── 17:00 工時結束 ───────────
Desktop 偵測到工時剛結束(tick 內判斷,而不是真的設 17:00 鬧鐘):
  ├─ 查本地 SQLite 拿到當天所有 unsynced records
  ├─ POST /api/work/sync-daily(records 陣列)
  │  server 批次 INSERT,return 成功 / 失敗的 idList
  └─ Desktop 把成功的 records 標記 synced=true

─────────── 管理員任何時候 ──────────
Admin Web:
  GET /api/admin/work/users?…             — 用戶清單
  GET /api/admin/work/records?userIds=…   — 查多人紀錄
  PATCH /api/admin/work/configs           — 批次改配置 { userIds, patch }
```

**為何不需要隊列**:

- 工時內 AI 同步調用,單秒峰值由 desktop 分散 tick 控制(jitter)
- 17:00 後批次上傳,500 人在 30min 視窗內隨機上傳,平均 0.28 req/s,峰值若 jitter 不足也最多 20 req/s
- SQL Server 2008 + ASP.NET Core 單實例輕鬆吃下

## 2. 對 doc 19 的衝突點明細

### 衝突 A:Server 是否落庫工時內的分析結果

- **doc 19**:落庫(於是要 user_id + jobId)
- **修訂**:**不落庫**,server 只當 LLM 代理
- **後果**:
    - `POST /api/work/analyze` 保持 stateless(跟現在一樣)
    - 落庫責任全部丟給 `POST /api/work/sync-daily`,**一天一次性 INSERT**
    - server 不需要 `work_analyze_queue` 表
    - 不需要 `job_id` UNIQUE 對齊(批次上傳直接帶 desktop 本地 id 做去重)

### 衝突 B:Desktop 在 16:55 關機怎麼辦

- **doc 19** 沒處理(因為即時上報)
- **修訂下的問題**:當天紀錄沒傳完
- **對策**:
    - desktop 啟動時**先檢查本地 SQLite 有沒有 synced=false 的 records**(無論哪一天的),統統補傳
    - 也就是說 sync 不只在 17:00 觸發,**任何時候啟動發現有未傳就傳**
    - 順便處理跨日 / 重啟 / 網路抖

### 衝突 C:管理員看到的數據新鮮度

- **doc 19**:即時(因為每筆都實時落庫)
- **修訂**:**T-1**(今天的數據要等到使用者今天 17:00 後才有)
- **對策**:
    - 管理頁面明確標註「數據截至昨日 23:59」
    - 若管理員想看「今天到目前」的數據,**沒有辦法**(除非引入即時上報,但這違背你的核心訴求)
    - 接受這個取捨

### 衝突 D:管理員改了 interval,使用者多久能看到

- **doc 19**:10 分鐘內(輪詢)
- **修訂**:**最遲第二天 08:00**(下次拉 config 時)
- **如果想當天生效**:管理員可在使用者下次 8:00 之前改;或加一個 manual `desktopApp:reload-config` IPC 通知(後續考慮)
- **接受方案**:T+1 對配置變更夠用,工作採集不是緊急場景

### 衝突 E:Idle 紀錄要不要也上傳

- **doc 19** §4.1:idle 走 `/sync-idle` 即時背景傳
- **修訂**:跟其它 records 一起,17:00 批次傳
- 簡化了:**只有一個批次上傳路徑**,idle 不另外特殊處理

### 衝突 F:資料庫差異

- **doc 19**:Postgres(用 JSONB / SERIAL / UPSERT)
- **修訂**:SQL Server 2008
- **影響**:
    - `BIGSERIAL` → `BIGINT IDENTITY(1,1)`
    - `JSONB` → 不存 JSON,改用獨立欄位 / 多行記錄
    - `INSERT ... ON CONFLICT DO UPDATE` → `MERGE` 語法(SQL 2008 支援)
    - 列名避開保留字(`user`、`group`、`order` 等)
    - 全部用 `NVARCHAR` 而非 `VARCHAR`(中文)
    - 索引最大 900 bytes / 16 columns(SQL 2008 上限)
    - 沒有 `CREATE INDEX IF NOT EXISTS`,migration 要手動判斷
    - 沒有 OPENJSON(2016+ 才有)

## 3. SQL Server 2008 表 / 索引設計

### 3.1 命名約定

- 表名:`Work_*`(底線 + 帕斯卡 / 雙連線)— 跟 TMBOM 既有風格對齊(假設 PascalCase)
- 欄位名:PascalCase(SQL Server 慣例)
- 主鍵:`Id` 或表名_Id
- 時間戳:`BIGINT`(Unix ms,跟 desktop SQLite 一致,免去時區轉換)
- 字串:`NVARCHAR(N)`,明確上限;`description` 等大欄位用 `NVARCHAR(MAX)`

### 3.2 `Work_Records` — 採集紀錄主表

```sql
-- ════════════════════════════════════════════════════════════════
-- 採集紀錄(每日批次從 desktop 上傳到此)
-- 預估容量:500 人 × 100 條/天 × 250 工作日/年 ≈ 12.5M 行/年
-- SQL Server 2008 標準版單表 hold 多年,2-3 年後再評估分區
-- ════════════════════════════════════════════════════════════════
CREATE TABLE [dbo].[Work_Records]
(
    [
    Id]
    BIGINT
    IDENTITY
(
    1,
    1
) NOT NULL,
    -- 使用者識別:跟 TMBOM 既有 user 系統的 EmployeeNo / AD account 對齊
    [UserId] NVARCHAR
(
    64
) NOT NULL,
    -- desktop 本地 SQLite 的 id(用於批次上傳去重;同一 desktop 上傳兩次同樣的紀錄不會重複)
    [LocalId] BIGINT NOT NULL,
    -- 採集時間,Unix ms,跟 desktop SQLite 一致
    [CapturedAt] BIGINT NOT NULL,
    [CapturedAtUtc] DATETIME2
(
    3
) NOT NULL, -- 冗餘存一份 DATETIME 給 SQL 查詢方便用
    [ActiveApp] NVARCHAR
(
    256
) NULL,
    [ActiveWindowTitle] NVARCHAR
(
    512
) NULL,
    -- 工作分類:coding / documenting / browsing / communicating / meeting / designing / idle / other
    [Category] NVARCHAR
(
    20
) NOT NULL,
    [Description] NVARCHAR
(
    MAX
) NOT NULL,
    [Confidence] REAL NULL,
    -- dHash 16 hex(8 byte 雜湊的十六進位字串)
    [ScreenshotHash] NVARCHAR
(
    16
) NULL,
    [Reason] NVARCHAR
(
    MAX
) NULL,
    -- server 接收時間(處理延遲分析用)
    [SyncedAt] BIGINT NOT NULL,
    CONSTRAINT [PK_Work_Records] PRIMARY KEY CLUSTERED
(
[
    Id]
    ASC
)
    );
GO

-- 主查詢索引:單人 + 時間區間(管理頁面詳情頁、desktop backfill 都走這條)
-- INCLUDE 把常用查詢欄位內聯,避免回表
CREATE
NONCLUSTERED INDEX [IX_Work_Records_User_CapturedAt]
    ON [dbo].[Work_Records] ([UserId] ASC, [CapturedAt] DESC)
    INCLUDE ([Category], [ActiveApp], [ActiveWindowTitle], [Confidence]);
GO

-- 按類別過濾(管理頁面「找誰在開會」之類)
CREATE
NONCLUSTERED INDEX [IX_Work_Records_Category_CapturedAt]
    ON [dbo].[Work_Records] ([Category] ASC, [CapturedAt] DESC)
    INCLUDE ([UserId]);
GO

-- 上傳冪等性保證:同一個 desktop 對同一條本地紀錄只能落一次
-- 為什麼用 UNIQUE 而不在程式碼判斷:SQL Server 的 UNIQUE 是 atomic,
-- 比 SELECT-then-INSERT 抗併發
CREATE UNIQUE
NONCLUSTERED INDEX [UX_Work_Records_User_LocalId]
    ON [dbo].[Work_Records] ([UserId] ASC, [LocalId] ASC);
GO
```

**索引取捨**:

- `IX_Work_Records_User_CapturedAt` 是 95% 查詢的主索引(管理員看單人歷史、desktop backfill、聚合統計)
- `IX_Work_Records_Category_CapturedAt` 給「找特定類別」場景;不常用但加上後查詢 < 50ms
- `UX_Work_Records_User_LocalId` 是冪等性保證,**不可省**(否則 desktop 網路抖重傳會重複)
- 不加 `CreatedAt` / `SyncedAt` 索引:這兩個欄位 admin 不會直接搜索,只用來統計

### 3.3 `Work_User_Configs` — 使用者配置

```sql

-- 每個使用者的採集配置(一行一人)
-- ════════════════════════════════════════════════════════════════
CREATE TABLE [dbo].[Work_User_Configs]

    [
    UserId]

(

    [WorkEndHour] INT NOT NULL DEFAULT 17,

    -- 改成 BIGINT updatedAt 同效;用版本號更顯式

(
    3

(

(
[
    UserId]

-- 管理頁面按 UpdatedAt 排序「最近改過的」用

```

**設計取捨**:

- 主鍵直接 `UserId` 而非自增 Id:配置永遠一對一,沒必要走代理鍵

- `UpdatedBy` 區分「自己改的」vs「管理員改的」,審計用(目前 desktop 不再讓使用者改 interval,但日後若放開仍保留欄位)

### 3.6 預設資料填充(seed)

```sql
-- 新使用者首次採集前,server 端若沒有配置先插入預設值
-- 這段不在 migration 裡跑,而是業務碼裡:
--   `Work_User_Configs_GetOrCreate(@UserId)`
```

業務碼示例(.NET / Dapper 風格):

```csharp
public async Task<WorkUserConfig> GetOrCreateConfig(string userId) {
    var existing = await db.QueryFirstOrDefaultAsync<WorkUserConfig>(
        "SELECT * FROM Work_User_Configs WHERE UserId = @UserId",
        new { UserId = userId });
    if (existing != null) return existing;

    var now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
    await db.ExecuteAsync(@"
        INSERT INTO Work_User_Configs
            (UserId, Enabled, IntervalMinutes, WorkStartHour, WorkEndHour,
             Version, UpdatedAt, UpdatedAtUtc, UpdatedBy)
        VALUES
            (@UserId, 1, 5, 8, 17, 1, @Now, SYSUTCDATETIME(), 'system')
    ", new { UserId = userId, Now = now });

    return await GetOrCreateConfig(userId);  // 遞迴一次即可命中
}
```

## 4. API 契約修訂

### 4.1 桌面端用

| Method                               | Path                                        | 跟 doc 19 差異                                     |
|--------------------------------------|---------------------------------------------|-------------------------------------------------|
| `GET`                                | `/api/work/my-config`                       | 一樣,8 點前拉                                        |
| `POST`                               | `/api/work/analyze`                         | **同步**返回 AI 結果(跟現狀一致),**不再 202**,**server 不落庫** |
| `POST`                               | `/api/work/sync-daily`                      | **新增**,批次上傳 records 陣列                          |
| ~~`GET` `/api/work/recent-results`~~ | ❌ 砍                                         | 不需要 polling                                     |
| ~~`POST` `/api/work/sync-idle`~~     | ❌ 砍                                         | idle 走批次,不另外路徑                                  |
| `GET`                                | `/api/work/my-records?since&until&category` | 保留,給跨機 / 重灌 backfill 用                          |

#### 4.1.1 `POST /api/work/sync-daily` 契約

```
Request:
  POST /api/work/sync-daily
  Body:
  {
    "records": [
      {
        "localId": 12345,                    // desktop 本地 SQLite 的 id
        "capturedAt": 1735000000000,
        "activeApp": "Code.exe",
        "activeWindowTitle": "src/main.ts - VSCode",
        "category": "coding",
        "description": "...",
        "confidence": 0.92,
        "screenshotHash": "a3f2c1d4...",
        "reason": "..."
      },
      // ... 一次最多 200 條(server 限制,避免單請求過大)
    ]
  }

Response 200:
  {
    "inserted": 198,
    "duplicates": 2,                          // 已存在的 localId,跳過
    "failed": [],                             // 失敗的 localId + 錯誤訊息
    "syncedAt": 1735086400000
  }
```

**冪等性**:server 端用 `MERGE` 或 `INSERT WHERE NOT EXISTS`,同一個 `(UserId, LocalId)` 不會重複落庫(對應
`UX_Work_Records_User_LocalId`)。

#### 4.1.2 SQL Server 2008 的批次 INSERT 寫法

```sql
-- 推薦:用 MERGE(SQL 2008 起支援),冪等 + 高效
MERGE INTO [dbo].[Work_Records] AS T
    USING (VALUES
    (@UserId, @LocalId, @CapturedAt,...),
    (@UserId, @LocalId, @CapturedAt,...)
    -- ... 最多 200 行
    ) AS S ([UserId], [LocalId], [CapturedAt],...)
    ON T.[UserId] = S.[UserId]
    AND T.[LocalId] = S.[LocalId]
    WHEN NOT MATCHED THEN
    INSERT (UserId, LocalId, CapturedAt, CapturedAtUtc,...)
    VALUES (S.UserId, S.LocalId, S.CapturedAt,...);
```

或用 TVP(Table-Valued Parameter):

```sql
CREATE TYPE [dbo].[Work_Record_TVP] AS TABLE (
    [UserId] NVARCHAR(64),
    [LocalId] BIGINT,
    [CapturedAt] BIGINT,
    [ActiveApp] NVARCHAR(256),
    [ActiveWindowTitle] NVARCHAR(512),
    [Category] NVARCHAR(20),
    [Description] NVARCHAR(MAX),
    [Confidence] REAL,
    [ScreenshotHash] NVARCHAR(16),
    [Reason] NVARCHAR(MAX)
    );

-- 在 .NET 端傳 DataTable 走 TVP,server 內一次 MERGE,200 行 < 50ms
```

### 4.2 管理頁面用(批次版)

| Method  | Path                                                           | 說明                                                                 |
|---------|----------------------------------------------------------------|--------------------------------------------------------------------|

#### 4.2.1 批次配置 PATCH 示例

```
PATCH /api/admin/work/configs
{
  "userIds": ["jane.doe", "john.smith", "alice.wu"],
  "patch": {
    "intervalMinutes": 10,
    "workStartHour": 9
  },
  "reason": "10月起調整全員工時"
}

Response:
{
  "updated": 3,
  "results": [
    { "userId": "jane.doe", "version": 5, "updatedAt": 1735100000000 },
    { "userId": "john.smith", "version": 3, "updatedAt": 1735100000000 },
    { "userId": "alice.wu", "version": 1, "updatedAt": 1735100000000 }
  ]
}
```

server 處理:

1. 開 SQL transaction
2. 逐 userId 跑 `UPSERT`(MERGE)
3. 每筆寫一條 `Work_Config_Audit`
4. commit
5. 若有任一失敗 → rollback,return 500 + 哪幾個失敗

#### 4.2.2 批次查紀錄

```
POST /api/admin/work/records:batch-query
{
  "userIds": ["jane.doe", "john.smith"],
  "since": 1735000000000,
  "until": 1735086400000,
  "category": "coding",      // 可選
  "page": 1,
  "size": 100
}

Response:
{
  "total": 540,
  "page": 1,
  "size": 100,
  "items": [ /* 100 條 records */ ]
}
```

SQL(SQL Server 2008 沒 OFFSET FETCH,要用 ROW_NUMBER):

```sql
WITH Filtered AS (SELECT r.*,
                         ROW_NUMBER() OVER (ORDER BY r.[CapturedAt] DESC) AS RowNum
                  FROM [dbo].[Work_Records] r
WHERE r.[UserId] IN @userIds -- TVP 或 STRING_SPLIT(2016+ 才有,2008 要 IN 清單)
  AND r.[CapturedAt] >= @since
  AND r.[CapturedAt]
    < @until
  AND (@category IS NULL
   OR r.[Category] = @category)
    )
SELECT *
FROM Filtered
WHERE RowNum BETWEEN ((@page - 1) * @size + 1) AND (@page * @size);
```

SQL Server 2008 沒 `STRING_SPLIT`,`userIds` 陣列要走 TVP 或拼 IN clause(注意 SQL injection,用參數化)。

## 12. 時區處理(全鏈路用北京時間)

### 12.1 為何要鎖時區

工時 `8-17` 必須有明確基準。當前桌面端用 `new Date().getHours()`,**取的是 OS 本機時區**。問題:

- 出差 / 海外辦公(VPN 接內網)的人,本機時區可能是 UTC+0 / UTC-8
- 用 OS 時區會造成「採集在凌晨 3 點啟動」這種混亂

**決策**:全鏈路鎖死 `Asia/Shanghai`(北京時間,UTC+8,**沒有 DST**)。

### 12.2 各層做法

| 層                          | 儲存                                                                          | 顯示 / 判斷                                                               |
|----------------------------|-----------------------------------------------------------------------------|-----------------------------------------------------------------------|
| **Desktop SQLite**         | `capturedAt` 仍是 Unix ms(UTC,跟現狀一致)                                          | UI 顯示時 `formatInTimeZone(ms, 'Asia/Shanghai')`                        |
| **Desktop Scheduler 工時判斷** | —                                                                           | **改寫 `isInWorkHours()`,顯式用北京時間取 hour**(不能用 `new Date().getHours()`)   |
| **Server SQL**             | `CapturedAt BIGINT`(UTC ms)+ `CapturedAtUtc DATETIME2(3)`(冗餘存 UTC datetime) | 查詢按北京日期切割時,在 SQL 內 `DATEADD(hour, 8, CapturedAtUtc)` 再 `CAST AS DATE` |
| **Admin Web 顯示**           | —                                                                           | 日期 picker / 圖表 X 軸全部 `formatInTimeZone(..., 'Asia/Shanghai')`         |
| **配置同步觸發**                 | `lastConfigSyncDay` 存 `'2026-01-15'`(北京日期)                                  | 每天比對「今天的北京日期」                                                         |

### 12.3 Desktop scheduler 改寫示例

```ts
// electron/main/work-collector.ts
// 北京時區固定 +8,沒有 DST,直接 +8h 後取 UTC hour
function getBeijingHour(): number {
    const now = new Date()
    // UTC 小時 + 8,對 24 取模
    return (now.getUTCHours() + 8) % 24
}

function getBeijingDate(): string {
    const now = new Date()
    const beijing = new Date(now.getTime() + 8 * 3600_000)
    // 用 UTC* 系列函式避開本機時區影響
    const y = beijing.getUTCFullYear()
    const m = String(beijing.getUTCMonth() + 1).padStart(2, '0')
    const d = String(beijing.getUTCDate()).padStart(2, '0')
    return `${y}-${m}-${d}`
}

// 工時判斷
private
isInWorkHours()
:
boolean
{
    const c = this.cfg.getConfig().workCollect
    const start = c?.workStartHour ?? 8
    const end = c?.workEndHour ?? 17
    const hour = getBeijingHour()         // ★ 改用北京時間,不再用 new Date().getHours()
    return hour >= start && hour < end
}

// 配置同步觸發判斷
private async
maybeSyncConfig()
:
Promise < void > {
    const today = getBeijingDate()
    if(this.lastConfigSyncDay === today
)
return
await this.fetchConfigFromServer()
this.lastConfigSyncDay = today
}
```

**注意**:

- 不引入 `date-fns-tz` / `moment-timezone`,直接用 `+8h` 偏移計算就夠(北京不切 DST)
- 寫進 DB 的 `capturedAt` 仍是 UTC ms,**不要存「北京時間 ms」**(那會跟其它層脫鉤)
- 顯示層全部從 UTC ms 計算成北京時間

### 12.4 Server SQL 查詢示例

「查 jane.doe 今天(北京日期)的紀錄」:

```sql
-- 今天的北京日期區間轉回 UTC ms
DECLARE
@BeijingDay DATE = CAST(DATEADD(hour, 8, SYSUTCDATETIME()) AS DATE);  -- 北京今天
DECLARE
@StartUtcMs BIGINT = DATEDIFF_BIG(ms, '1970-01-01', DATEADD(hour, -8, CAST(@BeijingDay AS DATETIME2)));
DECLARE
@EndUtcMs   BIGINT = @StartUtcMs + 86400000;  -- +24h

SELECT *
FROM [dbo].[Work_Records]
WHERE [UserId] = @UserId
  AND [CapturedAt] >= @StartUtcMs
  AND [CapturedAt]
    < @EndUtcMs
ORDER BY [CapturedAt] DESC;
```

**注意**:`DATEDIFF_BIG` 是 SQL Server 2016+ 的函式。SQL 2008 沒有,要用 `DATEDIFF(ms, ...)` 並注意溢位(秒級用 INT
夠,毫秒級會爆)。實際做法是在 .NET 端把北京日期換算成 UTC ms 再傳進 SQL:

```csharp
// .NET 端(C#)
var beijing = TimeZoneInfo.FindSystemTimeZoneById("China Standard Time");
var todayBeijing = TimeZoneInfo.ConvertTimeFromUtc(DateTime.UtcNow, beijing).Date;
var startUtc = TimeZoneInfo.ConvertTimeToUtc(todayBeijing, beijing);
var startUtcMs = new DateTimeOffset(startUtc, TimeSpan.Zero).ToUnixTimeMilliseconds();
var endUtcMs = startUtcMs + 86400000L;

// 再傳 startUtcMs / endUtcMs 給 SQL
```

這樣 SQL 2008 也兼容,不用依賴新函式。

### 12.5 邊界 case

| 場景                       | 處理                                                                                        |
|--------------------------|-------------------------------------------------------------------------------------------|
| 使用者在海外 VPN(本機時區 UTC-8)登入 | desktop 仍用北京時間 8-17 採集;UI 顯示也是北京時間;使用者看到時間軸是「北京日期」(可能跟他本地時鐘對不上,但合規 / 統計口徑統一)              |
| Server 部署在海外節點           | `SYSUTCDATETIME()` 永遠正確;`.NET ConvertTimeFromUtc("China Standard Time")` 不受 server 所在時區影響 |
| 跨日採集(凌晨 1 點還在加班)         | 工時 `8-17` 內才採集,凌晨自然 skip                                                                  |
| 跨月 / 跨年的當天統計             | 都按北京日期切割,2026-01-01 00:00 北京 = 2025-12-31 16:00 UTC,SQL 端用 `startUtcMs` / `endUtcMs` 切就對  |
