## 1. 現狀盤點(我們其實已經在 L1.5)

建議書說「目前停留在 Activity 層」—— 部分不準確。現有系統其實已經跨進半個語意層:

| 能力                                                                 | 現狀 | 對應建議書        |
|--------------------------------------------------------------------|----|--------------|
| 螢幕截圖 + dHash + idle 偵測                                             | ✅  | L1 Activity  |
| 可見視窗 / 前台 app / 時間軸                                                | ✅  | L1 Activity  |
| **Qwen-VL 回 `category` + `description` + `reason` + `confidence`** | ✅  | **已是 L2 雛形** |
| 集中化落庫(`Work_Records`)+ 管理頁聚合(類別佔比 / 每日趨勢 / 應用排行)                   | ✅  | L1→L2 統計     |

所以真正缺的不是「從零做語意」,而是:

1. **分類太粗**:`coding / documenting / browsing / communicating / meeting / designing / idle / other` ——
   是「動作類型」不是「工作主題」。建議書要的是「GM Governance / PLM / EDI / 風險治理」這種**業務主題**。
2. **沒有跨筆聚合的認知指標**:deep work 時長、context switching 次數、fragmentation —— 資料都在 `work_records`,只是沒算。
3. **沒有產出層 / 組織層 / 知識層**。

## 2. Phase 1 — Semantic Layer(語意化,現在就做)

> 投入最小、感知最強。核心是「改 AI 怎麼分類 + 加語意欄位」,不動採集鏈路。

### 2.1 分類從「動作」升級為「主題」

現有 `category` 是動作維度(coding/meeting…),**保留**(它有用,圖表靠它)。**新增**一個正交維度:

- `workTopic`(工作主題,自由文字 or 受控詞表):AI 從畫面推斷的業務主題,例「群創 BOM 維護」「IT 報修處理」「採購對帳」。
- `topicConfidence`:主題信心。

**怎麼做**:擴 Qwen-VL 的 `response_format` json_schema,在現有 4 欄位外加 `workTopic` / `topicConfidence`。Prompt
增加「除了動作類別,推斷正在處理的**業務主題**(看 app 標題 / 文件名 / 網頁標題 / ERP 模組名)」。

**改動點**:

- `WorkCollectAnalyzeResponse` 加 2 欄位
- `WorkCollectService.BuildQwenRequestJson` 的 schema + systemPrompt 擴充
- `work_records`(本地 SQLite)+ `Work_Records`(server)加 `workTopic` / `topicConfidence` 欄位(走 migration)
- `WorkCollectSyncRecordItem` / sync-daily 帶上

**工時**:後端 1d + desktop 0.5d + migration 0.5d ≈ **2d**。

### 2.2 受控主題詞表(可選,提升一致性)

自由文字主題會發散(「報修」vs「IT 報修」vs「維修單」)。若要聚合準確,給 AI 一個**部門級受控詞表**(從 TMBOM 既有模組 / 部門推):

- server 維護一張 `Work_Topic_Dict`(部門 → 允許主題清單)
- prompt 把該使用者部門的詞表塞進去,要 AI **優先從詞表選**,選不到才自由填

**取捨**:詞表要人維護,初期可跳過,先用自由文字 + 後續做一次性聚類歸併。**建議 Phase 1 先自由文字,Phase 2 再上詞表**。

### 2.3 「閒置」改名(建議書點名,順手做)

建議書要求避免「閒置」字樣 → 改 `No Observable Interaction`。

**怎麼做**:`category='idle'` 的**底層枚舉不動**(代碼 / 索引依賴它),只改**前端顯示文案**:

- desktop / admin 的 `categoryLabel`:`idle → '無可觀測互動'`(或保留中性「待機」)
- 不要動 DB category 值,否則所有歷史資料 + 索引要遷移

**工時**:前端文案 **0.2d**。

## 3. Phase 2 — Cognitive Layer(認知分析,資料已就緒)

> 不需要新採集,全靠 `work_records` 既有資料聚合。後端加聚合 API,前端加圖表。

### 3.1 Deep Work / Context Switching 指標

都能從現有 `(capturedAt, category, activeApp, workTopic)` 序列算:

| 指標                       | 算法                                                | 落點   |
|--------------------------|---------------------------------------------------|------|
| **Deep Work 時長**         | 連續 N 個 tick 同一 `workTopic`(或同類別)且非 idle → 算一段深度工作 | 後端聚合 |
| **Context Switching 次數** | 相鄰 tick 的 `workTopic` / `activeApp` 變化次數          | 後端聚合 |
| **Fragmentation Score**  | 一天內工作段數 / 總時長(段越多越碎)                              | 後端聚合 |

**改動點**:擴 `GetOverviewAsync` / 加 `GetCognitiveStatsAsync`(server),前端在 AnalyticsPanel / OverviewPanel 加卡片 +
圖。資料源就是現有表,**零採集改動**。

**工時**:後端 2d + 前端 2d ≈ **4d**。

### 3.2 Cognitive Timeline(按主題而非 APP 的時間軸)

建議書要「按工作主題顯示時間軸」。現有 TimelineList 是按筆顯示,改成**按 `workTopic` 分段聚合**的泳道圖(
同主題連續段合併成一條)。

**前提**:Phase 1 的 `workTopic` 已落地。**改動**:前端時間軸元件改聚合邏輯。**工時**:前端 1.5d。

### 3.3 Work Narrative(每日 AI 工作摘要)

建議書要「每日自動輸出工作故事」。**這是現有 Qwen-VL 的自然延伸**:

- 每天 17:00 sync 後,後端用當天 `work_records`(已含 description / workTopic)餵給 LLM,生成一段「今日工作敘事」
- 不需要再看截圖(用已存的文字結果),成本低
- 落 `Work_Daily_Narrative` 表,管理頁 / 個人頁顯示

**改動點**:server 加一個日結 job(可手動觸發先做)+ 一張表 + LLM 文字摘要調用。**工時**:後端 2-3d。

> ⚠️ 隱私 / 合規警示:Work Narrative 是「描述某人一天做了什麼」的文字,比原始紀錄更敏感。**必須**對齊 docs/20 §10 的權限決策(
> 誰能看)。建議只給本人 + 直屬主管,且管理頁明確標「AI 生成,僅供參考」。

## 4. Phase 3 — Improvement / Knowledge(大工程,先別衝)

> 建議書最吸引人的部分(組織 friction、知識圖譜、Ontology),也是**最容易變成堆名詞**的部分。誠實說:
> 這些要么需要跨人跨系統的資料規模,要么需要長期投入的本體建模,**沒到規模 / 沒對齊組織訴求前不要做**。

### 4.1 Organizational Friction Analytics(組織摩擦分析)

建議書要「分析跨系統來回切換、重複解釋、知識缺失」。

**現實**:

- 「跨系統切換」可從 context switching 資料推(Phase 2 已有基礎)
- 「重複解釋 / 知識缺失」需要**內容級語意理解 + 跨人關聯**,現有逐人逐筆資料推不出來,且結論主觀
- 真要做得先回答:**friction 找出來給誰看、要驅動什麼行動**?沒有 owner 的分析就是擺設

**建議**:Phase 2 的 context switching 做扎實後,**只做「跨系統切換熱點」這一個可量化子集**,其餘(重複解釋 / 知識缺失)**暫不做
**,標為「待組織訴求明確後評估」。

### 4.2 Knowledge Cluster / Ontology(知識圖譜 / 企業本體)

建議書要「自動聚類工作主題形成知識圖譜 + 企業語意模型」。

**現實**:這是 embedding + 聚類 + 人工本體建模的組合,**工程量以月計**,且:

- 要 embedding 服務(向量化 description / workTopic)
- 要向量庫(pgvector / 專用庫)
- Ontology(企業語意模型)**無法自動生成**,需要業務專家持續維護
- 投入產出比在「採集系統」這個 scope 下極不划算

**建議**:**Phase 3+ 才評估,且要獨立立項**,不混進採集系統。先用 Phase 1 的受控主題詞表(§2.2)拿到 80% 的「主題歸類」價值,知識圖譜這種
20% 長尾留給專門的知識管理項目。

### 4.3 AI Leverage Ratio(AI 貢獻度)

建議書要「分析 AI 對工作效率的貢獻」。**現有採集看不到「使用者用了多少 AI」**(只看到畫面),除非:

- 採集能識別「前台是 AI 工具(Claude / ChatGPT / Copilot)」→ 這個**能做**(看 activeApp / 視窗標題),算「AI 工具佔用時長比例」
- 但「AI 對效率的實際貢獻」是因果推斷,**做不到也不該假裝能做**

**建議**:只做「AI 工具使用時長佔比」這個可量化指標(Phase 2 順帶),不碰「貢獻度」這種偽科學。

## 5. UX 改善(對應建議書第五節,多數低成本)

| 建議書要求                             | 落地                                           | 成本    |
|-----------------------------------|----------------------------------------------|-------|
| 首頁顯示「今日工作結構」而非只有 APP Top5         | OverviewPanel 加「主題分佈」環圖(Phase 1 workTopic 後) | 低     |
| 「今日主工作主題」區塊                       | 個人頁 + 管理頁加卡片                                 | 低     |
| 「今日深度工作時間 / Context Switching 次數」 | Phase 2 認知指標卡片                               | 中     |
| 「工作成果」區塊                          | ❌ 採集看不到「產出」(完成了什麼文件 / 流程),除非接 ERP/PLM 的產出事件  | 高,暫不做 |
| 「AI 協作占比 / AI 使用模式」               | §4.3 的 AI 工具時長佔比                             | 中     |

## 6. AI 能力升級(對應建議書第六節)

| 建議書                          | 落地路徑                                                                                        |
|------------------------------|---------------------------------------------------------------------------------------------|
| Semantic Understanding(語義理解) | = Phase 1 的 workTopic 推斷,**prompt 工程就是第一步**,不需要先建大模型                                        |
| Enterprise Ontology(企業語意模型)  | §4.2,大工程,獨立立項                                                                               |
| 推論工作意圖而非辨識畫面                 | Phase 1 prompt 已在做(`reason` 欄位就是意圖推斷的雛形),持續調 prompt 精進                                      |
| Agent-ready Metadata         | 把 `workTopic` / 認知指標以結構化 JSON 落庫(現在的 schema 已經是),未來 Agent 要用直接查 —— **這條其實 Phase 1/2 順手就達成** |

**重點**:建議書把「語義理解 / Agent-ready」講得很高大上,實際上**Phase 1 的結構化欄位 + 好 prompt 就覆蓋了大半**
。不要為了「Ontology」這個詞去做一個沒人維護的本體庫。

## 7. 定位與邊界(對應建議書第七節 + 合規)

建議書主張**不要定位成「員工監控系統」**,認同。落到產品上:

- ✅ 對外口徑:「個人工作回顧 + 團隊工作結構分析」,本人優先看自己
- ✅ 沿用 docs/20 §10 已定的合規邊界:使用者本機可關採集、永久保留需對齊資料政策、權限第一版只判 admin role
- ❌ **不做**:即時監控告警、未採集名單點名、跨部門排行榜(docs/20 §11 已明確劃線)
- ⚠️ Work Narrative / 認知指標越往上,越像「評價個人」,**每加一個指標都要過合規 + 員工關係這關**,不是技術問題

## 8. 分期路線圖(誠實版)

| Phase                      | 內容                                                                                    | 工時     | 前置                   |
|----------------------------|---------------------------------------------------------------------------------------|--------|----------------------|
| **Phase 1 — 語意化**          | workTopic 欄位 + prompt 升級 + idle 改名 + 首頁主題分佈                                           | ~3d    | 無,現在能做               |
| **Phase 2 — 認知分析**         | deep work / context switching / fragmentation 聚合 + 認知時間軸 + AI 工具時長佔比 + Work Narrative | ~10d   | Phase 1              |

**現實預測**:Phase 1 兩三天就能讓使用者「明顯感覺變聰明了」(從 app 名變成業務主題)。Phase 2 是價值放大段。Phase 3 的 L4/L5 *
*不要因為建議書寫了就排期** —— 等有具體部門帶著具體問題來,再針對性做一小塊,別建空中樓閣。

## 9. 不做 / 要打回的(避免 scope 失控)

- ❌ **不為了「5 層架構」一次性重構** —— 按 Phase 漸進,每層獨立交付
- ❌ **不自動生成 Enterprise Ontology** —— 本體要人維護,自動生成的是垃圾
- ❌ **不做「工作成果層」** —— 螢幕採集看不到產出,要接 ERP/PLM 是另一條線
- ❌ **不做「AI 貢獻度」因果推斷** —— 只做可量化的「AI 工具時長佔比」
- ❌ **不做即時監控 / 排行榜** —— 合規紅線(docs/20 §11)
- ⚠️ **每個認知 / 敘事指標上線前過合規** —— 越往認知層走越敏感

---

## 10. 給決策者的一句話

建議書的願景對,但**價值曲線是前重後輕**:

- **Phase 1(語意化)**:3 天,感知提升最大 —— 從「在用 Excel」變成「在做第二季預算」
- **Phase 2(認知)**:10 天,把資料變洞察
- **Phase 3(組織 / 知識)**:月計投入,**邊際收益遞減 + 合規風險遞增**,沒有明確組織 owner 就不該啟動

**先做 Phase 1,把語意層做扎實,其餘讓真實訴求來拉動,而不是讓建議書的名詞來推動。**
