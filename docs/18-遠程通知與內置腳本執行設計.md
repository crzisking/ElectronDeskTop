# 遠程通知與內置腳本執行設計

> **前提**：內網環境，每台機器 IP 固定。  
> **約束**：Windows 防火牆預設攔截入站，客戶端**不能開端口監聽**。  
> **方案**：客戶端主動連服務端（WebSocket），服務端通過已建立的連接推送任務。

---

## 1. 為什麼是 WebSocket

```
┌─ 防火牆規則 ─────────────────────────────────────────────┐
│                                                          │
│  入站（外 → 內）：預設攔截                                 │
│     ❌ 服務端 POST http://192.168.1.5:12450  → 被封       │
│     ❌ 客戶端起 HTTP Server                      → 被封   │
│                                                          │
│  出站（內 → 外）：預設放行                                 │
│     ✅ 客戶端 GET http://server/api      → 通             │
│     ✅ 客戶端 ws://server/notifications  → 通             │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

WebSocket 的連接是**客戶端主動發起**的（出站），防火牆不攔。連接建立後，服務端就可以**經由這條已建立的 TCP 通道**
向客戶端推送消息，不需要客戶端開任何監聽端口。

## 3. 客戶端 NotificationClient 設計

### 3.1 生命週期

```
應用啟動
  │
  ▼
NotificationClient.start()
  │
  ├── 1. 取本機 IP (os.networkInterfaces)
  │
  ├── 2. 連接 ws://{server}/notifications?machine=192.168.1.5
  │     (用標準 WebSocket 庫，如 ws 或 Node.js 內建)
  │
  ├── 3. 連接成功 → 發送註冊消息 { type: "register", machine: "192.168.1.5" }
  │
  ├── 4. 監聽 message 事件：
  │     收到 { type: "task", action: "...", params: {...} }
  │       → 查找內置腳本
  │       → 執行
  │       → 回傳 { type: "result", taskId: "...", ok: true/false, summary: "..." }
  │
  ├── 5. 心跳：每 30 秒發 ping，服務端回 pong
  │     若連續 3 次無 pong → 斷線重連（指數退避：1s → 2s → 4s → 最多 30s）
  │
  └── 6. 應用退出 → NotificationClient.stop()
       發送 { type: "unregister" } → 關閉連接
```

### 3.2 核心程式碼結構（~80 行）

```typescript
import WebSocket from 'ws'

class NotificationClient {
    private ws: WebSocket | null = null
    private reconnectTimer: ReturnType<typeof setTimeout> | null = null
    private pingTimer: ReturnType<typeof setInterval> | null = null
    private reconnectDelay = 1000

    start(serverUrl: string, machineIP: string, scriptRunner: ScriptRunner) {
        const url = `${serverUrl}/notifications?machine=${machineIP}`
        this.connect(url, scriptRunner)
    }

    private connect(url: string, scriptRunner: ScriptRunner) {
        this.ws = new WebSocket(url)

        this.ws.on('open', () => {
            this.reconnectDelay = 1000  // 重置退避
            // 註冊本機
            this.ws!.send(JSON.stringify({type: 'register', machine: getLocalIP()}))
            // 開始心跳
            this.pingTimer = setInterval(() => {
                this.ws?.ping()
            }, 30_000)
        })

        this.ws.on('message', async (raw) => {
            const msg = JSON.parse(raw.toString())
            if (msg.type === 'task') {
                const result = await scriptRunner.execute(msg.action, msg.params ?? {})
                // 回傳執行結果
                this.ws?.send(JSON.stringify({
                    type: 'result',
                    taskId: msg.taskId,
                    ...result,
                }))
            }
        })

        this.ws.on('close', () => {
            // 非主動關閉 → 重連
            if (this.pingTimer) {
                clearInterval(this.pingTimer)
                this.pingTimer = null
            }
            this.scheduleReconnect(url, scriptRunner)
        })

        this.ws.on('error', () => {
            // error 後會觸發 close，在 close 中處理重連
        })
    }

    private scheduleReconnect(url: string, scriptRunner: ScriptRunner) {
        this.reconnectTimer = setTimeout(() => {
            this.connect(url, scriptRunner)
            this.reconnectDelay = Math.min(this.reconnectDelay * 2, 30_000)
        }, this.reconnectDelay)
    }

    stop() {
        if (this.reconnectTimer) clearTimeout(this.reconnectTimer)
        if (this.pingTimer) clearInterval(this.pingTimer)
        if (this.ws) {
            this.ws.send(JSON.stringify({type: 'unregister'}))
            this.ws.close()
            this.ws = null
        }
    }
}
```

### 3.3 為什麼不用 socket.io

|       | `ws` 庫             | `socket.io`         |
|-------|--------------------|---------------------|
| 包大小   | ~200KB             | ~1.2MB              |
| 協議    | 標準 RFC 6455        | 自訂協議（非標準 WebSocket） |
| 服務端要求 | 任何支持 WebSocket 的後端 | 必須用 socket.io 服務端   |
| 房間/廣播 | 自己實現（很簡單）          | 內建                  |
| 自動重連  | 自己寫（~10 行）         | 內建                  |

**結論**：用 `ws`（Node.js 端）或 `isomorphic-ws`。功能只需"連接 → 收消息 → 回結果"，不需要房間、不需要廣播到多端、不需要
fallback 到 HTTP long-polling。自己寫重連邏輯就 10 行。

> 如果 TMBOM 後端是 C# / Java / Go，它們各自都有標準 WebSocket 支持，不需要裝任何額外套件。

## 4. 服務端 WebSocket 接口契約（給後端參考）

### 4.1 客戶端 → 服務端

| 消息類型         |    方向     | Payload                                                        | 說明                                |
|--------------|:---------:|----------------------------------------------------------------|-----------------------------------|
| `register`   | 客戶端 → 服務端 | `{ type: "register", machine: "192.168.1.5" }`                 | 連接建立後立即發送，服務端記錄 IP → WebSocket 映射 |
| `unregister` | 客戶端 → 服務端 | `{ type: "unregister" }`                                       | 客戶端正常退出前發送                        |
| `result`     | 客戶端 → 服務端 | `{ type: "result", taskId: "uuid", ok: true, summary: "..." }` | 任務執行結果                            |
| `pong`       | 客戶端 → 服務端 | WebSocket Ping 幀                                               | 心跳回應                              |

### 4.2 服務端 → 客戶端

| 消息類型   |    方向     | Payload                                                                  | 說明   |
|--------|:---------:|--------------------------------------------------------------------------|------|
| `task` | 服務端 → 客戶端 | `{ type: "task", taskId: "uuid", action: "clear-cache", params: {...} }` | 推送任務 |

### 4.3 服務端邏輯（偽代碼）

```
連接註冊表: Map<IP, WebSocket>

// 客戶端連接時
ws.on('connect', (conn, req) => {
  const machineIP = parseQuery(req.url).machine  // from ?machine=192.168.1.5
  // 暫存，等收到 register 消息後確認
})

// 收到 register
ws.on('message', (conn, msg) => {
  if (msg.type === 'register') {
    registry.set(msg.machine, conn)
  }
  if (msg.type === 'result') {
    // 寫入任務執行歷史表
    saveTaskResult(msg.taskId, msg.ok, msg.summary)
  }
})

// 管理端調用 dispatch
POST /api/notifications/dispatch  { target: "192.168.1.5", action: "clear-cache", ... }
  → const conn = registry.get("192.168.1.5")
  → if (conn) conn.send({ type: "task", taskId: uuid(), action, params })
  → return { ok: true, taskId }
  → if (!conn) return { ok: false, error: "目標機器離線" }
```

## 5. 內置腳本（不變）

| 腳本名              | 描述                     | 參數                                             |
|------------------|------------------------|------------------------------------------------|
| `clear-cache`    | 清除舊日誌、舊截圖、舊工作記錄        | `olderThan: number` 天數，預設 14                   |
| `restart-app`    | 重新啟動應用                 | `delayMs?: number` 預設 3000                     |
| `collect-logs`   | 打包近 N 天日誌為 zip，回傳大小和路徑 | `days: number` 預設 7                            |
| `sync-config`    | 強制從 DB 重載配置            | —                                              |
| `run-diagnostic` | 收集系統資訊                 | —                                              |
| `show-message`   | 桌面通知                   | `title: string`、`body: string`、`level: string` |

## 6. 防護措施

| 威脅        | 緩解                                           |
|-----------|----------------------------------------------|
| **偽造服務端** | 內網環境，服務端地址硬編碼在 defaults.ts，不從用戶輸入獲取          |
| **惡意任務**  | `ScriptRunner` 只執行已註冊腳本，未知 action 返回錯誤       |
| **惡意參數**  | 每個腳本內部對 params 做校驗                           |
| **連接洩漏**  | 心跳 + 重連指數退避，最大 30 秒                          |
| **服務端重啟** | 客戶端檢測到斷線後自動重連，註冊表重建                          |
| **重複執行**  | 服務端生成 `taskId`（UUID），客戶端記錄最近 N 個已執行的 ID，拒絕重複 |

## 9. 與輪詢方案的對比（總結）

|             | 輪詢                         | WebSocket（最終方案）       |
|-------------|----------------------------|-----------------------|
| **防火牆**     | ✅ 出站                       | ✅ 出站                  |
| **即時性**     | N 秒延遲                      | **即時**                |
| **服務端開銷**   | 每台每 N 秒一個 HTTP 請求          | **一條長連接**             |
| **服務端複雜度**  | 任務隊列 + pending/acked 狀態    | 連接註冊表 + dispatch      |
| **客戶端程式碼量** | setInterval + fetch（~40 行） | WebSocket + 重連（~80 行） |
| **離線處理**    | 任務堆在隊列等下次輪詢                | 任務投遞失敗，管理端知道          |
| **npm 依賴**  | 無                          | `ws`（~200KB）          |
