# 日誌 DB 實作步驟清單

> 配套設計文件:[08-本地數據庫設計.md](./08-本地數據庫設計.md)
> 本檔**只列執行步驟**,每步盡量小、可獨立 commit、有明確驗收。設計理由 / 為什麼這樣做請看 08。
>
> 共 7 階段、29 個小步驟。建議按順序執行,前階段沒過不要進下階段。

---

## 階段 A:依賴與設定檔

### A1. 安裝 runtime 相依

```bash
npm i better-sqlite3 drizzle-orm --legacy-peer-deps
```

驗收:`package.json` `dependencies` 內出現 `better-sqlite3` 與 `drizzle-orm`。

---

### A2. 安裝 dev 相依

```bash
npm i -D @types/better-sqlite3 drizzle-kit vite-plugin-static-copy --legacy-peer-deps
```

驗收:`package.json` `devDependencies` 內出現以上三個。

---

### A3. `package.json` 加 db npm scripts

在 `scripts` 加兩條:

```json
"db:generate": "drizzle-kit generate",
"db:check":    "drizzle-kit check"
```

驗收:`npm run db:check` 能執行(此時會抱怨找不到 schema,正常,下面 B2 之後就 OK)。

---

### A4. `package.json` 加 `build.asarUnpack`

在 `build` 區塊加:

```json
"asarUnpack": [
  "**/node_modules/better-sqlite3/**"
]
```

驗收:`npm run build:win` 不要立刻跑,此步只改 JSON;留到 G2 一起驗。

---

### A5. 建 `drizzle.config.ts`(專案根)

```ts
import {defineConfig} from 'drizzle-kit'

export default defineConfig({
  schema: './electron/main/db/schema/index.ts',
  out: './electron/main/db/migrations',
  dialect: 'sqlite',
  verbose: true,
  strict: true
})
```

驗收:檔案存在;`npm run db:check` 仍會抱怨找不到 schema(下階段解)。

---

## 階段 B:Schema 與 Migration 產出

### B1. 建 `electron/main/db/schema/logs.ts`

按 [08 §8](./08-本地數據庫設計.md#8-drizzle-schema-寫法) 的 schema 抄入,export `logs` table + `LogRow` / `NewLog` / `LogLevel` / `LogSource` 型別。

驗收:`vue-tsc --noEmit` 通過(typecheck);此步不引用到 better-sqlite3 runtime,純型別檔。

---

### B2. 建 `electron/main/db/schema/index.ts`

```ts
export * from './logs'
```

驗收:`npm run db:check` 不再抱怨找不到 schema(可能會抱怨 migration journal,正常)。

---

### B3. 產第一份 migration

```bash
npm run db:generate
```

驗收:
- `electron/main/db/migrations/0000_*.sql` 生成
- `electron/main/db/migrations/meta/_journal.json` 生成
- 開啟 SQL 看一眼,內容是 `CREATE TABLE logs(...)` + 兩個索引

---

### B4. commit 階段 A + B

```bash
git add package.json package-lock.json drizzle.config.ts electron/main/db
git commit -m "feat(db): add better-sqlite3 + drizzle scaffolding, logs schema"
```

驗收:git status 乾淨,沒漏檔。

## 階段 E:主進程串接(logger 雙寫)

### E1. `electron/main/utils/logger.ts` 加 `attachLogService`

頂部加:

```ts
import type {LogService} from '../db/services/log.service'
let _logService: LogService | null = null
export function attachLogService(svc: LogService): void { _logService = svc }
```

驗收:typecheck 通過;此時 logger 行為仍跟舊版一樣(沒任何 caller `attach`)。

---

### E2. 主進程 logger 4 個 method 各加一行 DB 寫入

`debug` / `info` / `warn` / `error` 內,在原本 console / writeLine 邏輯之後各加一行:

```ts
_logService?.write({level: 'debug', source: 'main', module, message, args})
// info / warn / error 對應 'info' / 'warn' / 'error'
```

注意:`writeLine('main', line)` **保留不動**(只 ERROR 寫 txt 的原邏輯)。

驗收:typecheck 通過;dev 跑 App,看主進程 console 還是有原本格式輸出。

---

### E3. `writeRendererLog()` 加一行 DB 寫入

在最後加:

```ts
_logService?.write({
  level: level.toLowerCase() as LogLevel,
  source: 'renderer',
  module,
  message,
  args
})
```

驗收:typecheck 通過。

---

### E4. `electron/main/index.ts` 內 try/catch 初始化 DB

按 [08 §16](./08-本地數據庫設計.md#16-啟動順序) 的順序,在 `initLogFileWriter()` 之後、`ConfigManager` 之前加:

```ts
try {
  dbManager = new DatabaseManager()
  dbManager.init()
  const logService = new LogService(dbManager)
  attachLogService(logService)
  logService.cleanupOlderThan(14)
} catch (err) {
  console.error('[App] DB 初始化失敗,日誌只走 txt', err)
}
```

驗收:dev 啟動 App,console 看到正常啟動(沒看到 `[App] DB 初始化失敗`)。

---

### E5. `electron/main/index.ts` 的 `gracefulShutdown` 加 `dbManager?.close()`

在 `windowManager?.destroyAll()` 之前加 `dbManager?.close()`。

驗收:正常退出 App 不報錯(從托盤退出測試)。

---

### E6. dev 驗證:主進程日誌進 DB

dev 跑 App,讓主進程觸發任意一條 `logger.info(...)`(例如啟動 banner 內已有的)。

開 SQLite Viewer(VSCode 擴展)開 `<專案根>/app.db`,查 `logs` 表 — 應該有 `source = 'main'` 的紀錄。

驗收:看得到資料,`level` / `module` / `message` / `args` 都正確。

---

### E7. commit 階段 E

```bash
git commit -m "feat(logger): dual-write to SQLite alongside txt"
```

---

## 階段 F:渲染端 forward 改造

### F1. 把 `LogService` 注入到 log handler

修改 `registerLogHandlers` 簽名:

```ts
export function registerLogHandlers(logService: LogService | null): void { ... }
```

驗收:typecheck 通過。

---

### F2. handler 內 `LOG_WRITE` 加一行 DB 寫入

在原本 `writeRendererLog(...)` 之後加:

```ts
logService?.write({
  level: (entry.level ?? 'INFO').toLowerCase() as LogLevel,
  source: 'renderer',
  module: entry.module,
  message: entry.message,
  args: entry.args
})
```

驗收:typecheck 通過。

---

### F3. `registerAllHandlers` 接收並下傳 logService

`electron/main/ipc-handlers/index.ts` 內 `registerAllHandlers` 加 `logService` 參數,並把它傳給 `registerLogHandlers`。

驗收:typecheck 通過。

---

### F4. `electron/main/index.ts` 把 logService 傳給 `registerAllHandlers`

`logService` 變數要拉出 try/catch 作用域,以便傳遞;若 init 失敗則為 `null`。

驗收:typecheck 通過;dev 啟動 App 正常。

---

### F5. 渲染端 logger:debug / info / warn 三個方法都加 forward

`src/utils/logger.ts` 內,把原本只在 `error` 內的 `forwardToMain(...)` 也加到 `debug` / `info` / `warn` 三個方法的 console 輸出之後。

驗收:typecheck 通過。

---

### F6. dev 驗證:渲染端日誌進 DB

dev 跑 App,在任一 Vue 組件加一行 `logger.info('test from renderer', 'Test')`(臨時),觸發後查 `logs` 表,應有 `source = 'renderer'` 的紀錄。

驗收:測完把臨時加的那行移除。

---

### F7. commit 階段 F

```bash
git commit -m "feat(logger): forward all renderer levels to main process for DB"
```

```bash

```

驗收:安裝完啟動 App 不報錯;`%APPDATA%\ichiaDesktop\app.db` 存在;txt 檔在 `%APPDATA%\ichiaDesktop\logs\` 仍存在且只記 ERROR。

### G3. 拔掉 DB 模擬降級

關 App,把 `%APPDATA%\ichiaDesktop\app.db` 改名為 `app.db.lock-test`,然後手動建一個唯讀檔頂著 `app.db` 名字(或更直接:刪 `app.db` 然後讓 `%APPDATA%\ichiaDesktop\` 設唯讀)。

重啟 App,確認:

- App 照常啟動
- 主進程 console 看到 `[App] DB 初始化失敗,日誌只走 txt`
- 觸發 ERROR,txt 檔仍有寫入
- DB 寫入端靜默 noop,沒任何業務面異常

驗收完恢復原狀(把唯讀屬性拿掉、刪測試檔)。

## 收尾

跑完 G3 表示本期全部完成。整個過程預期:

- 階段 A-D:基礎建設,大致 1 commit/階段,共 4 個 commit
- 階段 E-F:核心改造,各分 1~2 commit
- 階段 G:驗證,通常不產 commit

實作過程中遇到偏離設計的取捨,先回頭看 [08](./08-本地數據庫設計.md) 對應章節;若 08 沒覆蓋的情況,**寫進 PR description**,別偷偷改了就過。
