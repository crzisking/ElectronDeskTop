# 靈感速記(Idea Capture)設計

> 狀態:已實作並上線(全域熱鍵速記 + 想法庫;桌面端 + tmbom 後端)。桌面端 + tmbom 後端各自的改動都在本文件。
> 存儲決策:**不落本地 SQLite,全部走後端**(桌面端是純採集/展示層,資料唯一真身在 tmbom)。
>
> **邊界(硬約束)**:本功能只做「收集 → 整理匯總」。
> - ❌ **絕不與工作採集(work-collect)的截圖 / 監控聯動** —— 不自動截屏、不探測你在用哪個 app/網址、不依賴 work-collect
    模組。想記什麼由使用者主動寫/貼。
> - 生命週期只做輕量狀態(收集類工具,不是開發工單系統)。
> - 「關聯」只在本功能內(idea ↔ idea),不跨功能牽連。

## 1. 目標與核心矛盾

隨時冒出的想法(改進點 / 靈感 / 待辦)要能**幾秒鐘記下來**,但幾天後再看要能**還原當時的現場**——否則只剩一句看不懂的碎片。

核心矛盾:**記的時候要快,看的時候要全**。設計原則:零成本的上下文(時間 / 活動視窗標題)自動帶上;現場畫面靠**系統截圖 + 直接貼上
**(人人都會 Win+Shift+S,不自造截圖功能),附件支援任意類型、皆可選。AI 在填寫完後幫忙**改善描述、追問補細節**,把碎片變成以後能看懂的記錄。

可見範圍:**自己** 或 **部門**。兩種都存後端,只是查詢範圍不同(自己 = 只有本人可見;部門 = 同部門可見)。

## 2. 使用者流程

### 記錄(目標 < 10 秒)

```
任意時刻按全域快捷鍵(預設 Ctrl+Shift+Space,可在設定改)
  → 彈出速記小窗(無邊框、置頂、居中,類 Spotlight)
      ┌────────────────────────────────────┐
      │ 類型:[改進點▾](改進點/問題/靈感/待辦)   │
      │ 💡 一句話說想法(必填)                 │
      │ ┌────────────────────────────────┐ │
      │ │ 例:BOM 匯入的重複料號可以自動合併   │ │
      │ └────────────────────────────────┘ │
      │ 🎯 現在的痛點是?(建議填)              │
      │ ┌────────────────────────────────┐ │
      │ │ 例:每次匯入都要手動排查,半小時起步  │ │
      │ └────────────────────────────────┘ │
      │ ✅ 期望改成怎樣?(可空)                │
      │ ┌────────────────────────────────┐ │
      │ │ 例:匯入時自動提示合併,點一下確認   │ │
      │ └────────────────────────────────┘ │
      │ 📎 附件:拖放 / 點選 / Ctrl+V 貼圖(可選)│
      │ [縮略圖][file.xlsx ×] …               │
      │ ○ 自己 ○ 部門  [✨AI完善] [保存]        │
      └────────────────────────────────────┘
  → Ctrl+Enter 保存(直傳後端) / Esc 丟棄 / Tab 跳欄位
```

- **[保存]**:純保存,秒回秒關窗,AI 不介入
- **[✨AI完善]**:保存 + **標記交給後台 AI**,一樣秒回秒關窗 —— 使用者不等待,AI 版稍後自己出現(見「AI 完善(後台異步)」)
- 兩顆分開 = 保留觸發控制:值得潤色的按 AI,流水賬直接存,不是每條都自動燒 AI

**引導式三段結構** —— 不給空白框,用「問題」引導把重要的寫出來:

1. **想法本體**(必填):一句話。這是唯一必填,趕時間敲一句也能存(快路徑不犧牲)
2. **場景/痛點**(建議填):回顧時「為什麼當時這麼想」全靠它 —— 這是還原現場最值錢的欄位
3. **期望/下一步**(可空):讓這條記錄變得可行動

引導的細節:

- **提示語是具體例句,不是欄位名** —— placeholder 直接示範一條寫得好的記錄長什麼樣
- **問題隨類型切換**(選了類型,三個引導問題跟著換):

| 類型  | 第 2 欄問的是     | 第 3 欄問的是   |
|-----|--------------|------------|
| 改進點 | 現在的痛點是?      | 期望改成怎樣?    |
| 問題  | 怎麼發生的(能重現嗎)? | 影響多大 / 多急? |
| 靈感  | 由什麼觸發的?      | 可以用在哪?     |
| 待辦  | 為什麼現在記下來?    | 期限 / 優先級?  |

- **保存時軟提醒不阻塞**:想法填了但場景空 → 保存鈕旁淡淡提示一句「補一句場景,以後更好回憶」,再按一次照存 ——
  引導不是強制,強制會讓人不用這功能
- **[AI完善]**:填完後 AI 幫忙改善描述 + 追問補細節,見下方「AI 輔助」

其餘採集:

- **自動(僅時間 + 一次性視窗標題)**:時間戳;快捷鍵按下瞬間**讀一次**前景視窗標題當上下文線索。⚠️ 就這麼多 —— **不截圖、不探測
  process/exe/URL、不做來源分類、不與 work-collect 聯動**(那是監控向,本功能明確不碰)。此標題獨立實作(一行 Win32/Electron
  讀取),不依賴 work-collect 模組;設定裡可整個關掉
- **附件(可選,任意類型)**:三種方式進來 —— 拖放、點選檔案、**Ctrl+V 直接貼**(系統截圖 Win+Shift+S 截完貼進來就是一張圖附件;貼上的圖自動轉
  PNG 附件並顯示縮略圖)。**不自造截圖按鈕** —— 系統截圖人人會用、還能框選,重複造只會更難用。保存時隨 multipart 直傳後端,桌面端不留副本

保存 = 一次 `POST api/IdeaCapture/create`(multipart:meta + 截圖 + 附件)。**成功才關窗**;失敗(後端不通 / 超時)
小窗不關、內容保留,顯示錯誤 + 重試按鈕 —— 想法不會丟,但**離線時不可用**(此為全走後端的已知取捨,見 §5)。

### 回顧

主窗側邊欄新入口「**想法庫**」,資料全部即時查後端:

- 頂部篩選:狀態(預設隱藏 `archived`)/ 類型 / 標籤(`tag:ERP` 或點 tag);tab 分「我的 / 部門」
- 卡片:類型徽章 + 狀態徽章 + AI 徽章(`pending`→「AI整理中…」/`done`→「✨AI版」)+ 縮略圖(第一個圖片附件,有才顯)+ 標題 + 標籤 +
  時間
- 詳情:三段分區 **想法 / 場景 / 期望**(空欄不顯)+ 有 AI 版可切「原文 / ✨AI版」+ **Action Items(checklist)** + AI 待確認點(
  可就地補寫)+ 附件大圖/下載 + **關聯想法**區 + 狀態下拉(改生命週期)+ 手動加減標籤
- `none`/`failed` 的記錄有 [✨完善] 補觸發 / 重試;主窗收 `push:idea-refined` 且本頁開著就就地刷新徽章與 AI 版
- 縮略圖 / 附件走 `GET attachment` 按需拉,前端記憶體級快取,不落盤

### AI 完善(後台異步)

**定位:AI 是後台任務,不是同步流程。** 使用者按 [✨AI完善] 保存後,窗立刻關、人立刻走;完善在背景跑,一分鐘後 AI 版自己出現在想法庫。同步等
AI 的體驗差、久了沒人點,所以不做「填完站在那等結果」。

復用現有模型設定的 active provider(LlmClient,與 agent 同一套)。

**觸發(保留控制,不是每條都自動)**:

- 記錄時按 [✨AI完善](vs [保存])—— 逐條選,值得的才花 AI
- 想法庫詳情頁對「還沒 AI 版 / 失敗」的記錄有 [✨完善] 按鈕,可補觸發 / 重試

**流程(全程不阻塞使用者)**:

```
按 [✨AI完善] → create(帶 wantAI=true)→ 後端記 RefineStatus='pending' → 秒回關窗
      ↓ (背景,主進程)
主進程完善佇列:讀該筆 → 呼叫 LLM → PATCH 回後端(標題/潤色版/標籤/RefineStatus='done')
      ↓
想法庫該卡狀態徽章隨之變:AI整理中… → ✨AI版
```

**AI 產出**(一次呼叫回 JSON,寫進 PolishedText / Title / Tags / ActionItems / AiQuestions):

- **潤色版**:把三段整合潤成更清楚的版本(原文三欄不動,潤色版另存,詳情頁「原文 / AI版」可切換)
- **標題 + 多個領域標籤**:自動打 domain tag(如 `ERP` / `MES` / `RAG` / `SQL` / `UI`…),一條可多標籤 —— 之後想法庫可按 tag
  篩選 / 搜(`tag:ERP`),量大時這是找回東西的關鍵(見 §2.5)
- **Action Items**(可空):把想法拆成可執行的下一步(例:「①調查原因 ②統計影響範圍 ③找採購確認」)—— 比純潤色值錢,想法能直接變待辦。詳情頁以
  checklist 呈現
- **待確認點**(可空):AI 覺得影響回顧理解的缺口,以問句列出(例:「『很慢』約幾秒?」)—— 使用者日後在想法庫看到,想補就補(
  補完可再點 [✨完善] 跑一輪);不是彈框追問,不打斷

**原則**:AI 只依使用者寫的內容完善,**不瞎編**(拿不準保留原話 / 列成待確認點);模型未配置時 [✨AI完善] 置灰只留 [保存];AI
全程失敗只是沒 AI 版,不影響記錄本身 —— 增強不是依賴。

**取捨**:完善計算放主進程(復用模型設定的 provider,與 agent 一致)。代價:按完 AI 立刻關 app,佇列裡沒跑完的會停在 `pending`(
重開後不自動續,靠想法庫的 [✨完善] 重試)。若日後要「關機也能跑完」的耐久性,再把計算移到後端(改用後端自己的
AI,喪失使用者選模型的一致性)—— 目前不值得,想法完善不是即時剛需。

## 2.5 生命週期 / 標籤 / 關聯

三個讓「一年後兩千條」不變成廢墟的機制。**刻意保持輕**:本功能是收集 + 整理匯總,不是工單/專案系統。

### 生命週期(輕量狀態)

沒有狀態的話,一年後全是「想到就記」的平鋪流水,分不清哪些還活著、哪些早處理完了。但也**不做 Draft/Doing/Done 那套開發工作流
**(那是 issue tracker 的事)。只留 4 個狀態:

| 狀態             | 含義                          |
|----------------|-----------------------------|
| `inbox`(待整理)   | 預設。剛記下、還沒回頭看的               |
| `accepted`(採納) | 看過、認為值得推進的                  |
| `done`(已落實)    | 已經做了 / 已解決                  |
| `archived`(歸檔) | 不採納 / 過時 / 重複 —— 收起來不刪,保留痕跡 |

- 狀態在想法庫手動改(下拉 / 快捷操作),不自動流轉
- 想法庫預設濾掉 `archived`,可切「全部」;可按狀態篩選
- 只是一個 enum 欄位 + 篩選,不掛任何審批 / 指派 / 期限邏輯

### 標籤(自動為主)

- 主要靠 §「AI 完善」自動打**領域多標籤**(ERP/MES/RAG/SQL/UI…),使用者也能手動加減
- 想法庫支援 tag 篩選與搜尋(`tag:ERP`);tag 雲/常用 tag 快速點選
- 標籤獨立成表(見 §4.1),不塞 JSON —— 方便聚合統計「哪類想法最多」

### 關聯(僅本功能內)

⚠️ 只關聯 **idea ↔ idea**,不跨功能(不連 BPM / 報修 / work-collect 等)。

- **手動關聯**(P2):詳情頁「關聯想法」區,**只能搜自己的其他想法**掛上(不能關聯別人的)
  ,雙向可見。解決「同一件事分幾天記了好幾條」——串起來看就完整了。只關聯自己的,天然避免「私密想法 A 關到部門想法 B、別人看 B
  就發現 A 存在」的跨可見性洩露
- **AI 建議相似**(P3,較重):要語意相似度得上 embedding + 向量檢索,後端沒有現成能力,列後期;真做時也只是「建議」,由使用者確認才建立關聯,不自動連

## 3. 桌面端設計

### 3.1 新視窗:`idea-capture` 速記小窗

第 6 個視窗,照多視窗慣例走(`electron.vite.config.ts` 兩處 input + window-manager):

- `src/windows/idea-capture/index.html` + 獨立 Vue app + `electron/preload/idea-capture.preload.ts`
- 無邊框(`frame:false`)、置頂(`alwaysOnTop`)、不進工作列(`skipTaskbar`)、失焦即隱藏
- **首次建立後只 hide 不 destroy** —— 快捷鍵喚起要瞬間出現,重建 BrowserWindow 太慢;順帶讓「保存失敗時的草稿」天然留在窗內
- ⚠️ 跟 agent 視窗一樣**不設 parent**(子視窗會跟主窗一起最小化)

### 3.2 全域快捷鍵

- 主進程 `globalShortcut.register`,app ready 時註冊;熱鍵字串存 config DB(`ideaCapture.hotkey`),改動後重註冊
- 註冊失敗(被別的程式佔用)→ log + 主窗通知,引導去設定換一個
- 悬浮球右鍵選單加同名入口(不依賴快捷鍵也能用)

### 3.3 附件與貼圖(無截圖功能)

**不做截圖按鈕** —— 使用者用系統截圖(Win+Shift+S,可框選)截完直接 Ctrl+V 貼進小窗。三種進附件的方式:

- **貼上**:小窗監聽 paste 事件,`clipboardData` 有圖 → 轉 PNG buffer 當附件(名字自動 `貼圖-HHmmss.png`),顯示縮略圖;剪貼簿是檔案列表也收
- **拖放**:drop 事件拿檔案路徑
- **點選**:`dialog.showOpenDialog`(任意類型,`multiSelections`)

任意類型都收,單檔 / 總量上限跟後端 `RequestSizeLimit(30MB)` 對齊,超限即時提示。貼上的圖暫存記憶體(renderer 傳 buffer
給主進程掛在草稿上),路徑型附件保存時主進程讀檔 —— 都只在 multipart 上傳時用,不落本地磁碟。

### 3.4 資料訪問層(無本地存儲)

**不建本地表、不加 db feature、無同步服務。** 桌面端只有一層薄 API client:

- HTTP 由**主進程**發(同 auth / work-collect 慣例,躲 CORS;截圖 buffer 也在主進程手上)
- `electron/main/services/idea-capture/api-client.ts`:create / listMy / listDept / detail / getAttachment / patch(
  狀態·文字·tag)/ relations / delete,對後端 §4.2
- userName 由桌面端從本機 JWT 解出顯式傳(同 work-collect 的 AllowAnonymous 慣例)
- 後端 URL 走 config(主進程讀不到 VITE_*,照現有規矩從 config/IPC 取)

### 3.5 IPC(照慣例:channels → bridge → handlers)

- `idea:create`(meta + `wantAI` + 附件:路徑型主進程讀檔、貼圖型收 buffer,組 multipart 上傳;`wantAI=true`
  時主進程於回應後把該筆丟進背景完善佇列)
- `idea:list-my` / `idea:list-dept`(分頁 + 篩選,轉發後端)/ `idea:detail`
- `idea:get-attachment`(拉縮略圖 / 附件,回 buffer/dataURL)
- `idea:patch`(改狀態 / 文字 / tag)/ `idea:relate` / `idea:unrelate`
- `idea:refine`(想法庫手動補觸發 / 重試:標 pending 後丟進同一背景佇列)
- `idea:delete`
- push:`push:idea-refined`(某筆完善完成 → 發給**主窗 renderer**;想法庫頁開著就就地刷新該卡,主窗最小化成悬浮球時 push
  無意義、下次開頁重查即可)
- 高風險(讀使用者指定路徑的檔案上傳)的 `idea:create` 按規範加 zod schema

**背景完善佇列**(主進程,`services/idea-capture/refiner.ts`):in-memory 佇列,單並發,復用 AgentRuntime 同款 LlmClient;每筆
LLM 完成後 `PATCH .../ai` 寫回,再 push `idea-refined`。失敗退避重試數次仍失敗 → PATCH `RefineStatus='failed'`。app
退出佇列即清(未跑完停 pending,靠想法庫重試)。

### 3.6 想法庫頁面

`views/IdeaLibrary/`(路由骨架)+ `features/idea-capture/`(api/types/composables/components)標準形態。列表 /
詳情都即時查後端,無離線快取。

## 4. 後端設計(tmbom)

照 WorkCollectController 的 desktop 慣例:`[AllowAnonymous]` + client 顯式傳 `userName`(`ResolveUser` 兜底 CurrentUser);*
*所有入庫文字(create 三段 + AI 回寫的 polished/actionItems/questions/tags)一律過 `ITextSanitizeService`(繁化+escape)**
;部門**由後端查員工表解析**,不信任 client 傳的部門。

### 4.1 表(sql-migrations 加 DDL)

主表 + 3 張子表(附件 / 標籤 / 關聯都獨立,不塞 JSON):

```
IdeaCapture(主表)
  Id             bigint PK identity
  ClientId       nvarchar(64)  UNIQUE   (desktop 產的 uuid,重試冪等鍵)
  UserId         nvarchar(32)           (工號)
  UserName       nvarchar(64)           (姓名,冗余方便列表)
  DeptNo         nvarchar(32)           (create 時後端查員工表寫入)
  Visibility     nvarchar(16)           ('private' | 'dept')
  IdeaType       nvarchar(16)           ('improve' | 'issue' | 'inspiration' | 'todo')
  Status         nvarchar(16)           ('inbox' | 'accepted' | 'done' | 'archived';預設 inbox)
  Title          nvarchar(200) NOT NULL (create 時 Content 首句截斷兜底,AI 版回來覆蓋)
  Content        nvarchar(max)          (想法本體,必填)
  Scene          nvarchar(max) NULL     (場景/痛點 —— 引導第 2 欄)
  Expectation    nvarchar(max) NULL     (期望/下一步 —— 引導第 3 欄)
  PolishedText   nvarchar(max) NULL     (AI 潤色整合版)
  ActionItems    nvarchar(max) NULL     (JSON string[];AI 拆的可執行下一步)
  AiQuestions    nvarchar(max) NULL     (JSON string[];AI 的待確認點)
  RefineStatus   nvarchar(16)           ('none' | 'pending' | 'done' | 'failed')
  ActiveWindow   nvarchar(300) NULL     (一次性視窗標題;可空)
  CreatedAt / UpdatedAt datetime2
  IsDeleted      bit DEFAULT 0          (軟刪)

IdeaAttachment(1:N,Q8 —— 不用 JSON,以後排序/刪除/OCR/hash/縮略圖都靠欄位)
  Id          bigint PK identity
  IdeaId      bigint FK → IdeaCapture.Id
  FileName    nvarchar(260)
  StoredPath  nvarchar(300)      (伺服器磁碟相對路徑)
  Size        bigint
  IsImage     bit
  Sha256      nvarchar(64)       (去重 / 校驗;先留欄位)
  SortOrder   int
  CreatedAt   datetime2

IdeaTag(標籤,可聚合統計)
  Id        bigint PK identity
  IdeaId    bigint FK → IdeaCapture.Id
  Tag       nvarchar(50)
  Source    nvarchar(8)          ('ai' | 'user')
  UNIQUE(IdeaId, Tag)

IdeaRelation(關聯,僅 idea↔idea;P2/P3)
  Id          bigint PK identity
  IdeaId      bigint FK
  RelatedId   bigint FK
  CreatedBy   nvarchar(8)          ('user' | 'ai')
  UNIQUE(IdeaId, RelatedId)       (無方向,存一條、查雙向)
```

附件檔落伺服器磁碟(路徑進 appsettings),不進 DB。

### 4.2 接口(`IdeaCaptureController : ApiControllerBase`)

| 方法     | 路由                                                       | 說明                                                                                                                                                                                                                                                                                              |
|--------|----------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| POST   | `api/IdeaCapture/create`                                 | multipart:`meta`(JSON,含 `wantAI`)+ `attachments`(多檔,任意類型,可空)。`wantAI=true` 時後端存 `RefineStatus='pending'`,否則 `'none'`。**冪等**:ClientId 已存在 → 不重插、**不重存附件**,直接回既有 `{id}`(容忍首次成功但響應丟失的重試)。**Title 兜底**:meta 沒帶 title 時後端用 Content 首句截斷填(Title NOT NULL),AI 版回來再覆蓋。`RequestSizeLimit(30MB)`。回 `{id}` |
| PATCH  | `api/IdeaCapture/{clientId}/ai`                          | 背景完善寫回:`{title,polishedText,actionItems,aiQuestions,tags[],refineStatus}`(tags 落 IdeaTag,source='ai')。**文字欄位同樣過 ITextSanitizeService(繁化+escape)**,與 create 一致。校驗擁有者                                                                                                                             |

| GET    | `api/IdeaCapture/my`                                     | 自己的記錄(private + dept 都回),分頁 + `since` + 篩選(`status` / `type` / `tag`)。想法庫「我的」tab。預設不含 archived                                                                                                                                                                                                  |
| GET    | `api/IdeaCapture/dept`                                   | 部門想法列表。後端由 userName 查部門 → 同部門且 `Visibility='dept'`,分頁 + `since` + 同篩選                                                                                                                                                                                                                           |
| GET    | `api/IdeaCapture/{clientId}`                             | 單筆詳情(含附件清單 / 標籤 / 關聯的精簡卡片)                                                                                                                                                                                                                                                                      |

## 5. 全走後端的取捨(明確接受)

| 得到                                  | 代價                                                    |
|-------------------------------------|-------------------------------------------------------|
| 無本地表 / 無 migration / 無同步狀態機,桌面端大幅簡化 | **離線 / 後端掛了就不能記**(保存失敗時小窗保留內容可重試,但不持久 —— app 退出草稿就沒了) |
| 「我的」記錄天然跨機器(換電腦 / 重灌即見)             | 想法庫列表 / 縮略圖每次都要打後端(內網延遲可忽略)                           |
| 部門可見性只在後端一處控制                       | 後端成為 P1 硬依賴,不能純桌面端先行                                  |

公司內網場景後端可用性高,接受上述代價。若日後真需要離線,再補「失敗草稿落暫存 JSON」的輕量兜底,不回退到本地 SQLite。

## 6. 分期

| 期           | 內容                                                                                                                                                               | 依賴              |
|-------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------|-----------------|
| **P1(MVP)** | 後端主表 + IdeaAttachment/IdeaTag 子表 + 基礎接口(create / my / dept / attachment / patch / delete)+ 桌面端快捷鍵 / 小窗(三段引導 + 貼圖附件)+ 想法庫「我的」「部門」tab + **生命週期狀態(篩選/改)** + 手動標籤    | 前後端一起動          |
