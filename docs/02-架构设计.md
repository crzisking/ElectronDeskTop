# 架構設計文檔

## 1. 整體架構概覽

本應用基於 Electron 的三進程模型構建，嚴格遵循安全最佳實踐。

```
┌────────────────────────────────────────────────────────────────────┐
│                           操作系統                                  │
│                                                                    │
│  ┌─────────────────────┐      ┌──────────────────────────────────┐ │
│  │    主進程 Main       │      │         渲染進程 Renderer         │ │
│  │  (Node.js 環境)     │      │   (Chromium 沙盒環境)            │ │
│  │                     │      │                                  │ │
│  │  • 窗口生命週期      │ IPC  │  ┌────────────────────────────┐  │ │
│  │  • 系統托盤          │◄────►│  │     Vue 3 主窗口應用        │  │ │
│  │  • 浮球窗口控制      │      │  └────────────────────────────┘  │ │
│  │  • 配置文件讀寫      │      │                                  │ │
│  │  • Auth Token 存儲  │      │  ┌────────────────────────────┐  │ │
│  │  • Token 加密/解密   │      │  │   浮球 Vue 應用             │  │ │
│  │                     │      │  └────────────────────────────┘  │ │
│  └──────────┬──────────┘      └──────────────────────────────────┘ │
│             │                                                       │
│  ┌──────────▼──────────┐                                           │
│  │   預加載腳本 Preload │                                           │
│  │  (沙盒橋接層)       │                                           │
│  │  • 暴露 electronAPI │                                           │
│  │  • 封裝 IPC 調用    │                                           │
│  │  • 禁止直接訪問 Node│                                           │
│  └─────────────────────┘                                           │
└────────────────────────────────────────────────────────────────────┘
```

### 安全原則

| 設置 | 值 | 說明 |
|------|-----|------|
| `contextIsolation` | `true` | 渲染進程無法訪問 Node.js 環境 |
| `nodeIntegration` | `false` | 渲染進程無法直接 `require('fs')` |
| `sandbox` | `false` | 保留以讓 preload 腳本能使用 Electron/Node 能力 |

所有特權操作必須通過 Preload 定義的 `window.electronAPI` IPC 橋接調用。

## 2. IPC 通信設計

IPC（Inter-Process Communication）是主進程和渲染進程通信的唯一方式。

### 三種通信模式

**模式 A：請求-響應（invoke / handle）**

渲染進程需要從主進程獲取數據時使用，返回 Promise。

```
渲染進程                    預加載腳本                  主進程
   │                           │                        │
   │ electronAPI.config.read() │                        │
   │──────────────────────────►│                        │
   │                           │ ipcRenderer.invoke()   │
   │                           │───────────────────────►│
   │                           │                        │ ipcMain.handle()
   │                           │                        │ 讀取配置文件
   │                           │     Promise<Config>    │
   │                           │◄───────────────────────│
   │    resolve(config)        │                        │
   │◄──────────────────────────│                        │
```

使用場景：讀取配置、讀取 Token、查詢窗口狀態

---

**模式 B：單向命令（send / on）**

渲染進程發送操作命令，不需要返回值。

```
渲染進程                    預加載腳本                  主進程
   │                           │                        │
   │ electronAPI.window.hide() │                        │
   │──────────────────────────►│                        │
   │                           │ ipcRenderer.send()     │
   │                           │───────────────────────►│
   │                           │                        │ ipcMain.on()
   │                           │                        │ win.hide()
```

使用場景：最小化窗口、顯示浮球、隱藏浮球、退出應用

---

**模式 C：主進程推送（webContents.send）**

主進程主動通知渲染進程，例如操作系統事件發生時。

```
主進程                          渲染進程
  │                                │
  │  窗口被最大化（OS 事件）         │
  │                                │
  │  mainWin.webContents.send()    │
  │───────────────────────────────►│
  │                                │ ipcRenderer.on()
  │                                │ 更新最大化按鈕狀態
```

使用場景：窗口最大化/還原狀態同步、托盤點擊事件、配置文件外部變更通知

---

### IPC 頻道常量

所有頻道名稱統一定義在 `electron/shared/ipc-channels.ts`，以常量形式使用，禁止在調用處寫裸字符串（防止拼寫錯誤難以排查）。

主要頻道分類：
- **配置管理**：`config:read`、`config:write`
- **窗口控制**：`window:minimize`、`window:maximize`、`window:close`、`window:open-child`
- **浮球**：`floating-ball:show`、`floating-ball:start-drag`、`floating-ball:stop-drag`
- **Token**：`auth:get-token`、`auth:set-token`、`auth:delete-token`
- **主進程推送**：`push:config-changed`、`push:window-maximized`、`push:tray-clicked`

## 3. 窗口管理設計

### 主窗口

- 尺寸：1200×800，最小 900×600
- `frame: false`：隱藏 OS 原生標題欄，由 `TitleBar.vue` 自定義渲染
- `show: false`：先隱藏，`ready-to-show` 事件後再顯示（防止白屏閃爍）
- 關閉按鈕行為：調用 `win.hide()`（隱藏到浮球），不是真的關閉進程

### 浮球窗口

- 尺寸：80×80 像素（正方形）
- `transparent: true`：整個窗口透明，配合 CSS `border-radius: 50%` 呈現圓形
- `alwaysOnTop: true`：跨所有應用置頂
- `skipTaskbar: true`：不出現在 Windows 任務欄
- Windows 上額外調用 `setAlwaysOnTop(true, 'screen-saver')`，防止被全屏窗口壓住

> **注意**：浮球的 preload 腳本源碼是 `floating-ball.preload.ts`，構建後輸出為 `floatingBall.js`。主進程引用的是輸出路徑，不要混淆。

## 4. 浮球技術方案

### 圓形外觀

窗口本身是 80×80 正方形，通過以下組合實現視覺上的圓形：

1. `BrowserWindow` 設置 `transparent: true`（窗口背景透明）
2. HTML/CSS 設置 `background: transparent`（頁面背景透明）
3. `.ball` 元素設置 `border-radius: 50%`（圓形裁切）

四個角因為透明而不可見，只有中間的圓形區域顯示，無需 OS 層面的窗口遮罩。

### 拖動方案

**不使用** `-webkit-app-region: drag`，因為它會阻止所有鼠標事件（點擊失效）。

採用鼠標坐標輪詢方案：

1. 用戶按下鼠標 → 渲染進程發送 `BALL_START_DRAG`
2. 主進程記錄初始偏移量，啟動 16ms 定時器（~60fps）
3. 每次定時器觸發，讀取 `screen.getCursorScreenPoint()`，更新窗口位置（含邊界限制）
4. 用戶鬆開鼠標 → 發送 `BALL_STOP_DRAG`，停止定時器，觸發邊緣吸附動畫

### 邊緣吸附

停止拖動後，計算浮球中心到四個邊的距離，移動到距離最近的那條邊，用平滑動畫過渡。

### 右鍵菜單

浮球窗口只有 80×80px，Vue 覆蓋層菜單會被窗口邊界裁剪，因此改用 **Electron 原生菜單**：

```
用戶右鍵浮球
    ↓
FloatingBall.vue 捕獲 @contextmenu，阻止默認行為
    ↓
window.electronAPI.showContextMenu()（IPC 發送）
    ↓
主進程讀取 config.floatingBall.quickMenu
    ↓
Menu.buildFromTemplate(...).popup({ window: ballWin })
    ↓
用戶點擊菜單項 → 根據 action.type 執行對應操作
```

支持的 `action.type`：
- `show-main-window` → 顯示主窗口，隱藏浮球
- `navigate` → 顯示主窗口並跳到指定路由
- `open-url` → 調用 `shell.openExternal(url)` 在瀏覽器打開
- `quit-app` → `app.quit()` 完全退出

## 6. Auth Token 安全存儲方案

Token 的整個生命週期：

```
登錄成功
    ↓
明文 token（內存）
    ↓ safeStorage.encryptString(token)  →  OS Credential Store 加密
    ↓ Buffer → base64
    ↓ electron-store.set('encryptedToken', base64)
    → 寫入磁盤（加密後的 base64 字串）

應用重啟
    ↓
electron-store.get('encryptedToken')
    ↓ Buffer.from(base64, 'base64')
    ↓ safeStorage.decryptString(buffer)
    → 解密得到明文 token → 存入 authStore.accessToken（內存）

Axios 請求
    ↓
auth.interceptor.ts 讀取 authStore.accessToken
    ↓
注入 Authorization 請求頭（直接帶 token，不加 Bearer 前綴）
```

**絕對禁止的做法：**

- `localStorage.setItem('token', ...)` — 不加密，XSS 可竊取
- `sessionStorage` 或 Cookie 存 token — 同樣不安全
- `electron-store` 直接存明文 token — 磁盤上可讀

## 7. 路由設計

### 當前路由表

| 路徑 | 路由名 | 組件 | 說明 |
|------|--------|------|------|
| `/login` | `login` | `LoginView.vue` | 登錄頁，不需要認證 |
| `/unified-platform` | `unified-platform` | `UnifiedPlatformView.vue` | 統一平台，默認首頁 |
| `/internal-functions` | `internal-functions` | `InternalFunctionsView.vue` | 內部功能入口頁 |
| `/ai-bpm-finder` | `ai-bpm-finder` | `BpmFinderView.vue` | BPM 負責人查詢 |
| `/it-repair` | `it-repair` | `ITRepairView.vue` | IT 報修工單 |
| `/business` | `business` | `BusinessView.vue` | 業務安排與尋找 |

訪問 `/` 重定向到 `/unified-platform`。訪問未定義路由同樣重定向到 `/unified-platform`。

### 路由守衛邏輯

```
每次路由跳轉前執行 beforeEach：
  1. 更新 document.title（顯示「頁面名 - 企業桌面客戶端」）
  2. 若 isRestoringSession === true → 直接放行（等啟動恢復完成）
  3. 若目標路由 requiresAuth && 未登錄 → 強制跳 /login
  4. 若已登錄 && 試圖訪問 /login → 直接跳 /unified-platform
```
