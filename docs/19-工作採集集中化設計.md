# 工作採集集中化設計

> **承接**:[11-工作自動採集設計.md](./11-工作自動採集設計.md) 是當前桌面端「本地優先 + 個人視角」實作。
>
> **本文目標**:把採集從「100% 本地」演進為「服務端集中化 + 管理頁面 + 桌面端離線兼容」,支援:
>  1. 獨立**管理頁面**(web,跑在伺服器,**不嵌進 desktop**),共用 TMBOM 後端與認證
>  2. 管理員按使用者設定**採集間隔**(原本只有使用者自己在 desktop 內改)
>  3. 接口可查單一使用者的**全量採集數據**
>  4. **500 人同時採集**的後端架構承載
>
> **不改的契約**:
>  - desktop 端的 dHash / idle 偵測 / 工時區間 / 截圖端到端不落地 政策保留
>  - 既有 `POST /api/WorkCollect/analyze`(jpeg + window 標題 → category)接口語意保留,只擴充返回 / 落庫行為

---

## 0. TL;DR

```
┌──────────────────┐    HTTPS(JWT)    ┌─────────────────────────┐
│  Desktop (500x)  │ ────tick────────▶│  TMBOM 後端 (.NET API)  │
│  - dHash         │ ◀──config────────│  - /api/work/analyze     │
│  - idle 偵測      │                 │  - /api/work/my-config   │
│  - local SQLite  │                 │  - /api/work/my-records  │
│  - 個人 UI       │                  │                          │
└──────────────────┘                  │  Server SQLite/Postgres  │
                                      │  ├─ work_records         │
                                      │  ├─ work_user_configs    │
                                      │  └─ work_analyze_queue   │
┌──────────────────┐                  │                          │
│   Admin Web      │ ──HTTPS(JWT)────▶│  /api/admin/work/*       │
│  (新增,獨立)    │                  │                          │
│  - 用戶清單      │                  │  Worker Pool             │
│  - 改 interval   │                  │  - Qwen-VL 呼叫           │
│  - 查 records    │                  │  - rate limiter           │
└──────────────────┘                  └─────────────────────────┘
```

**核心設計取捨**:

1. **不放棄本地 SQLite** — 離線時桌面 UI 仍能看本人歷史;網路恢復後對齊(伺服器是 source of truth,本地是 cache)
2. **AI 分析改為佇列異步** — `POST /analyze` 變成入隊 + 202 Accepted,worker 拉佇列調 Qwen-VL,結果回推 desktop
3. **配置中心化** — desktop 不再讓使用者改 `intervalMinutes`,只允許讀 server config(管理員在 admin web 改);仍保留總開關(
   `enabled`)讓使用者本機關掉
4. **500 並發架構** — HTTP 入口無 LLM 同步調用,削峰靠佇列;worker pool 水平擴展;DashScope 多 key 輪詢 + rate limiter

## 1. 當前實現盤點

### 1.1 桌面端核心流程

```
WorkCollectorScheduler.tick()(每 N 分鐘,主進程)
  │
  ├─ 1. 螢幕鎖 / 非工時 / 主視窗未就緒 → skip
  ├─ 2. desktopCapturer 抓全屏 NativeImage
  ├─ 3. computeDHash(9x8 灰階差分,16 hex)
  ├─ 4. collectVisibleWindowTitles()(最多 10 個)
  │
  ├─ 5. detectIdle:
  │     - 系統 idle 秒數 >= 間隔秒數 - 5s grace
  │     - OR dHash 距離 <= 10 且前台視窗未變
  │     → 直接寫 idle 紀錄到本地 SQLite,不打 AI
  │
  └─ 6. 推 IPC PUSH_WORK_COLLECT_TICK → renderer
        renderer 走 createHttpClient(帶 JWT)
        POST /api/WorkCollect/analyze(multipart):
          screenshot=jpeg blob
          activeWindow / appName / allWindows / capturedAt
        ← 後端返回 {category, description, confidence, reason}
        renderer 透過 IPC 把結果送回 main,handler 寫進 work_records
```

### 1.2 桌面端本地 schema

```sql
CREATE TABLE work_records
(
    id                INTEGER PRIMARY KEY AUTOINCREMENT,
    capturedAt        INTEGER NOT NULL, -- Unix ms
    activeApp         TEXT,
    activeWindowTitle TEXT,
    category          TEXT    NOT NULL, -- coding / documenting / meeting / ...
    description       TEXT    NOT NULL,
    confidence        REAL,
    screenshotHash    TEXT,             -- dHash 16 hex
    reason            TEXT              -- AI 分類理由
)
CREATE INDEX idx_work_capturedAt ON work_records (capturedAt)
```

config 走 `app_settings_kv`:`workCollect.enabled` / `intervalMinutes` / `workStartHour` / `workEndHour`。

### 1.3 與後端的現有契約

| API                             | 用途                        | 認證                      | 後端職責              |
|---------------------------------|---------------------------|-------------------------|-------------------|
| `POST /api/WorkCollect/analyze` | 上傳 jpeg + 視窗清單,回 category | JWT(走 auth interceptor) | 同步呼叫 Qwen-VL,回傳結果 |

**目前**:後端是純 stateless,**不落庫**。所有資料在 desktop SQLite。

## 2. 需求拆解

| 需求                         | 對現有架構的衝擊                                                                      |
|----------------------------|-------------------------------------------------------------------------------|
| **R1** 管理頁面查單一使用者數據        | 後端必須**落庫**所有 user 的紀錄(目前不存)                                                   |
| **R2** 管理員設定 per-user 採集間隔 | 後端要有 **user_config 表** + desktop 啟動 / 定期拉取                                    |
| **R3** API 取一個人全量數據        | 在 R1 落庫前提下,加 `GET /api/work/my-records` + `GET /api/admin/users/{id}/records` |
| **R4** 500 人並發             | 現在同步調 Qwen-VL,500 並發堵死 → 必須改**佇列 + worker**                                   |
| **R5** 管理頁面不嵌 desktop      | 獨立 web 工程,跟 TMBOM 認證打通(已有 JWT)                                                |
| **R6** 接口共用 TMBOM 後端       | 不另起服務,在 TMBOM .NET API 內加新 controller                                         |

## 3. 整體架構

### 3.1 三條獨立數據流

```
數據流 A:配置同步(管理員 → desktop)
─────────────────────────────────────────
Admin Web ──PATCH /api/admin/users/{id}/work-config──▶ TMBOM API
                                                          │
                                                          ▼
                                                   work_user_configs 表
                                                          │
Desktop 每 N 分鐘輪詢 ◀──GET /api/work/my-config────────┘
        ├─ 比對本地 cache,變更 → 觸發 scheduler.restart()
        └─ 寫本地 SQLite 供離線兜底


數據流 B:採集上報(desktop → 後端)
─────────────────────────────────────────
Desktop tick
  ├─ idle 命中 → 寫本地 SQLite + 異步 sync 到後端(背景)
  └─ 非 idle  → POST /api/work/analyze
                 ├─ 入隊 → 202 Accepted + jobId
                 │
                 ▼
                Worker Pool
                 ├─ 從佇列拉
                 ├─ call Qwen-VL(rate limit + retry)
                 ├─ 寫 server work_records(打上 userId)
                 └─ Server push 結果回 desktop(WebSocket / SSE / polling)
                                          │
                                          ▼
                              Desktop 收到 → 寫本地 SQLite + UI 更新


數據流 C:管理員查詢
─────────────────────────────────────────
Admin Web ──GET /api/admin/users───────▶ TMBOM API
          ──GET /api/admin/users/{id}/records?since&until&category
          ──GET /api/admin/users/{id}/work-config
                                              │
                                              ▼
                                        server DB(權威源)
```

### 3.2 為何不走「desktop 不存本地、純讀 server」

簡單但**犧牲離線體驗**:

- 內網工程師午休出去吃飯回來想看上午採集 → 網路抖一下整個 UI 拉不到
- desktop 既有 UI(時間軸 / 圖表)已經寫好查本地 SQLite,改成 HTTP 是無收益的大改動

**選擇**:server 是 source of truth,desktop 本地 SQLite 作為 read-through cache。對齊策略見 §6.3。

## 4. 後端 API 契約

### 4.1 桌面端用(已有 JWT,從現有 auth 走)

| Method | Path                                        | 說明                                            |
|--------|---------------------------------------------|-----------------------------------------------|
| `GET`  | `/api/work/my-config`                       | 拉當前使用者的採集配置(管理員可改的那份)                         |
| `POST` | `/api/work/analyze`                         | 上傳 jpeg + 視窗清單,**入隊**,返回 202 + jobId          |
| `GET`  | `/api/work/recent-results?since=N`          | 輪詢拉最近結果(配合 §5.2 polling 模式;若上 WebSocket 則不需要) |
| `GET`  | `/api/work/my-records?since&until&category` | 查自己的歷史紀錄(離線恢復 / 跨機同步用)                        |
| `POST` | `/api/work/sync-idle`                       | 桌面端 idle 命中後背景上報,server 不過 AI 直接落庫            |

#### 4.1.1 `GET /api/work/my-config` 回應

```json
{
  "enabled": true,
  "intervalMinutes": 5,
  "workStartHour": 8,
  "workEndHour": 17,
  "updatedAt": 1735000000000
}
```

桌面端拿到後跟本地 `app_settings_kv` 比對:`updatedAt` 比本地新 → 覆蓋本地 + 重啟 scheduler。

#### 4.1.2 `POST /api/work/analyze` 請求 / 回應

```
Request(multipart/form-data):
  screenshot: jpeg blob
  activeWindow: string
  appName: string
  allWindows: string(JSON 陣列)
  capturedAt: number(Unix ms)
  screenshotHash: string(dHash 16 hex)

Response(202 Accepted):
  {
    "jobId": "uuid",
    "queuedAt": 1735000000000,
    "estimatedWaitMs": 1500
  }
```

注意:**不再同步返回 AI 結果**。Desktop 拿到 jobId 後:

- WebSocket 模式:訂閱 `/api/work/ws/results`,收到推送即寫本地
- Polling 模式:每 30s 一次 `/api/work/recent-results?since={上次拉的 timestamp}`,結果以 `jobId` 對齊

### 4.2 管理頁面用(需 admin role,JWT 內帶角色 claim)

| Method  | Path                                                                    | 說明                                                       |
|---------|-------------------------------------------------------------------------|----------------------------------------------------------|
| `GET`   | `/api/admin/work/users?keyword=&page=&size=`                            | 分頁列出有採集資料 / 配置的使用者                                       |
| `GET`   | `/api/admin/work/users/{userId}/config`                                 | 查單一使用者配置                                                 |
| `PATCH` | `/api/admin/work/users/{userId}/config`                                 | 改配置(interval / workHours / enabled);返回新版,bumps updatedAt |
| `GET`   | `/api/admin/work/users/{userId}/records?since&until&category&page&size` | 查單一使用者紀錄                                                 |
| `GET`   | `/api/admin/work/users/{userId}/stats?period=day\|week\|month`          | 聚合統計(類別分布 / 總數 / 工時)                                     |

| `GET`   | `/api/admin/work/health`                                                | 佇列深度 / worker 數 / Qwen-VL 速率 / 錯誤率                       |

## 5. 500 並發後端架構

### 5.1 流量模型

| 場景                              | 公式                   | 結果           |
|---------------------------------|----------------------|--------------|
| 平均 QPS(intervalMinutes=5,500 人) | 500 / 300            | 1.67 req/s   |
| 平均 QPS(intervalMinutes=1)       | 500 / 60             | 8.3 req/s    |
| 峰值(若整點 / 整 5 分鐘對齊)              | ≤ 500 req 同秒到達       | 500 req/s 瞬間 |
| 每 req payload                   | jpeg 70% 品質,1080p 縮圖 | ~150-300 KB  |
| Qwen-VL 單次推理                    | DashScope 實測         | 1-3 s        |

**結論**:HTTP 入口承壓不難(.NET Kestrel 單實例輕鬆 1k+ QPS),**瓶頸在 LLM 上游**。

### 5.2 削峰:佇列 + Worker Pool

```
┌─────────────────────┐
│ Desktop POST jpeg   │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────────────────────┐
│ API Controller(同步)               │
│ - 驗 JWT                            │
│ - 校驗 payload(zod-like 等)       │
│ - 把 jpeg 寫到 blob 暫存(Redis    │
│   / 本地檔案,key=jobId)           │
│ - 元數據入隊(Redis Stream /       │
│   RabbitMQ / 簡化:.NET Channel<T>)│
│ - 返回 202 + jobId                  │
└──────────┬──────────────────────────┘
           │ enqueue 完即返回
           ▼
┌─────────────────────────────────────┐
│ Worker Pool(N=20-50 worker)        │
│ for each job:                       │
│   ├─ 從 blob 取 jpeg                │
│   ├─ 限流通過(token bucket)        │
│   ├─ call Qwen-VL → category       │
│   ├─ insert work_records            │
│   ├─ push 結果到使用者的 WS / SSE   │
│   └─ 刪 blob                        │
└─────────────────────────────────────┘
```

#### Phase 1(MVP,單實例,~50 人):

- 隊列用 **.NET `Channel<T>`** 內存佇列
- Blob 用本地 fs 暫存(`/tmp/work-collect/{jobId}.jpeg`)
- Worker = `IHostedService`,N=10 並發
- 結果回推 = polling(`/api/work/recent-results`)
- Rate limit = AsyncSemaphore 限制同時 in-flight Qwen-VL ≤ 5

#### Phase 2(~200 人,單實例壓力上升):

- 隊列改 **Redis Stream**(or RabbitMQ)
- Blob 用 Redis(設 5min TTL)
- 多 worker 從同一 stream 消費,consumer group
- 結果回推改 **SignalR / WebSocket**

#### Phase 3(500+ 人,多實例):

- API 服務水平擴展(K8s replicas),Nginx LB
- Worker 服務獨立部署,K8s replicas
- Qwen-VL **多 key 輪詢**(DashScope 單 key RPM 限 ~120-300),N 個 key → N 倍吞吐
- Postgres 取代 SQLite(SQLite 不支援並發寫)

### 5.3 桌面端側的削峰

**Desktop 自己分散 tick 時間**,避免整點對齊砸到 server:

```ts
// work-collector.ts 改造
const intervalMs = c.intervalMinutes * 60_000
// 啟動延遲一個隨機 jitter,避免 500 台 desktop 整點對齊
const jitterMs = Math.floor(Math.random() * intervalMs * 0.5)
setTimeout(() => {
    this.tick()  // 第一次
    this.timer = setInterval(() => void this.tick(), intervalMs)
}, jitterMs)
```

每台機器隨機在前 50% 區間開始 → 500 台分佈在 30s 內,單秒峰值降到 ~17 req/s,backend 完全 hold 得住。

### 5.4 服務端 schema(.NET / EF Core 視角)

```sql
-- 採集紀錄(server side,所有人共用)
CREATE TABLE work_records
(
    id                  BIGSERIAL PRIMARY KEY,
    user_id             VARCHAR(64) NOT NULL,
    captured_at         BIGINT      NOT NULL, -- Unix ms
    active_app          TEXT,
    active_window_title TEXT,
    category            VARCHAR(20) NOT NULL,
    description         TEXT        NOT NULL,
    confidence          REAL,
    screenshot_hash     VARCHAR(16),
    reason              TEXT,
    job_id              UUID UNIQUE,          -- 對齊 desktop polling 結果
    created_at          BIGINT      NOT NULL  -- server 寫入時間,跟 captured_at 區分(處理延遲可量化)
);
CREATE INDEX idx_work_records_user_captured ON work_records (user_id, captured_at DESC);
CREATE INDEX idx_work_records_category ON work_records (category);

-- 使用者配置(每人一行)
CREATE TABLE work_user_configs
(
    user_id          VARCHAR(64) PRIMARY KEY,
    enabled          BOOLEAN NOT NULL DEFAULT TRUE,
    interval_minutes INT     NOT NULL DEFAULT 5,
    work_start_hour  INT     NOT NULL DEFAULT 8,
    work_end_hour    INT     NOT NULL DEFAULT 17,
    updated_at       BIGINT  NOT NULL,
    updated_by       VARCHAR(64) -- 自己改 vs admin 改(審計)
);

-- 待處理任務(僅 Phase 1 用 DB 暫存;Phase 2 後遷到 Redis Stream)
CREATE TABLE work_analyze_queue
(
    job_id        UUID PRIMARY KEY,
    user_id       VARCHAR(64) NOT NULL,
    payload_path  TEXT        NOT NULL, -- jpeg 暫存路徑
    metadata      JSONB       NOT NULL, -- active_window / app_name / all_windows
    status        VARCHAR(20) NOT NULL, -- queued / processing / done / failed
    attempts      INT         NOT NULL DEFAULT 0,
    error_message TEXT,
    enqueued_at   BIGINT      NOT NULL,
    started_at    BIGINT,
    completed_at  BIGINT
);
CREATE INDEX idx_work_queue_status ON work_analyze_queue (status, enqueued_at);
```

### 5.5 失敗 / 重試策略

| 失敗點                               | 處理                                                                           |
|-----------------------------------|------------------------------------------------------------------------------|
| Qwen-VL 暫時失敗(429 / 5xx / timeout) | worker 內 retry 3 次 + 指數退避;仍失敗 → status=failed,desktop polling 拿到 failed 知道丟棄 |
| Qwen-VL 持續失敗(可用性 < 50%)           | health endpoint 暴露指標,触发 admin 告警                                             |
| Blob 暫存區域寫滿                       | 監控 disk,> 80% 拒收新 job(503)                                                   |
| 桌面端網路中斷                           | 本地 SQLite 仍記錄(透過 idle 路徑 / 暫存隊列),恢復後背景補傳                                     |
| 跨機(換電腦)同步                         | desktop 啟動拉 `/api/work/my-records?since=lastLocal`,補齊本地 SQLite               |

### 6.3 對齊策略(server vs local)

**寫路徑**:server 是 source of truth,desktop 收到 server push / polling 結果後寫本地。

**讀路徑**:

- UI 主要讀本地 SQLite(快、離線可用)
- 跨機 / 重灌情境:啟動時用 `/api/work/my-records?since=lastLocalMax` 補齊缺口

**衝突場景**:
| 場景 | 處理 |
|---|---|
| 同一 jobId 在 server / local 都有 | server 為準,local 用 UPSERT |
| local 有但 server 無(historically before centralization) | 保留 local,標記 `synced=false` 暫不對齊;後續若要遷移做一次性
upload |
| server 有但 local 無(換機) | polling / backfill 拉下來 INSERT |

## 7. 管理頁面設計

### 7.1 技術棧

- 獨立 Vue 3 工程,**不嵌進 desktop**
- 走 TMBOM 同套認證(共用 JWT,SSO)
- UI 庫:Element Plus(跟 desktop 主窗一致,降低學習成本)
- 部署:作為 TMBOM 的子路徑(`/admin/work-collect/*`)或獨立子域

### 7.2 主要頁面

```
管理頁面
├─ 使用者列表
│   - 列出有採集資料的人:工號 / 姓名 / 部門 / 上次採集時間 / 7天紀錄數 / 當前配置摘要
│   - 篩選:部門 / keyword / 配置鎖定狀態
│   - 點某人 → 詳情頁
│
├─ 使用者詳情
│   ├─ 配置編輯(interval / workHours / enabled / 重置按鈕)
│   ├─ 時間軸(復用 desktop 的圖表元件 — components/TimelineList.vue 等)
│   └─ 聚合統計(類別分布 / 工時統計 / 應用排行)
│
└─ 系統健康
    - 佇列深度 / Worker 數 / Qwen-VL 速率 / 失敗率
    - 近 24h 處理量趨勢
```

### 7.3 元件復用

desktop 的 `src/features/work-collect/components/*`(StatCards / TimelineList / CategoryDonut / DailyTrendLine /
WeeklyHeatmap 等)**直接搬到新工程**,只改:

- 資料源:從 `useWorkCollectStore` → admin store(走 HTTP /api/admin/)
- props.userId 注入,讓圖表知道在看誰

復用率預計 80%,新工程主要工作是 layout + 使用者列表 + 配置編輯。

## 8. 遷移路徑(分階段)

### Phase A(本月)— 後端基建 + 配置中心化

- [ ] DB schema:`work_records` / `work_user_configs` 落地(server 端)
- [ ] API:`GET /api/work/my-config` + `PATCH /api/admin/work/users/{id}/config`
- [ ] Desktop:`syncConfigFromServer()` + UI 隱藏 interval 編輯
- [ ] 既有 `/api/WorkCollect/analyze` 保留同步行為(暫不改隊列)— 落庫加上 `user_id` + `job_id`

**驗收**:管理員在 (未上線的) admin web 改一個人的 interval,desktop 10 分鐘內生效;後端開始累積 work_records。

### Phase B(下月)— 異步化 + 管理頁面

- [ ] API:`POST /analyze` 改為 202 + 入隊
- [ ] Worker pool(`.NET Channel<T>` + N=10)
- [ ] Desktop polling `/recent-results`
- [ ] 管理頁面 v1:使用者列表 + 詳情 + 配置編輯(復用 desktop 圖表元件)

**驗收**:50 人並發採集,後端佇列深度 < 100,管理員能看每個人時間軸。

### Phase C(後續)— 規模化

- [ ] 隊列改 Redis Stream
- [ ] 結果回推改 SignalR
- [ ] DB 改 Postgres
- [ ] Qwen-VL 多 key 輪詢
- [ ] 監控告警(Prometheus / Grafana)

**驗收**:500 人並發,峰值 P99 < 5s 拿到結果,可用性 > 99.5%。

## 9. 風險與緩解

| 風險                        | 緩解                                                                                       |
|---------------------------|------------------------------------------------------------------------------------------|
| Qwen-VL 限流 / 漲價           | 多 key 輪詢;備援模型(切到本地 vision LLM / OpenAI-compatible 兼容);worker 內 retry + 失敗降級為「unknown」分類  |
| 異步化後 UX 變差(原本即時看到結果)      | desktop UI 改成「採集已上傳,結果稍後出現」狀態,30s 內 polling 拉到一般無感;失敗也有 UI 提示                            |
| 跨機同步衝突                    | jobId UNIQUE 保證 UPSERT 冪等;本地 SQLite 保留 `synced` flag 處理歷史資料                              |
| 管理員濫改 interval(設成 1 秒)    | server config schema 校驗 `interval >= 1 minute`;hard 上限 60 分鐘                             |
| 截圖 jpeg 上傳到 server 引發隱私顧慮 | 跟現狀一致(原本就會傳給後端);server worker 處理完即刪 blob,**僅儲存文字結果**(category / description) — **不存原始圖** |
| Desktop 離線期間紀錄丟失          | idle 紀錄在本地補齊;非 idle 上傳失敗的 job 透過 desktop 端內存佇列重試(主進程崩潰丟,接受)                              |
| 既有資料怎麼辦                   | desktop 本地 SQLite 保留;若需要看歷史,可一次性 `POST /api/work/import-legacy` 批次上傳(可選,看實際需求)           |

## 11. 不做的事(明確劃線)

避免 scope 失控:

- ❌ **不上 Kafka / EventBridge / Pulsar** — Redis Stream 對 500 人量級夠用,引入大型 MQ 是 over-engineering
- ❌ **不存原始截圖** — 跟既有政策一致,server 只存 AI 文字結果
- ❌ **不做即時告警 / 異常檢測** — 管理員手動巡視;有需要再加
- ❌ **不做跨組織共享 / 排行榜** — 隱私敏感,不開放
- ❌ **不做手機 / 平板採集** — 只跟 desktop 對接
- ❌ **不重寫 desktop 的 dHash / idle 偵測** — 既有實作已穩,沿用
- ❌ **不換 i18n / 不換 UI 框架** — 管理頁面用 Element Plus 跟 desktop 統一
- ❌ **不引入單獨的 analytics 服務(Mixpanel / Amplitude)** — 自己的 work_records 已是分析資料源

## 12. 對外暴露的決策點(需確認再執行)

開工前需要跟相關方對齊:

1. **管理員是否能完全鎖定使用者本機開關?** 目前設計給管理員權限改 interval,但**使用者本機關掉採集(enabled=false)的權限保留
   ** — 這條一旦改成「管理員強制開啟,使用者不可關」會涉及合規 / 員工關係,要先確認
2. **server 落庫保留期** — work_records 永久?180 天?要 GDPR / 公司資料保留政策對齊
3. **管理頁面的部門 / 組織結構從哪拉** — 是直接走 TMBOM 既有 user 結構,還是另開人事系統介接
4. **Qwen-VL 用個人 key 還是企業 key** — 涉及計費 + 限流分配
5. **管理頁面的訪問權限** — 哪些角色能看(IT 主管 / 直屬主管 / HR / 老闆)?有沒有看「同部門 / 全公司」的維度

這些不在本文設計範圍,但啟動 Phase A 前要先有結論。
