# 24 - 每日晨報設計

> **狀態**:設計定稿,待實作。由現有「每日學習建議」(dailyAdvice)**升級取代** —— 學習建議無參考價值,退場;
> 晨報是**服務端組裝、XXL-JOB 定時、發到本人郵箱**的每日報表。
> 決策來源:與使用者 2026-07-17 的設計討論。

最後更新:2026-07-17

---

## 1. 定位與目標

**痛點**:代辦 / 各系統待辦(維修單、審批…)散在各處,人不主動看就忘;而且電腦沒開時桌面提醒完全失效。

**解法**:每天早上自動把「今天該關注的事」從各系統彙整成**一封晨報郵件,發到本人信箱** —— 不依賴電腦開著,打開郵箱就看到。

**一句話架構**:
```
XXL-JOB(每天 8:00,HTTP 任務)
        │ 只負責「敲一下」
        ▼
TMBOM  /api/report/run
   遍歷目標用戶 → 每人組裝各「欄目」數據 → Qwen 彙總 → 發本人郵箱
   (取數 / 彙總 / 發信 全在 TMBOM;調度不在 TMBOM)
```

## 2. 核心決策(已與使用者確認)

| 項 | 決策 |
|---|---|
| 調度 | 用公司現有 **XXL-JOB**(HTTP 任務類型),**不在 TMBOM 內置定時器** |
| 邏輯與取數 | **全在 TMBOM 後端**(XXL-JOB 只是觸發器) |
| 收件人 | **只發本人**(工號對應郵箱)—— 順帶消除「內部數據外發」風險 |
| 電腦沒開 | 也能收到(∵ 服務端跑、走郵件) |
| 配置分層 | **數據源/欄目 = 開發接入**(寫代碼,用戶碰不到);**「要哪些欄目」= 一期由管理員定死,以後再開放用戶訂閱** |
| 與學習建議 | **合併**:舊 dailyAdvice(桌面端生成 + 首頁展示)退場,晨報取代 |

## 3. 欄目機制(核心)—— 一個數據源 = 一個類,自動註冊

晨報 = 若干**欄目(Block)**拼成(今日代辦、待處理維修單、待審批…)。每個欄目是一段「取數 + 呈現」邏輯。

**契合 TMBOM 的程序集掃描 DI**:定義 `IReportBlock`,每接一個數據源就寫一個實現類,**自動被掃描收集,無需改註冊代碼**。「數據源註冊表」= 所有 `IReportBlock` 實現的集合。

```csharp
public interface IReportBlock
{
    string Key { get; }        // "todos" / "repair" / "approval"
    string Title { get; }      // "今日代辦"
    int Order { get; }         // 晨報內排序(小在前)
    // 回該用戶這塊的內容;null = 此人這塊沒東西 → 晨報跳過該欄目
    Task<ReportBlockResult?> BuildAsync(ReportContext ctx);
}

public sealed class ReportContext          // 每個用戶一份
{
    public string EmpNo { get; init; }     // 工號
    public string Email { get; init; }     // 收件郵箱
    public DateTime Today { get; init; }   // 生成日(調度傳入,避免跨零點歧義)
}

public sealed class ReportBlockResult
{
    public string Title { get; init; }
    public List<ReportItem> Items { get; init; } = new();
    public string? Summary { get; init; }  // 該欄目一句話(可為 AI 摘要輸入)
}

public sealed class ReportItem
{
    public string Text { get; init; }              // "招標材料 · 今天 17:00"
    public string? Tag { get; init; }              // "逾期" / "待審" …
    public int Urgency { get; init; }              // 0 普通 / 1 提醒 / 2 緊急(郵件配色用)
    public string? Link { get; init; }             // 可選:點進系統的深鏈
}
```

**新增一個公司數據 = 新增一個 `IReportBlock` 類**,互不影響。這就是「想加就加一塊」。

## 4. 觸發端點契約 + 安全

```
POST /api/report/run
Header: X-Report-Token: <約定密鑰>      # XLL-JOB 任務裡配上;缺失/不符 → 401
Body:   { "date": "2026-07-17",         # 可空,預設服務端當天
          "empNos": ["A123"] }          # 可空,空=全部目標用戶(見 §6)
Resp:   { "date": "...", "sent": 12, "failed": 1, "errors": [...] }
```

**安全**(它會群發郵件,必須防亂調):
- 約定密鑰 Header(最簡),或限內網 IP / 網關白名單。
- 冪等:同一 `date` 重複觸發 → 覆蓋當天、不重發(靠 §9 run-log 去重),避免 XXL-JOB 重試轟炸。
- 建議 `[AllowAnonymous]` + 自己校驗密鑰(對齊桌面端慣例),不要掛通用 JWT(XXL-JOB 沒登入態)。

## 5. AI 彙總(Qwen)

- 復用現有 Qwen(DashScope OpenAI-compatible)。
- 輸入:各欄目的 `Items` + `Summary`(結構化)。
- 產出:**開頭一句人話總結**(「早安 —— 今天 3 件待辦、1 件逾期,重點先處理招標材料」)+ 可選逐欄目潤色。
- **降級**:沒配 / 調用失敗 → 晨報照發,只是少那句 AI 總結(欄目數據原樣列出)。AI 是錦上添花,不阻塞發信。

## 6. 運行流程(per-user fan-out)

```
run(date, empNos):
  users = empNos ?? 取目標用戶清單(§9 訂閱表;一期=白名單/全員)
  for each user (可並行 + 限流):
      blocks = 所有 IReportBlock,按 Order
      results = blocks.Select(b => b.BuildAsync(ctx)).Where(非 null)
      if results 為空: 跳過該用戶(不發空晨報)
      html = 渲染(AI 總結 + results)
      EmailSend(user.Email, "每日晨報 · M月d日", html)
      寫 run-log
```
- **量級**:N 用戶 × M 欄目 次取數。並行需限流(避免壓垮被調系統),失敗單用戶隔離、不影響他人。

## 7. 郵件

- 復用 `ichia.Infrastructure/Common/EmailSend.cs`(報修工單已用它發 HTML 郵件)。
- HTML body 由 TMBOM 渲染:標題欄 + AI 總結條 + 各欄目卡(緊急度配色)。樣式對齊桌面 dock 卡片觀感。
- 主旨:`每日晨報 · 7月17日 週四`。

## 8. 配置分層(回答「開放用戶 vs 自己配」)

| 層 | 誰定 | 一期做法 |
|---|---|---|
| **欄目本身**(怎麼取維修單、連哪個庫) | 開發 | 寫 `IReportBlock` 類,自動註冊 |
| **晨報包含哪些欄目、順序** | 一期管理員定死 | 靠 `IReportBlock.Order` + 一份啟用清單(config / 常量) |
| **收件人範圍** | 一期定死 | 白名單 → 驗證後轉全員 |
| **用戶訂閱**(我要哪幾塊、幾點發) | P3 才開放 | 加 `report_subscription` 表 + 桌面設置頁 |

**原則**:數據源永遠開發接;「選哪些」先定死,跑通看效果,再談開放用戶。

## 9. 數據模型(後端 SQL Server)

一期最小:
- `report_run_log`(可選但建議):`id, empNo, date, status, blockKeys, error, sentAt` —— 用於冪等去重 + 排查。

P3 開放訂閱時再加:
- `report_subscription`:`empNo, blockKeys(JSON), sendTime, enabled`。

一期若想更輕,可先不落庫(純跑+發),但**冪等**建議至少留 run-log。

## 10. 桌面端連帶

- **刪**:桌面 `dailyAdvice` 那套(scheduler 08:00 生成 + 首頁「今日學習建議」區塊 + 相關 handler / bridge / 類型)——
  學習建議無價值,隨晨報退場。
- **首頁**(可選,P3):改成拉「我最近一封晨報」展示(唯讀),與郵件同源。一期可先不動首頁。

## 11. 分期落地

1. **P1 · 管線跑通(不開放配置)**
   - `/api/report/run` 端點 + 密鑰保護;`IReportBlock` 機制 + 掃描收集。
   - 先寫 **2 個現成欄目**:`todos`(需先把代辦上雲 or 先用一個佔位/後端已有數據)、`workReview`(昨日工作回顧)。
     > ⚠️ 注意:桌面代辦目前**本地 SQLite、不在後端**。晨報在服務端拿不到本地代辦 ——
     > P1 的 `todos` 欄目要麼(a)先接一個後端已有的數據當首發欄目,要麼(b)代辦補一條「上雲」同步。**這是 P1 要先定的**(見 §13)。
   - Qwen 總結 + `EmailSend` 發**白名單本人**;XXL-JOB 配 HTTP 任務。
   - 產出:收到第一封真實晨報。
2. **P2 · 接真公司數據欄目**:按使用者定的來(維修單 / 審批 / …),各寫一個 `IReportBlock`;收件轉全員。
3. **P3 · 開放用戶訂閱**:`report_subscription` + 桌面設置頁(勾欄目、設時間);首頁展示最近一封。

## 12. 為什麼這樣設計(要點回顧)

- XXL-JOB 只當「鬧鐘」→ 調度與業務解耦,不動 TMBOM 調度;邏輯集中在 TMBOM 一處好維護。
- 欄目走 `IReportBlock` + 程序集掃描 → 數據集成「加一塊算一塊」,天然可擴展,契合現有 DI 習慣。
- 只發本人 → 無外發風險;服務端跑 → 不依賴電腦。
- 配置分層 + 分期 → 先把「一封有用的晨報」跑出來,再談用戶自助。

## 13. 待定 / 風險(P1 前要定)

1. **`todos` 欄目數據從哪來**:代辦目前本地不在後端。首發欄目是先接「後端已有數據」(繞開此問題),還是給代辦補一條「上雲同步」?**建議 P1 先用一個後端現成數據當首發欄目**,代辦上雲另議。
2. **第一個真實公司數據欄目挑哪個**(P2 輸入):維修單積壓?待審批?—— 需使用者指定。
3. **目標用戶清單**從哪取(全員名冊 / HR / AD)、如何拿到每人郵箱(工號→郵箱映射)。
4. **XXL-JOB 執行環境**:確認可用「HTTP 任務」類型直接調 TMBOM 端點(多數版本支持;若只有 GLUE/執行器模式,需另議承接方式)。
5. **並發/限流**:全員 fan-out 對被調系統的壓力,需設並行上限 + 超時。
