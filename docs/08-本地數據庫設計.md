>
> 註:下文「本期」等字樣是最初日誌系統增強期的措辭,DB 基建後續已擴展到 work_records / user_profiles / agent_configs / todos 等多個 feature,不再逐一改寫歷史段落。

## 本期已定案的 5 項關鍵決議

> 寫在最前面,後面各章節已按此落實。

1. **渲染端 forward 範圍** —— `src/utils/logger.ts` 內的 `forwardToMain` 從「只 ERROR」改為「所有等級」forward 到主進程;主進程持久化時
   **debug 不落庫**,DB 僅存 info / warn / error(讓排查能看 error 前後的 info 上下文)。詳見 §12.2。
2. **DB 保留天數 = 14** —— 跟現有 `log-file-writer` 的 `RETENTION_DAYS` 對齊;啟動時 `cleanupOlderThan(14)`。詳見 §14.1。
3. **DB 檔路徑** —— dev 在**專案根**(`<projectRoot>/app.db`),prod 在 **`<userData>/app.db`**。跟現有 txt 檔的 dev/prod 路徑分流規則完全一致。詳見 §9。
4. **DB init 失敗 → 靜默降級** —— 只 `console.error` 報出,**不 `attachLogService`**;主進程 logger 內 `_logService?.write(...)` 自動 noop;txt 寫入仍正常;App 照常啟動。詳見 §17.1。
5. **本期不加任何渲染端可呼叫的 DB IPC,不加 UI** —— 寫入端沿用既有 `LOG_WRITE`,查詢端不曝露;`LogService.query()` 仍實作但只在主進程內部可用,留給未來 dev-only 面板。詳見 §13、§18。

## 1. 本期範圍與目標

### 1.1 要做

- 引入 `better-sqlite3 + drizzle-orm + drizzle-kit` 基礎建設
- 設計 `logs` 表 schema,主 / 渲染兩端的日誌**同時**寫進 DB
- 改造主進程 logger:寫 txt 的同時也寫 DB
- 改造渲染端入口(log handler):收 LOG_WRITE 時同時寫 txt 與 DB
- 自動清理:啟動時刪除 N 天前的舊紀錄
- 為後續其他表 / 模組保留延伸位置(DatabaseManager 設計成可擴的)

### 1.2 不做

- **任何使用者面向的查詢 UI**(SQLite 是內部排查資源,不曝露給終端使用者)
- 其他資料表(通知、最近紀錄、審計等),全部留給後續
- 改變 logger 對 caller 的 API 簽名

### 1.3 成功標準

- 任何呼叫 `logger.*` 的地方,**業務代碼零修改**仍照舊運作
- 寫入 txt 的行為跟原本完全一致(同樣的格式、同樣的檔案命名、同樣只 ERROR 落地)
- **info / warn / error 同時進 SQLite 的 `logs` 表**,可日後查詢(**debug 不落庫**,只走 console;高頻 debug
  進庫會淹沒排查重點 — 後續調整,見 §12.1)
- 啟動時 DB 失敗 → txt 仍正常寫入,App 正常啟動,DB 寫入端口靜默 noop
- DB 檔自動清理 N 天前的舊紀錄

1. **目前實際只有 ERROR 等級會落到 txt 檔**。其他等級只在 console 出現,關掉 App 就沒了。
2. **節流 100/秒由 ipc handler 實作**。本期 LogService 寫入端如果也要節流要評估,目前傾向**不重複加節流**(複用 handler 既有的)。

## 3. 雙寫策略

### 3.1 為什麼雙寫

| 端 | 角色 | 為什麼留 |
|---|---|---|
| **txt 檔** | 給支援人員 / IT 拿走的「物理工件」 | 就算 DB 損壞,txt 也在;ZIP 壓給支援打包簡單;熟悉的工作流程不破壞 |
| **SQLite** | 開發 / 排查時用結構化查詢 | 跨日跨進程關聯查詢、按 level / module 過濾、按關鍵字搜尋,純文字做不到 |

兩者**獨立寫入**:其中一端失敗不影響另一端。

### 3.2 兩端寫入規則

| 等級 | txt 檔 | SQLite |
|---|---|---|
| debug | ✗(維持現狀) | ✓ |
| info  | ✗(維持現狀) | ✓ |
| warn  | ✗(維持現狀) | ✓ |
| error | ✓(維持現狀) | ✓ |

理由:

- **txt 維持現狀只 ERROR**:支援人員從使用者拿到 txt 時,只關心錯誤;塞 debug/info 進去反而混淆
- **DB 寫 info / warn / error(debug 除外)**:儲存便宜、索引讓查詢便宜、排查時看 error 前後的 info 上下文;debug 高頻噪音不落庫,需要時看
  dev console

### 3.3 寫入時機

- 主進程 `logger.*` 內呼叫:**同步** `logService.write(...)` + 原本邏輯(只 ERROR 寫 txt)
- 渲染端透過 `LOG_WRITE` IPC 進來,handler 內:**同步** `logService.write(...)` + 原本邏輯(只 ERROR 寫 txt)

兩端走同一個 `LogService.write()` 入口,只是 `source` 欄位不同。

## 7. logs 表 Schema 設計

### 7.1 欄位

| 欄位 | 型別 | 必填 | 說明 |
|---|---|---|---|
| `id` | INTEGER PK AUTOINCREMENT | ✓ | — |
| `createdAt` | INTEGER(Unix ms) | ✓ | 查詢主軸,索引 |
| `level` | TEXT(`debug` / `info` / `warn` / `error`) | ✓ | 跟現有 logger 對齊,小寫 |
| `source` | TEXT(`main` / `renderer`) | ✓ | 來源進程 |
| `module` | TEXT | optional | logger 第 2 參數 |
| `message` | TEXT | ✓ | 主訊息 |
| `args` | TEXT(JSON) | optional | 第 3+ 參數 `JSON.stringify` |
| `errorStack` | TEXT | optional | args 含 Error 時抽出 stack,便於搜尋 |

### 7.2 索引

- `idx_logs_createdAt`:列「最近 N 筆」用
- `idx_logs_level_createdAt`:按 level 過濾用

`module` 索引等實際看使用模式再加。

### 7.3 為什麼不全部塞欄位(例如把每個 arg 展開)

- args 是不定型,塞 JSON 最通用
- `errorStack` 例外拉出來,因為**「找 stack 含 X」**是高頻排查動作,做不到 JSON 內檢索

## 8. Drizzle Schema 寫法

`electron/main/db/schema/logs.ts`:

```ts
import {sqliteTable, integer, text, index} from 'drizzle-orm/sqlite-core'

export type LogLevel = 'debug' | 'info' | 'warn' | 'error'
export type LogSource = 'main' | 'renderer'

export const logs = sqliteTable(
    'logs',
    {
      id: integer('id').primaryKey({autoIncrement: true}),
      createdAt: integer('createdAt').notNull(),
      level: text('level').$type<LogLevel>().notNull(),
      source: text('source').$type<LogSource>().notNull(),
      module: text('module'),
      message: text('message').notNull(),
      args: text('args'),
      errorStack: text('errorStack')
    },
    (table) => ({
      idxCreatedAt: index('idx_logs_createdAt').on(table.createdAt),
      idxLevelCreated: index('idx_logs_level_createdAt')
          .on(table.level, table.createdAt)
    })
)

export type LogRow = typeof logs.$inferSelect
export type NewLog = typeof logs.$inferInsert
```

## 9. DatabaseManager 主進程單例

`electron/main/db/database-manager.ts`:

```ts
import Database from 'better-sqlite3'
import {drizzle, BetterSQLite3Database} from 'drizzle-orm/better-sqlite3'
import {migrate} from 'drizzle-orm/better-sqlite3/migrator'
import {app} from 'electron'
import path from 'path'
import * as schema from './schema'

export class DatabaseManager {
  private sqlite: Database.Database | null = null
  private db: BetterSQLite3Database<typeof schema> | null = null

  /**
   * 啟動順序見 §16。
   * 內部不呼叫 logger(避免循環依賴 —— logger 反過來要呼叫本類);
   * 失敗用原生 console.error 報出後 rethrow,由 index.ts 決定降級。
   */
  init(): void {
    // dev 時跟 txt 一致,放專案根 db/;prod 放 userData
    const baseDir = app.isPackaged ? app.getPath('userData') : app.getAppPath()
    const dbPath = path.join(baseDir, 'app.db')
    const migrationsFolder = path.join(__dirname, 'migrations')

    this.sqlite = new Database(dbPath)
    this.sqlite.pragma('journal_mode = WAL')
    this.sqlite.pragma('foreign_keys = ON')

    this.db = drizzle(this.sqlite, {schema})
    migrate(this.db, {migrationsFolder})
  }

  getDb(): BetterSQLite3Database<typeof schema> {
    if (!this.db) throw new Error('DatabaseManager.init() 尚未呼叫')
    return this.db
  }

  isReady(): boolean { return this.db !== null }

  close(): void {
    this.sqlite?.close()
    this.sqlite = null
    this.db = null
  }
}
```

**為什麼 dev 跟 prod 路徑不同**(本期決議 #3):跟現有 `log-file-writer` 行為對齊 —— dev 在專案根(`<projectRoot>/app.db`),直接看資料、清資料方便;prod 在 `<userData>/app.db`,規範安裝路徑。

## 11. LogService 設計

`electron/main/db/services/log.service.ts`:

```ts
import {lt, desc, type SQL, and, eq, gte, like, inArray} from 'drizzle-orm'
import type {DatabaseManager} from '../database-manager'
import {logs, type LogRow, type LogLevel, type LogSource} from '../schema/logs'

export interface LogQueryParams {
  level?: LogLevel | LogLevel[]
  source?: LogSource
  module?: string
  since?: number
  until?: number
  search?: string
  limit?: number
  offset?: number
}

export class LogService {
  constructor(private readonly dbManager: DatabaseManager) {}
  private get db() { return this.dbManager.getDb() }

  /**
   * 寫一筆。logger / handler 直接呼叫。
   * 內部 try / catch:寫失敗只報 console,絕不擴散(避免日誌失敗炸業務)。
   * dbManager 未 ready 時 noop —— 啟動順序保險(極早期日誌可能落空,可接受)。
   */
  write(entry: {
    level: LogLevel
    source: LogSource
    module?: string
    message: string
    args?: unknown[]
  }): void {
    if (!this.dbManager.isReady()) return
    try {
      const {argsJson, errorStack} = serializeArgs(entry.args)
      this.db.insert(logs).values({
        createdAt: Date.now(),
        level: entry.level,
        source: entry.source,
        module: entry.module ?? null,
        message: entry.message,
        args: argsJson,
        errorStack
      }).run()
    } catch (err) {
      console.error('[LogService] DB 寫入失敗', err)
    }
  }

  /** 預留:給未來 dev-only 查詢面板使用 */
  query(params: LogQueryParams): LogRow[] {
    const conds: SQL[] = []
    if (params.level) {
      conds.push(Array.isArray(params.level)
          ? inArray(logs.level, params.level)
          : eq(logs.level, params.level))
    }
    if (params.source) conds.push(eq(logs.source, params.source))
    if (params.module) conds.push(eq(logs.module, params.module))
    if (params.since != null) conds.push(gte(logs.createdAt, params.since))
    if (params.until != null) conds.push(lt(logs.createdAt, params.until))
    if (params.search) conds.push(like(logs.message, `%${params.search}%`))

    return this.db.select().from(logs)
        .where(conds.length ? and(...conds) : undefined)
        .orderBy(desc(logs.createdAt))
        .limit(params.limit ?? 200)
        .offset(params.offset ?? 0)
        .all()
  }

  /** 啟動時清理 N 天前舊紀錄 */
  cleanupOlderThan(days: number): number {
    if (!this.dbManager.isReady()) return 0
    const cutoff = Date.now() - days * 24 * 60 * 60 * 1000
    const result = this.db.delete(logs).where(lt(logs.createdAt, cutoff)).run()
    return result.changes
  }
}

function serializeArgs(args?: unknown[]): {argsJson: string | null; errorStack: string | null} {
  if (!args || args.length === 0) return {argsJson: null, errorStack: null}

  let errorStack: string | null = null
  const serialized = args.map((a) => {
    // 渲染端來的 Error 已經被 sanitizeArg 轉成 {__error__, name, message, stack}
    if (a instanceof Error) {
      errorStack = errorStack ?? a.stack ?? null
      return {name: a.name, message: a.message, stack: a.stack}
    }
    if (a && typeof a === 'object' && (a as {__error__?: boolean}).__error__) {
      const e = a as {stack?: string}
      errorStack = errorStack ?? e.stack ?? null
    }
    return a
  })
  return {argsJson: JSON.stringify(serialized), errorStack}
}
```

**注意 `query` / `cleanupOlderThan` 是預留 API**,本期本期不曝露給 renderer,只在主進程內部用(cleanup 由啟動鉤子呼叫)。未來要做 dev-only 面板時再加 IPC channel。

## 12. 整合現有 logger.ts(主進程 + 渲染)

### 12.1 主進程 logger.ts 改造重點

**目標**:info / warn / error 呼叫 `logService.write(...)` 落庫,**同時**保留寫 txt 的邏輯(只 ERROR)。
**debug 不落庫** —— 純 dev console 噪音,持久化只會讓排查時撈不到重點;要追高頻流程看 dev terminal。

```ts
// electron/main/utils/logger.ts
import {writeLine} from './log-file-writer'
import type {LogService} from '../db/services/log.service'

let _logService: LogService | null = null

/** index.ts 在 DatabaseManager.init() 後呼叫,把 service 注入。失敗時不呼叫 → DB 寫入靜默 noop */
export function attachLogService(svc: LogService): void {
  _logService = svc
}

export const logger = {
  debug(message: string, module?: string, ...args: unknown[]): void {
    // 只走 console,不落庫(避免高頻 debug 淹沒 DB)
    if (process.env.NODE_ENV !== 'production') {
      console.debug(formatLine('DEBUG', message, module, args))
    }
  },
  info(message: string, module?: string, ...args: unknown[]): void {
    console.info(formatLine('INFO', message, module, args))
    _logService?.write({level: 'info', source: 'main', module, message, args})
  },
  warn(message: string, module?: string, ...args: unknown[]): void {
    console.warn(formatLine('WARN', message, module, args))
    _logService?.write({level: 'warn', source: 'main', module, message, args})
  },
  error(message: string, module?: string, ...args: unknown[]): void {
    const line = formatLine('ERROR', message, module, args)
    console.error(line)
    writeLine('main', line)                                       // ← 既有,不動
    _logService?.write({level: 'error', source: 'main', module, message, args})
  }
}

// writeRendererLog 同樣加一行 _logService?.write({source:'renderer', ...})
```

### 12.2 渲染端 logger.ts(`src/utils/logger.ts`)

**對外 API 完全不變**。內部仍走 `window.electronAPI.log.write(...)`,IPC payload 形狀也不變。**渲染端業務代碼零修改**。

**本期決議:`forwardToMain` 從「只 ERROR」改為「所有等級」**(對應本期決議 #1)。

理由:

- 排查問題時最缺的就是「錯誤前的渲染端上下文」(error 前面的 info / debug),DB 拿到完整時間線才有價值
- 100/秒節流(由 [log.handlers.ts](../electron/main/ipc-handlers/log.handlers.ts) 既有實作)足以擋住失控的流量,不需要動
- 影響面只在 `src/utils/logger.ts` 內部一行邏輯,業務 caller 不感知

對應修改點:`src/utils/logger.ts` 的 `debug` / `info` / `warn` 三個方法,在原本只 `console.*` 之外都加上 `forwardToMain(...)`。`error` 維持現狀(已經 forward)。

### 12.3 渲染端 handler 改造

`electron/main/ipc-handlers/log.handlers.ts` 內:

```ts
ipcMain.on(IpcChannels.LOG_WRITE, (_event, entry) => {
  // ...既有節流 + writeRendererLog (只 ERROR 寫 txt) 完全保留
  writeRendererLog(entry.level ?? 'INFO', entry.message, entry.module, entry.args)

  // 新增:同步寫 DB
  logService.write({
    level: entry.level.toLowerCase() as LogLevel,    // renderer 用大寫,DB 統一小寫
    source: 'renderer',
    module: entry.module,
    message: entry.message,
    args: entry.args
  })
})
```

## 16. 啟動順序

`electron/main/index.ts` 修改要點:

```ts
let dbManager: DatabaseManager
let logService: LogService

app.whenReady().then(async () => {
  // ── 既有 ──
  initLogFileWriter()
  app.setAppUserModelId('com.ichia.desktop.client')
  ensureAutoLaunchRegistered()

  // ── 新增:DatabaseManager + LogService ──
  // 包 try/catch,失敗只報 console,let App 繼續啟動(txt 仍正常)
  try {
    dbManager = new DatabaseManager()
    dbManager.init()
    logService = new LogService(dbManager)
    attachLogService(logService)             // 主進程 logger 注入
    logService.cleanupOlderThan(14)
  } catch (err) {
    console.error('[App] DB 初始化失敗,日誌只走 txt', err)
  }

  // ── 既有 ──
  configManager = new ConfigManager()
  await configManager.load()
  // ...

  // 註冊 handler 時傳入 logService(若 null 則 handler 內 noop)
  registerAllHandlers(windowManager, configManager, floatingBallMgr, updateMgr, logService)

  // ...
})

function gracefulShutdown(): void {
  // ...既有
  dbManager?.close()    // 確保 WAL 落盤
  // ...
}
```

**順序重點**:

1. `initLogFileWriter()` 必須最先,因為下面如果 DB init 失敗會用 console.error,但即便 console 不到位 file writer 不依賴 logger
2. DatabaseManager.init() **必須在 logger 任何寫入之前**完成,否則早期日誌寫不進 DB(可接受 —— 早期失敗本來就罕見,且 txt 仍會寫到)
3. logService 注入 logger 後,後續所有主進程 logger 呼叫都會雙寫

## 17. 錯誤處理與降級

### 17.1 DB init 失敗(本期決議 #4)

`dbManager.init()` 拋例外 → index.ts catch 住 → **不 `attachLogService`** → 主進程 logger 內 `_logService?.write(...)` 自動 noop → **txt 寫入仍正常**。

App **照常啟動**,使用者完全不感知。只有開發者在主進程 console 能看到 `[App] DB 初始化失敗,日誌只走 txt` 的錯誤輸出。

### 17.2 寫入失敗

LogService.write 內 try/catch,失敗只 console.error。**logger 呼叫永遠不拋**,業務代碼可以放心使用。

### 17.3 DB 損毀

`migrate()` 失敗會拋,落到 17.1 路徑。使用者完全不感知,只在主進程 console 可見錯誤。

緊急時手動操作:刪 `<userData>/app.db` + `app.db-wal` + `app.db-shm`,下次啟動自動重建。

## 2. 3 個關鍵概念

如果腦中只能記三件事,記這些:

### 概念 ① — 「全等級進 DB,只 ERROR 進 txt」

| 等級 | DB(SQLite) | txt 檔 |
|---|---|---|
| debug | ✓ | ✗ |
| info  | ✓ | ✗ |
| warn  | ✓ | ✗ |
| error | ✓ | ✓ |

**為什麼**:txt 是給客戶 / 支援人員拿走排查用的「物證」,只放重要的;DB 是給開發者後續查詢用的「全紀錄」,什麼都進。

### 概念 ② — 「DB 只活在主進程」

渲染進程**不認識** better-sqlite3、不會 import drizzle。任何寫日誌都透過 IPC 轉發到主進程,主進程才真正動手寫檔。

**為什麼**:Electron 渲染進程是 sandbox + contextIsolation 環境,不能直接 require native module。主進程當「DB 管家」,渲染進程當「客戶」,中間隔一道 IPC 牆。

### 概念 ③ — 「DB 失敗時 App 不掛」

若 DB 開不起來(權限問題、磁碟壞了),App **照常啟動**,只是 SQLite 那一支不寫東西;txt 跟 console 仍正常運作。

**為什麼**:日誌是輔助,不是業務核心。日誌系統掛掉不能拖累整個 App。

## 3. 檔案一個一個過

按重要性順序,**全部 6 個檔加 1 個 plugin**。

### 3.1 `electron/main/db/schema/logs.ts`

**幹什麼**:用 drizzle 的 TS DSL **聲明 logs 表長什麼樣**。一張表 8 個欄位 + 2 個索引。

**為什麼存在**:這個檔是「**唯一真相來源**」——
- drizzle runtime 看它知道有哪些表能查
- drizzle-kit (CLI) 看它差異產 migration SQL
- 整個 app 的 TypeScript 型別也從它推導(`LogRow` / `NewLog`)

改 schema 永遠改這個檔,**不要手動寫 SQL**。

**怎麼讀**:

```ts
export const logs = sqliteTable('logs', {
  id: integer('id').primaryKey({autoIncrement: true}),
  createdAt: integer('createdAt').notNull(),
  level: text('level').$type<LogLevel>().notNull(),
  // ...
})
```

每一行對應 SQL 一個欄位定義。

**怎麼用 import 的型別**:

```ts
import {logs, type LogRow, type NewLog, type LogLevel} from './schema/logs'

// LogRow = SELECT 出來的一行(所有欄位都齊)
// NewLog = INSERT 要塞的物件(可選欄位是 optional)
// LogLevel = 'debug' | 'info' | 'warn' | 'error'
```

### 3.2 `electron/main/db/schema/index.ts`

**幹什麼**:1 行轉發,把 logs 表 re-export 出去。

```ts
export * from './logs'
```

**為什麼存在**:讓 `import * as schema from './schema'` 一次拿到所有表(目前只一張,未來加表時不用改 import)。drizzle runtime 跟 drizzle-kit 都認這個入口。

### 3.3 `electron/main/db/database-manager.ts`

**幹什麼**:**SQLite 連線單例**。整個 App 只開一個 DB 連線,所有人共用。

**3 個關鍵方法**:

| 方法 | 做什麼 | 誰呼叫 |
|---|---|---|
| `init()` | 開 DB 檔、設 WAL pragma、跑 migration | `electron/main/index.ts` 啟動時 |
| `getDb()` | 拿 drizzle instance 操作 DB | LogService 內部 |
| `close()` | 關閉連線,把 WAL 內容 commit 進主檔 | App 退出前 (gracefulShutdown) |

**檔案位置邏輯**:

```ts
// init() 內:
const dbPath = app.isPackaged
  ? path.join(app.getPath('userData'), 'app.db')      // prod: %APPDATA%\ichiaDesktop\app.db
  : path.join(app.getAppPath(), 'app.db')              // dev: <專案根>\app.db
```

dev / prod 路徑分流,跟 [log-file-writer.ts](../electron/main/utils/log-file-writer.ts) 行為對齊。

### 3.4 `electron/main/db/services/log.service.ts`

**幹什麼**:**對 logs 表的所有業務 API**。包一層在 drizzle 上面,讓 caller 不用看到 drizzle 細節。

**3 個對外方法**:

| 方法 | 做什麼 |
|---|---|
| `write(entry)` | 寫一筆日誌(主 / 渲染 logger 都呼叫這個) |
| `query(params)` | 預留給未來查詢面板,本期不用 |
| `cleanupOlderThan(days)` | 啟動時呼叫一次,刪掉 N 天前的舊紀錄 |

**容錯保證**:`write()` 內部 try/catch,寫失敗只 `console.error`,**不會拋例外**。意思是 logger 呼叫永遠成功,業務代碼可以放心。

**args 怎麼存**:
- 如果 args 含 Error 物件 → 自動把 `stack` 抽到單獨的 `errorStack` 欄位
- 其他物件 → JSON.stringify 整包塞 `args` 欄位

### 3.5 `electron/main/db/migrations/0000_faithful_sunspot.sql`

**幹什麼**:drizzle-kit 第一次跑 `npm run db:generate` 自動產出的 SQL 檔。內容就是 `CREATE TABLE logs (...)` + 兩個 `CREATE INDEX`。

**為什麼存在**:App 第一次在使用者電腦啟動時,migrate() 跑這個 SQL **建出 `logs` 表**。

**鐵律**:**這個檔 commit 進 git 後就不要再改**。改了會破壞所有已部署使用者的升級流程(hash 對不上)。要改 schema → 加新表 / 改欄位 → 跑 `db:generate` 產出 `0001_xxx.sql`,**永遠用「加新檔案」處理變更**。

### 3.6 `electron/main/db/migrations/meta/`

**幹什麼**:drizzle-kit 內部用的 metadata。

- `_journal.json` — 列出所有 migration 的順序 + hash
- `0000_snapshot.json` — 第 0000 次 migration 後 schema 的快照

**你不需要動這個資料夾**。每次跑 `db:generate` 會自動更新。

### 3.7 `drizzle.config.ts`(專案根)

**幹什麼**:給 drizzle-kit CLI 看的設定檔。告訴它:
- schema 在哪(`./electron/main/db/schema/index.ts`)
- migrations 放哪(`./electron/main/db/migrations`)
- DB 方言是 sqlite

**只在 dev 用**,runtime 不會載入這個檔。`npm run db:generate` 跟 `npm run db:check` 都吃它。

### 3.8 `electron.vite.config.ts` 內的 `copyMigrationsPlugin`

**幹什麼**:電腦打包時把 `electron/main/db/migrations/` 整個資料夾**複製**到 `out/main/migrations/`。

**為什麼必要**:electron-vite 預設只處理 `.ts` / `.js`,SQL 是純文字它會忽略。但 runtime migrate() 要讀 SQL 檔,必須讓 SQL 檔出現在打包後的位置。

**為什麼自寫不用 `vite-plugin-static-copy`**:第三方套件的 glob 行為會把來源目錄結構也帶過去,變成 `out/main/migrations/electron/main/db/migrations/0000_xxx.sql` 這種四層巢狀。自寫 13 行 `cpSync` 直接攤平,維護成本反而更低。

## 5. 啟動與退出順序

### 啟動(`electron/main/index.ts` 內 `app.whenReady().then(async () => {...})` 區塊)

```
1. initLogFileWriter()              ← txt 寫入器先起來
2. setAppUserModelId(...)
3. ensureAutoLaunchRegistered()
4. ┌─ try {
   │   dbManager = new DatabaseManager()
   │   dbManager.init()              ← 開 DB + 跑 migration
   │   logService = new LogService(dbManager)
   │   attachLogService(logService)  ← 把 LogService 注入 logger
   │   logService.cleanupOlderThan(14)
   │ } catch (err) {
   │   console.error('DB 初始化失敗')  ← 失敗只 console 報,不擴散
   │   dbManager = null
   │ }
5. configManager = ...
6. windowManager = ...
7. ...
```

**第 4 步包在 try/catch**,失敗時 `attachLogService` 不會被呼叫 → logger 內 `_logService?.write(...)` 自動 noop → DB 寫入靜默失效,但 txt 跟 console 仍正常 → App 照常往下走。

### 退出(`gracefulShutdown()` 內)

```
1. windowManager.setQuitting(true)
2. trayManager.destroy()
3. floatingBallMgr.dispose()
4. updateMgr.dispose()
5. dbManager?.close()              ← 關 DB,WAL checkpoint 進主檔
6. windowManager.destroyAll()
```

**第 5 步在主視窗銷毀前**,給 SQLite 機會把記憶體裡未落地的內容寫回主檔,避免下次啟動讀到舊資料。

## 6. Drizzle ORM 速通

只列你目前代碼會用到的:

```ts
import {logs} from '@main/db/schema/logs'
import {eq, desc, lt, sql} from 'drizzle-orm'

const db = dbManager.getDb()

// SELECT * FROM logs WHERE level='error' ORDER BY createdAt DESC LIMIT 10
db.select().from(logs)
  .where(eq(logs.level, 'error'))
  .orderBy(desc(logs.createdAt))
  .limit(10)
  .all()

// INSERT INTO logs (...) VALUES (...)
db.insert(logs).values({
  createdAt: Date.now(),
  level: 'info',
  source: 'main',
  message: 'hi'
}).run()

// DELETE FROM logs WHERE createdAt < ?
db.delete(logs).where(lt(logs.createdAt, cutoff)).run()

// 跳過 ORM 直接跑 raw SQL(極端情況用)
db.run(sql`VACUUM`)
```

**`.all()` vs `.get()` vs `.run()` 差別**:
- `.all()` — SELECT,返回陣列(0 或多筆)
- `.get()` — SELECT,只取第一筆(或 undefined)
- `.run()` — INSERT / UPDATE / DELETE,返回 `{changes, lastInsertRowid}`

## 7. Migration 機制怎麼確保不重跑

每次 App 啟動跑 `migrate()`,做 4 步:

```
1. 列出 migrations/*.sql 檔(目前: 0000_faithful_sunspot.sql)
2. 對每份 SQL 算內容 hash(SHA-256)
3. 查 __drizzle_migrations 表已套用過哪些 hash
4. 跳過已套用的,只跑沒紀錄的;跑完插一筆紀錄
```

`__drizzle_migrations` 是 drizzle 啟動時自動建的系統表,**你完全不用碰它**。

### 場景對照表

| 場景 | DB 怎麼變 |
|---|---|
| 全新使用者第一次裝 | `__drizzle_migrations` 不存在 → 從 0000 全跑 |
| 你發新版含 0001 的 migration | 跳過 0000,只跑 0001 |
| 使用者卸載重裝(`deleteAppDataOnUninstall: false`) | DB 還在 → 0000 / 0001 都跳過,不重跑 |
| 使用者手動刪 `app.db` | 等同「全新裝」,從 0000 全跑 |
| App crash 寫一半 | 每份 migration 包 transaction,沒成功的不算數,下次重試 |

**鐵律**:已 commit 的 migration **永遠不要改**。要變更 schema 永遠加新檔。

## 10. FAQ

**Q1:渲染進程能不能直接讀 DB?**
不能,也不該。Electron sandbox + contextIsolation 設計就是不讓渲染進程碰 native module。所有讀寫都走 IPC。

**Q2:寫日誌會卡 UI 嗎?**
不會。
- 寫 txt 是 `appendFile` 異步
- 寫 DB 雖然 better-sqlite3 是同步 API,但只在主進程跑,主進程沒有 UI,不影響渲染。單條 INSERT 微秒級。

**Q3:DB 損毀會有什麼後果?**
- App 啟動時 `dbManager.init()` 拋例外 → try/catch 抓住 → `attachLogService` 跳過 → logger 內 `_logService?.write(...)` 變 noop → 整個 SQLite 寫入鏈路靜默失效
- App 主功能不受影響,正常啟動
- txt + console 仍正常工作
- 修復方式:刪 `app.db*` 三件套,下次啟動重建

**Q4:為什麼 txt 不去掉,只留 SQLite 就好?**
- 支援人員 / 客戶手上的工具最簡單就是「壓縮資料夾打包寄回」,不可能要他們學 SQL
- 純文字格式跨平台跨工具最通用
- DB 是給開發者後續查詢用,角色不衝突

**Q5:有人改了已 commit 的 migration SQL 內容會怎樣?**
drizzle 啟動時對該檔重算 hash,跟 `__drizzle_migrations.hash` 比對不一致 → 拋警告 / 報錯。所以**規則是只追加不修改**。要改舊行為 → 開新 migration ALTER 過去。

**Q6:WAL 是什麼?三個檔(`.db` / `.db-wal` / `.db-shm`)能不能單獨刪?**
WAL = Write-Ahead Log,SQLite 高並發模式。寫入先進 `.db-wal`,定期 checkpoint 進 `.db` 主檔。`.db-shm` 是 shared memory。
**要刪就三個一起刪**,不然會破壞一致性。`dbManager.close()` 退出時會自動 checkpoint,所以正常退出後重啟,WAL 可能是空的或不存在。

**Q7:為什麼選 better-sqlite3 不選 sql.js / node-sqlite3?**
- node-sqlite3 callback / Promise 風格,代碼啰嗦,性能弱
- sql.js 是 WASM,DB 整顆塞記憶體,只適合小資料 / 唯讀
- better-sqlite3 同步 API 在主進程是優勢(沒 UI 阻塞問題)、性能最好、TS 型別最完整

**Q8:為什麼選 drizzle 不選 Prisma / TypeORM?**
- Prisma 需要獨立 query engine binary,Electron 打包繁瑣,Native binary 平台分歧大
- TypeORM 用 Decorator + Reflection,啟動有開銷,TS 型別推導不純粹
- drizzle 純 TS schema,自動推導型別,query builder 寫起來像 SQL,沒額外 runtime,Electron 打包友善
