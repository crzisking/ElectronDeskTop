## 一、總體評價

**整體健康,可以進公測;沒有阻斷性 P0(無正確性 / 安全缺陷)。** 分層是真的(薄 handler → service/feature → `db/features/*/service.ts`),失敗軟處理一致,安全不變量(權限閘、腳本白名單、硬刪名單、明文 key 的信任邊界)都成立且是**有意設計,不要動**。退場的功能(my-nodes、memos)清得乾淨。

真正的債集中在三類:①**退場殘壳**(project-flow 只剩一個"今日活動聚合"卻還頂著死功能的名字 + 目錄 + 通道);②**一致性漂移**(response envelope 四種約定共存、兩個老 handler 用位置參數 + `any`);③**最新的代辦功能測試幾乎零覆蓋**,且不少純邏輯困在 `App.vue` 裡沒抽出來。此外有一批死通道 / 死 bridge / 死 i18n 是歷次改版(浮球、功能退場)留下的,drift guard 抓不到(它只校驗"內聯字串∈共享常量",不校驗反向)。

## 二、發現(按優先級,已跨四份審計去重)

### P0 —— 無
無正確性 / 安全阻斷項。下面唯一的邏輯 bug(週日邊界)實際影響很小,列在 P2。

### P1 —— 公測前值得修

- **P1-1 徹底清掉 `project-flow` 殘壳(命名誤導 + 跨死模塊依賴)**
  功能已退場,只剩「今日活動聚合」(首頁熱力圖用)。但:目錄仍叫 `electron/main/services/project-flow/`;唯一存活的 `ai-local.ts` **已不做任何 AI**,只有聚合 + 一個通用 `stripJsonFence`;`daily-advice/scheduler.ts:26` 跨進這個死模塊只為拿那個 util;渲染層 `src/features/project-flow/`(僅剩 `api.ts`/`types.ts`)是幽靈模塊;通道還叫 `PROJECT_FLOW_TODAY_ACTIVITY`。
  **改**:聚合遷到 `work-collect/`(或新建 `dashboard`/`features/activity`);`stripJsonFence` 移到 `utils/`;通道改名;刪 `project-flow/` 目錄與渲染層幽靈模塊。

- **P1-2 response envelope 四種約定共存,`ipc.types.ts` 頭註與實際不符**
  `{ok,data}|{ok,error}`(agent/todo/idea)、**裸值**(config/work-collect/auth/saved-credentials/user-profile/update)、`{ok,...adhoc}`(work-analysis)、`{ok,reason}`(notification)。`unwrapIpc` 只對 4 個 feature 成立,新人無法假設任何形狀。且主進程有**三份**同義 envelope helper(`safeRun`/`safe`/`ok`+`fail`)。
  **改**:抽一個共享 helper + 統一 `IpcResult<T>`;暫不遷移的裸值 handler 要在 `ipc.types.ts` 明確記錄"哪些走信封、哪些裸值"。

- **P1-3 代辦純邏輯沒抽出 / 沒測(+ 一個週日邊界 bug)**
  `electron/main/todo/runner.ts` 的 `resolveDate`(113/135)、`toPatch` 未 export、無測 —— 這是**清洗後端 AI 輸出的信任邊界**(欄位白名單、優先級 0/2 else 1、日期換算)。`resolveDate:148` 的週末計算 `now.getDate() + (7 - now.getDay())` 在**週日**(getDay()=0)會 +7 天,把"本週"算到下週日 → `dueKind` 在週日誤標 `thisweek`(實際影響小,∵ dock 排序/配色走 `dueAt` 不走 `dueKind`)。`src/windows/todo-dock/App.vue` 的 `urgency`/`fmtTime`/`presetDue`/`needsEnrich`/`sameDay`/`priLabel` 全是純邏輯卻內聯在 SFC,無法測。
  **改**:`runner.ts` export `resolveDate`/`toPatch` 並測(含週日/週六、缺 time、壞 dueDate);dock 純邏輯抽到 `todo-dock/dock-utils.ts`(顯式傳 `now`,對齊 `dashboard-utils.ts`)並測;修週日邊界。

- **P1-4 `HomeView.vue` 是路由視圖卻扛三個域的核心邏輯**
  `src/views/Home/HomeView.vue:176-342` 內聯 熱力圖 / 代辦 / 每日建議 三攤:`loadActivity`/`loadTodos`/`completeTodo`/`dueText` + 各自 IPC push 接線(腳本約 166 行)。違反「views = 骨架、無核心邏輯」。
  **改**:抽 `useTodayActivity` / `useHomeTodos` / `useDailyAdvice` 三個 composable(照 agent 窗那套),視圖只留佈局。

- **P1-5 兩個老 handler 用位置參數 + `any`,和約定漂移**
  `work-analysis.handlers.ts:94`(7 個位置參數)、`work-collect.handlers.ts:73`(6 個);後者驗證函數還用 `(p: any)`(30/44/50/82)—— 恰在**不可信 IPC 邊界**違反「禁 `any` 用 `unknown`」。新 handler(agent/idea/todo)都已用單一 deps 對象 + `unknown`。
  **改**:兩者轉 `deps:{...}` 對象;`any` → `unknown`。

- **P1-6 代辦備注刷新用魔法字符串,逃過 source-of-truth + drift guard**
  `todo-note-window.ts:39` 與 `todo-note.preload.ts` 兩端硬編碼 `'todo:note-target-changed'`,不在 `TodoChannels` 裡,易和 invoke 通道 `TODO_NOTE_TARGET:'todo:note-target'` 混。
  **改**:加 `PUSH_TODO_NOTE_TARGET_CHANGED: 'todo:note-target-changed'` 常量,兩端引用。

- **P1-7 `index.ts` 服務構造 / 置空三處同步,改動面 ×3**
  `electron/main/index.ts:167-206`:8 個 service 在 DB-init 塊 new 出來、catch 裡再逐個置 null、外加 8 個模塊級 `let`。加一個 service 要同步改三處(不規則縮排已是徵兆)。
  **改**:抽 `createServices(dbManager)` 工廠回一個結構(或 null),三處收斂成一處。
